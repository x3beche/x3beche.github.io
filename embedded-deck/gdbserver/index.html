00 Uzak debug neden

Gömülü hedef üzerinde IDE çalıştırmak mümkün değildir — gdbserver ile hedef üzerinde minimal bir debug stub çalıştırarak host'taki güçlü GDB'yi kullanabilirsiniz.

Embedded hedefin kısıtları

Gömülü Linux hedefi çoğunlukla şunlardan yoksundur: ekran, klavye, fare, IDE çalıştıracak RAM ve işlemci gücü. Hedef üzerinde doğrudan gdb çalıştırmak mümkün olsa bile kullanışsızdır. Uzak debug modeli bu sorunu çözer: hedefte yalnızca küçük ve hafif gdbserver çalışır; asıl GDB host'tadır ve TCP veya seri port üzerinden bağlanır.

Host (geliştirici makinesi)
├── arm-linux-gnueabihf-gdb  ← tam özellikli debugger
├── unstripped binary        ← debug sembolleri burada
└── sysroot                  ← target kütüphaneleri

    ↕ TCP (192.168.1.100:2345) veya /dev/ttyUSB0

Hedef cihaz (ARM/RISC-V/...)
├── gdbserver                ← minimal stub (~500 KB)
└── uygulama (çalışıyor)     ← debug edilen process

Alternatif debug yöntemleri

YöntemAvantajSınır
gdbserver (bu rehber)Tam GDB özelliği, Ethernet/USB seriOS ve uygulama boot etmeli
JTAG / SWD (OpenOCD)Bare-metal, boot öncesi, kernel debugDonanım bağlantısı gerekli
printk / printfBasit, her ortamda çalışırYavaş, intrusive, timing değişir
straceSistem çağrısı izleme, hedefte çalışırSembolik debug yok
perf / ftracePerformans analizi, kernel traceDebug sembolü genelde gereksiz

GDB vs LLDB

Gömülü Linux ekosisteminde GDB standarttır: Yocto/Buildroot toolchain'leri GDB içerir, vendor BSP'leri GDB script'leri sağlar, kernel source'u GDB-aware'dir. LLDB daha modern bir API sunar ancak cross-debug desteği ve ekosistemi gömülü Linux'ta GDB kadar olgun değildir.

Bu bölümde

  • Embedded hedef: IDE yok — gdbserver minimal stub çözümü
  • Cross-debug modeli: host GDB + hedef gdbserver + TCP
  • JTAG, strace, printk alternatifleri ve kullanım senaryoları
  • Gömülü Linux'ta GDB standardı — LLDB karşılaştırması

01 Toolchain ve GDB kurulumu

Cross-debug için host'ta cross-GDB, hedefte gdbserver ve host'ta sysroot — üç bileşenin doğru yapılandırılması gerekir.

Cross toolchain kurulumu

shell — host (Ubuntu/Debian)●
# ARM 32-bit (armhf) cross toolchain + GDB
apt install gcc-arm-linux-gnueabihf gdb-multiarch

# ARM 64-bit (aarch64) için
apt install gcc-aarch64-linux-gnu gdb-multiarch

# Yocto SDK'sından gelen cross-GDB
# SDK kurulduktan sonra:
source /opt/poky/4.0/environment-setup-armv7at2hf-neon-poky-linux-gnueabi
which $GDB   # arm-poky-linux-gnueabi-gdb

gdbserver kurulumu — hedef cihaz

shell — hedef (Debian tabanlı)●
# Hedef üzerinde gdbserver kur
apt install gdbserver

# Yocto image'ına ekle
# local.conf veya image recipe:
# IMAGE_INSTALL:append = " gdbserver"

# Buildroot:
# Target packages → Debugging, profiling → gdb debugger → gdbserver

# Versiyonu kontrol et
gdbserver --version

Debug sembol derlemesi

shell / Makefile●
# -g3: maksimum debug bilgisi (makro genişletme dahil)
arm-linux-gnueabihf-gcc -g3 -O0 -o myapp myapp.c

# Production: sembolü ayır (debug binary büyük, hedefte yer yok)
arm-linux-gnueabihf-objcopy --only-keep-debug myapp myapp.debug
arm-linux-gnueabihf-strip --strip-debug myapp
# myapp → hedefte; myapp.debug → host'ta GDB için

# ELF bilgisini kontrol et
file myapp
# myapp: ELF 32-bit LSB executable, ARM, EABI5, not stripped

# Debug section'ları var mı?
arm-linux-gnueabihf-readelf -S myapp | grep debug

Sysroot yapılandırması

Sysroot, hedef cihazın dosya sistemi yapısını host'ta yansıtan bir dizindir. GDB sysroot sayesinde hedef kütüphanelerin (libc.so, libpthread.so) sembollerini bulur ve doğru satır bilgisi gösterir:

shell — sysroot hazırlama●
# Hedeften sysroot kopyala (rsync ile)
rsync -avz root@192.168.1.100:/lib /opt/sysroot/
rsync -avz root@192.168.1.100:/usr/lib /opt/sysroot/usr/

# Veya Yocto SDK sysroot'u kullan
# /opt/poky/4.0/sysroots/armv7at2hf-neon-poky-linux-gnueabi/

# GDB'de set et
# (gdb) set sysroot /opt/sysroot

Bu bölümde

  • arm-linux-gnueabihf-gdb host'ta; gdbserver hedefte
  • -g3 -O0 ile debug binary; objcopy ile sembol ayırma
  • file ve readelf -S ile ELF debug bilgisi kontrolü
  • Sysroot: hedef kütüphane sembolleri için host'a kopyala

02 gdbserver ile ilk debug

Hedefte gdbserver başlatın, host'ta cross-GDB ile bağlanın ve ilk debug oturumunuzu tamamlayın.

Hedefte gdbserver başlatma

shell — hedef cihaz●
# Programı gdbserver ile başlat (TCP port 2345)
gdbserver :2345 ./myapp

# Argüman geçmek için
gdbserver :2345 ./myapp arg1 arg2

# Çalışan process'e attach (PID ile)
gdbserver :2345 --attach 1234

# Seri port üzerinden (USB-serial)
gdbserver /dev/ttyAMA0 ./myapp

# Bekleme mesajı:
# Process ./myapp created; pid = 1234
# Listening on port 2345

Host'ta GDB ile bağlanma

shell — host●
# Unstripped binary ile GDB başlat
arm-linux-gnueabihf-gdb ./myapp

# GDB prompt'unda:
(gdb) set sysroot /opt/sysroot
(gdb) set solib-search-path /opt/sysroot/lib:/opt/sysroot/usr/lib
(gdb) target remote 192.168.1.100:2345

# Bağlandı! gdbserver tarafta:
# Remote debugging from host 192.168.1.5, port 12345

(gdb) continue           # programı çalıştır
(gdb) info sharedlibrary  # yüklü shared lib'ler
(gdb) info threads        # tüm thread'ler

gdb-multiarch ile

shell — host (gdb-multiarch)●
# gdb-multiarch birden fazla mimariyi destekler
gdb-multiarch ./myapp

(gdb) set architecture arm
(gdb) set sysroot /opt/sysroot
(gdb) target remote 192.168.1.100:2345

stdio modu — seri port üzerinden

shell●
# Hedefte — seri port
gdbserver /dev/ttyUSB0 ./myapp

# Host'ta
(gdb) target remote /dev/ttyUSB0
# Baud rate ayarlamak için:
(gdb) set serial baud 115200
DİKKAT

Host'taki GDB binary'si ve hedefte çalışan binary aynı derlemeden gelmelidir. Debug semboller unstripped binary'dedir ve host'ta olmalıdır. Hedefte stripped binary, host'ta unstripped binary — bu standart production kurulumudur. Sysroot versiyonu hedef kütüphane versiyonlarıyla eşleşmelidir.

Bu bölümde

  • gdbserver :PORT ./myapp — TCP debug; --attach PID — çalışan process
  • target remote IP:PORT ile bağlantı
  • set sysroot ve set solib-search-path — kütüphane sembolleri
  • gdbserver /dev/ttyUSB0 — seri port modu

03 Breakpoint ve watchpoint

Breakpoint'ler programı belirli bir noktada durdurur; watchpoint'ler bellek konumundaki değişikliği izler — donanım watchpoint'leri memory-mapped I/O debug için kritiktir.

Yazılım breakpoint'leri

gdb session●
(gdb) break main                      # fonksiyon adı
(gdb) break src/network.c:142          # dosya:satır
(gdb) break process_packet             # başka fonksiyon
(gdb) break 87                          # mevcut dosyada satır

# Koşullu breakpoint — var == 0 ise dur
(gdb) break network.c:142 if retval == 0

# One-shot: bir kez tetiklenince silinir
(gdb) tbreak main

# Breakpoint bilgisi
(gdb) info breakpoints
(gdb) delete 2              # 2 numaralı breakpoint'i sil
(gdb) disable 1             # geçici devre dışı bırak
(gdb) enable 1              # tekrar etkinleştir

Donanım breakpoint'leri

Yazılım breakpoint'leri hedef memory'ye trap instruction yazar — bu çoğu durumda çalışır ama memory-mapped I/O adreslerinde sorun çıkarır. Donanım breakpoint'leri CPU'nun debug register'larını kullanır ve hiçbir şey yazmaz:

gdb session●
(gdb) hbreak *0x400520         # donanım breakpoint — adres
(gdb) hbreak gpio_set_value     # donanım breakpoint — fonksiyon
# Not: ARM Cortex-A'da genellikle 4-8 donanım breakpoint mevcuttur

Watchpoint'ler — bellek değişikliği izleme

gdb session●
(gdb) watch global_counter        # yazma watchpoint
(gdb) watch *0x20001000            # adres watchpoint
(gdb) watch buf[idx]               # dizi elemanı

(gdb) rwatch sensor_value          # okuma watchpoint
(gdb) awatch critical_flag         # okuma + yazma

# Watchpoint tetiklenince GDB durur ve eski/yeni değeri gösterir
# Hardware watchpoint 3: global_counter
# Old value = 0
# New value = 1

Breakpoint komutları

Breakpoint tetiklendiğinde otomatik çalışacak komutlar tanımlayabilirsiniz:

gdb session●
(gdb) break process_data
(gdb) commands 1
> print data_len
> print *data_ptr@8
> continue
> end
# Artık her tetiklenişte otomatik yazdırır ve devam eder

Bu bölümde

  • break fonksiyon/dosya:satır — yazılım breakpoint
  • Koşullu break: break loc if expr — şart sağlanınca dur
  • hbreak — donanım breakpoint: MMIO debug için zorunlu
  • watch/rwatch/awatch — bellek değişikliği izleme

04 Stack ve değişken inceleme

GDB ile call stack'i gezip frame'ler arasında geçiş yapabilir, yerel değişkenler ve belleği inceleyebilirsiniz.

Call stack (backtrace)

gdb session●
(gdb) backtrace         # bt kısaltması da çalışır
#0  parse_header (buf=0x7ffd1020, len=512) at parser.c:88
#1  process_packet (pkt=0x7ffd1000) at network.c:142
#2  main () at main.c:56

(gdb) bt full            # yerel değişkenlerle birlikte
(gdb) bt 5              # sadece üst 5 frame

# Frame seç
(gdb) frame 1           # #1 frame'e geç
(gdb) up               # bir üst frame
(gdb) down             # bir alt frame

Değişken inceleme

gdb session●
(gdb) info locals        # mevcut frame'deki yerel değişkenler
(gdb) info args          # fonksiyon argümanları

# print ile değer göster
(gdb) print retval
(gdb) print *ptr                  # pointer'ın işaret ettiği
(gdb) print arr[0]@10             # dizi: arr[0] başlayarak 10 eleman
(gdb) print /x counter            # hex formatında
(gdb) print (struct Header)*ptr   # cast ile

# Her adımda otomatik yazdır
(gdb) display retval
(gdb) display /x *gpio_base
(gdb) undisplay 1

Bellek inceleme (x komutu)

gdb session●
(gdb) x/10xw 0x80000000   # 10 word, hex, 0x80000000'dan
(gdb) x/20xb buf           # 20 byte, hex, buf adresinden
(gdb) x/s str_ptr          # string olarak
(gdb) x/4i $pc             # 4 instruction, PC'den disassemble

# Format: /[sayı][format][boyut]
# format: x(hex) d(dec) u(unsigned) o(octal) t(bin) a(addr) s(string) i(insn)
# boyut:  b(byte) h(halfword) w(word) g(giant=8byte)

Tip ve struct inceleme

gdb session●
(gdb) ptype struct Packet          # struct tanımı
(gdb) ptype /o struct Packet        # offset ve boyutlarla
(gdb) whatis my_var                 # değişkenin tipi

# Struct üyelerine eriş
(gdb) print pkt.header.type
(gdb) print pkt->payload[0]@16

Bu bölümde

  • bt / bt full: call stack ve frame içerikleri
  • frame N / up / down: frame'ler arası geçiş
  • print, print *ptr, print arr@N — değişken inceleme
  • x/Nxw addr: bellek hex dump — MMIO register okuma

05 Çok thread'li program debug

Çok thread'li programlarda GDB thread listesini gösterir, thread'ler arası geçiş yapmanıza ve deadlock'u tespit etmenize olanak tanır.

Thread yönetimi

gdb session●
(gdb) info threads
#  Id   Target Id         Frame
*  1    Thread 1234.1234  0x76543210 in recv () from /lib/libc.so.6
   2    Thread 1234.1235  process_data (data=0x...) at worker.c:42
   3    Thread 1234.1236  nanosleep () from /lib/libc.so.6

(gdb) thread 2             # thread 2'ye geç
(gdb) backtrace            # thread 2'nin call stack'i

# Tüm thread'lerin backtrace
(gdb) thread apply all bt

# Tüm thread'lerin backtrace (yerel değişkenlerle)
(gdb) thread apply all bt full

scheduler-locking

gdb session●
(gdb) set scheduler-locking off   # tüm thread'ler çalışır (varsayılan)
(gdb) set scheduler-locking on    # yalnızca mevcut thread çalışır
(gdb) set scheduler-locking step  # step sırasında kilitli, continue'da serbest

# step modunda inceleme için on tercih edilir:
# diğer thread'ler bekler, race condition izolasyonu kolaylaşır

Deadlock tespiti

gdb session — deadlock analizi●
(gdb) thread apply all bt full
# Thread 2 (çıktı):
#0  __lll_lock_wait () at lowlevellock.S:135
#1  pthread_mutex_lock ()
#2  worker_func (arg=...) at worker.c:88   ← mutex_b bekliyor

# Thread 3 (çıktı):
#0  __lll_lock_wait () at lowlevellock.S:135
#1  pthread_mutex_lock ()
#2  handler_func (arg=...) at handler.c:55 ← mutex_a bekliyor

# Thread 2 mutex_b bekliyor, Thread 3 mutex_a bekliyor
# → Klasik deadlock: her thread diğerinin tuttuğu lock'u bekliyor

# Mutex sahibini bul
(gdb) thread 2
(gdb) frame 2
(gdb) print mutex_b     # __owner alanını göster

Thread'e sinyal gönderme

gdb session●
(gdb) thread 2
(gdb) signal SIGUSR1      # thread 2'ye SIGUSR1 gönder
(gdb) signal 0            # sinyal yok, yalnızca resume

Bu bölümde

  • info threads / thread N: thread listesi ve geçiş
  • thread apply all bt: tüm thread'lerin stack trace'i
  • scheduler-locking: adım atarken diğer thread'leri durdur
  • Deadlock tespiti: bt full ile mutex bekleyenleri tespit et

06 Core dump analizi

Core dump, process'in crash anındaki tam bellek görüntüsüdür — crash'i sonradan ve cihaz olmaksızın analiz etmenizi sağlar.

Core dump aktif etme

shell — hedef cihaz●
# Core dump boyut limitini kaldır
ulimit -c unlimited

# Kalıcı yapılandırma (systemd olmayan)
echo "* soft core unlimited" >> /etc/security/limits.conf

# Core dosyasının nereye kaydedileceği
echo "/var/crash/core.%e.%p.%t" > /proc/sys/kernel/core_pattern
# %e: executable adı, %p: PID, %t: timestamp

# sysctl ile kalıcı
echo "kernel.core_pattern=/var/crash/core.%e.%p" >> /etc/sysctl.conf
sysctl -p

systemd-coredump entegrasyonu

shell — systemd●
# systemd-coredump: core'ları otomatik kaydeder ve indexler
# core_pattern: |/usr/lib/systemd/systemd-coredump %P %u %g %s %t %c %h

# Core dump listesini gör
coredumpctl list

# Belirli executable'ın son core dump'ını analiz et
coredumpctl gdb myapp

# Core dump'ı dışarı çıkar
coredumpctl dump myapp -o myapp.core

Cross core dump analizi (host'ta)

shell — host●
# Core dump dosyasını hedeften kopyala
scp root@192.168.1.100:/var/crash/core.myapp.1234 .

# Cross GDB ile aç
arm-linux-gnueabihf-gdb ./myapp core.myapp.1234

(gdb) set sysroot /opt/sysroot

# Crash stack trace
(gdb) bt
#0  0x00000000 in ?? ()   ← NULL pointer deref
#1  process_config (cfg=0x0) at config.c:77
#2  main () at main.c:34

# Crash anındaki değişkenler
(gdb) frame 1
(gdb) info locals
(gdb) print cfg         # 0x0 → NULL pointer!

# Hangi sinyal ile çöktü?
(gdb) info signals

addr2line ile hızlı lokalizasyon

shell●
# Crash adresi kaynak satırına çevir
arm-linux-gnueabihf-addr2line -e myapp 0x00008abc
# /home/user/project/src/config.c:77

# Birden fazla adres
arm-linux-gnueabihf-addr2line -e myapp -f 0x8abc 0x9012 0xa234
NOT

systemd-coredump cihaz depolamasında yer kaplar. Embedded sistemlerde /etc/systemd/coredump.conf ile maksimum core sayısını ve boyutunu sınırlayın: MaxUse=100M, KeepFree=50M. Disk dolmasını önlemek için core'ları sıkıştırın: Compress=yes.

Bu bölümde

  • ulimit -c unlimited + kernel.core_pattern ile core dump aktif
  • systemd-coredump: coredumpctl list / gdb / dump
  • Cross core dump: arm-linux-gnueabihf-gdb ./myapp core
  • bt → frame N → info locals: crash anını yeniden canlandır

07 QEMU ile donanımsız debug

QEMU'nun dahili GDB stub'ı, gerçek hedef donanımı olmadan cross-binary ve hatta kernel debug yapılmasını sağlar.

QEMU sistem emülasyonu ile kernel debug

shell — QEMU + GDB stub●
# -s: port 1234'te GDB stub başlat
# -S: ilk instruction'dan önce dur (GDB bağlanana kadar bekle)
qemu-system-arm \
  -M vexpress-a9 \
  -cpu cortex-a9 \
  -m 512M \
  -kernel zImage \
  -dtb vexpress-v2p-ca9.dtb \
  -append "console=ttyAMA0 root=/dev/mmcblk0p2" \
  -nographic \
  -s -S  # ← GDB stub
gdb session — kernel debug●
(gdb) file vmlinux                      # kernel debug sembolü
(gdb) target remote localhost:1234
(gdb) break start_kernel               # kernel entry point
(gdb) continue

# Yüklü modül sembolü ekle
(gdb) add-symbol-file drivers/gpio/gpio-mydev.ko 0xbf000000

# Page fault handler'da breakpoint
(gdb) break do_page_fault

qemu-user ile kullanıcı alanı cross-binary debug

shell●
# ARM binary'yi x86 host'ta çalıştır + GDB stub
qemu-arm -g 1234 -L /opt/sysroot ./myapp arg1

# Host'ta GDB ile bağlan
arm-linux-gnueabihf-gdb ./myapp
(gdb) target remote localhost:1234
(gdb) continue

QEMU ile otomatik test

shell — CI test scripti●
# QEMU başlat (arkaplanda)
qemu-arm -g 1234 ./myapp &

# GDB batch modunda test
arm-linux-gnueabihf-gdb -batch -ex "target remote localhost:1234" \
  -ex "break test_entry" \
  -ex "continue" \
  -ex "print result" \
  -ex "quit" \
  ./myapp
NOT

QEMU sistem emülasyonu (-s -S) ile kernel debug yaparken, vmlinux'un CONFIG_DEBUG_INFO=y ile derlenmiş olması gerekir. CONFIG_FRAME_POINTER=y de backtrace kalitesini artırır. Yocto'da: EXTRA_OEMAKE += "CONFIG_DEBUG_INFO=y" kernel recipe'ye eklenebilir.

Bu bölümde

  • qemu-system-arm -s -S: dahili GDB stub, port 1234
  • vmlinux + add-symbol-file ile kernel modülü debug
  • qemu-user -g 1234: kullanıcı alanı cross-binary, donanımsız
  • GDB batch modu ile CI/CD otomatik test

08 .gdbinit ve scripting

.gdbinit dosyaları tekrar eden bağlantı ve yapılandırma adımlarını otomatikleştirir; Python scripting ise GDB'yi özelleştirmenizi sağlar.

Proje .gdbinit dosyası

project/.gdbinit●
# Proje dizinindeki .gdbinit — her oturumda otomatik yüklenir
# İzin gerekli: ~/.gdbinit içinde "add-auto-load-safe-path /path/to/project"

set architecture arm
set sysroot /opt/sysroot
set solib-search-path /opt/sysroot/lib:/opt/sysroot/usr/lib

# remote bağlantı için kısayol
define connect
  target remote 192.168.1.100:2345
end

# Uygulama loglarını göster (özel komut)
define show-log
  print log_buffer
  print log_level
end

# Her stop'ta register dump
define hook-stop
  info registers pc sp lr
end

Python pretty-printer

project/gdb-pretty.py●
import gdb

class PacketPrinter:
    """Pretty-printer for struct Packet"""
    def __init__(self, val):
        self.val = val

    def to_string(self):
        t    = self.val['header']['type']
        size = self.val['header']['length']
        return f"Packet{{type={int(t)}, len={int(size)}}}"

def register_printers(objfile):
    pp = gdb.printing.RegexpCollectionPrettyPrinter("myapp")
    pp.add_printer('Packet', '^Packet$', PacketPrinter)
    gdb.printing.register_pretty_printer(objfile, pp)

register_printers(gdb.current_objfile())
gdb session — Python script yükle●
(gdb) source gdb-pretty.py
(gdb) print my_packet
# $1 = Packet{type=3, len=512}

# Python interpreter doğrudan
(gdb) python print(gdb.current_objfile())
(gdb) python print(gdb.selected_frame().name())

Batch modu — CI/CD headless analiz

shell — commands.gdb●
# commands.gdb — core dump otomatik analiz scripti
set pagination off
set confirm off
set sysroot /opt/sysroot

# Crash bilgisini çıkar
backtrace full
info registers
info threads
thread apply all backtrace

quit
shell — CI pipeline●
# Headless core dump analizi
arm-linux-gnueabihf-gdb -batch -x commands.gdb \
  ./myapp core.myapp.1234 > crash-report.txt 2>&1

# Raporu artifact olarak sakla
DİKKAT

Güvenlik açısından: proje dizinindeki .gdbinit dosyaları varsayılan olarak yüklenmez. ~/.gdbinit içine add-auto-load-safe-path /path/to/project eklemediyseniz GDB güvenlik uyarısı verir ve proje .gdbinit'ini yok sayar. Bu uyarıyı göz ardı etmeyin — kötü amaçlı .gdbinit dosyaları tehlikelidir.

Bu bölümde

  • Proje .gdbinit: set sysroot, define komutları, hook-stop
  • Python pretty-printer: struct'ları okunabilir göster
  • gdb -batch -x commands.gdb: CI/CD headless analiz
  • add-auto-load-safe-path: güvenli .gdbinit yükleme

09 VS Code ile uzak debug

VS Code'un launch.json konfigürasyonu, hedef cihazda gdbserver çalışırken host'ta tam GUI debug deneyimi sunar.

launch.json konfigürasyonu

.vscode/launch.json●
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Remote ARM Debug",
      "type": "cppdbg",
      "request": "launch",
      "program": "${workspaceFolder}/build/myapp",
      "miDebuggerPath": "/usr/bin/arm-linux-gnueabihf-gdb",
      "miDebuggerServerAddress": "192.168.1.100:2345",
      "stopAtEntry": false,
      "cwd": "${workspaceFolder}",
      "environment": [],
      "externalConsole": false,
      "additionalSOLibSearchPath": "/opt/sysroot/lib:/opt/sysroot/usr/lib",
      "setupCommands": [
        {
          "description": "Set sysroot",
          "text": "set sysroot /opt/sysroot",
          "ignoreFailures": false
        },
        {
          "description": "Enable pretty printing",
          "text": "-enable-pretty-printing",
          "ignoreFailures": true
        }
      ]
    }
  ]
}

Debug oturumu akışı

1. Hedef cihazda: gdbserver :2345 ./myapp
2. VS Code'da: Run → Start Debugging (F5)
3. launch.json okunur → arm-linux-gnueabihf-gdb başlar
4. miDebuggerServerAddress'e bağlanır (192.168.1.100:2345)
5. Breakpoint koy (editörde tıkla) → F5 → breakpoint'e kadar çalış
6. Değişkenler paneli, call stack, watch expressions GUI'de

Otomatik gdbserver başlatma (preLaunchTask)

.vscode/tasks.json●
{
  "tasks": [
    {
      "label": "Start gdbserver",
      "type": "shell",
      "command": "ssh root@192.168.1.100 'pkill gdbserver; gdbserver :2345 /usr/bin/myapp &'",
      "isBackground": true
    }
  ]
}
launch.json — preLaunchTask ekle●
{
  "preLaunchTask": "Start gdbserver",
  "name": "Remote ARM Debug (auto)",
  /* ... diğer alanlar aynı ... */
}

Cortex-Debug extension — JTAG/SWD

launch.json — Cortex-Debug (OpenOCD)●
{
  "name": "Cortex-Debug (OpenOCD)",
  "type": "cortex-debug",
  "request": "launch",
  "servertype": "openocd",
  "configFiles": ["interface/stlink.cfg", "target/stm32f4x.cfg"],
  "executable": "${workspaceFolder}/build/firmware.elf",
  "svdFile": "STM32F407.svd"
}

Remote SSH + VS Code

VS Code Remote SSH extension ile hedef cihaza SSH bağlantısı kurulur ve editörden doğrudan hedef üzerinde kod düzenlenebilir. Bu durumda gdbserver'ı hedef üzerinde doğrudan başlatıp aynı makinede localhost:2345 ile debug yapabilirsiniz — bant genişliği gereksinimi azalır.

launch.json — Remote SSH + localhost●
{
  "miDebuggerServerAddress": "localhost:2345",
  /* Remote SSH portforward: ssh -L 2345:localhost:2345 root@device */
  /* veya VS Code Remote SSH otomatik port forward */
}

Bu bölümde

  • launch.json: miDebuggerPath, miDebuggerServerAddress, additionalSOLibSearchPath
  • setupCommands: set sysroot ve pretty-printing GDB'ye geçilir
  • preLaunchTask: F5 öncesi hedefte gdbserver otomatik başlatılır
  • Cortex-Debug extension: OpenOCD ile JTAG/SWD GUI debug
  • VS Code Remote SSH: hedef üzerinde edit + localhost port forward ile debug
