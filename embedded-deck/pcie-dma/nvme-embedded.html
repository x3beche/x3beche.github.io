00 NVMe nedir? — PCIe üzerinde depolama

NVM Express (NVMe), PCIe veri yolu üzerinden flash depolama erişimi için tasarlanmış yüksek performanslı bir protokoldür. SATA ve SAS'ın sunduğu depolama protokollerinden çok daha düşük gecikme ve yüksek paralellik sağlar.

SATA/SAS'a göre NVMe'nin üstünlükleri

ÖzellikSATA IIISAS-3NVMe (PCIe Gen4)
Bant genişliği (seq)~550 MB/s~1.2 GB/s~7 GB/s (x4)
Okuma IOPS (random 4K)~100K~200K~1.5M
Gecikme (ortalama)~100 µs~50 µs~20 µs
Max kuyruk derinliği32 (NCQ)25465535 × 65535
Kuyruk sayısı1164K I/O queue
Protokol overheadYüksek (ATA)Orta (SCSI)Düşük (NVMe)

NVMe tasarım felsefesi

NVMe, flash belleğin özelliklerine göre sıfırdan tasarlandı. SATA ve SAS, dönen manyetik disk için geliştirilen protokollerden türetilmişti — sıralı erişim, tek kuyruk ve yüksek gecikme toleransı bu varsayımların izini taşır. NVMe'nin temel yenilikleri şunlardır:

Çoklu I/O kuyruğuHer CPU çekirdeği için ayrı SQ/CQ çifti — kilit çekişmesi sıfır
Düşük protokol overheadNVMe komutu 64 bayt; SCSI CDB + SAM overhead'i yok
PCIe doğrudan erişimKöprü chip yok; SATA kontrolcüsünün getirdiği ek gecikme yok
Host yönetimli kuyrukCPU, SSD'ye komut göndermek için syscall yapmaz — doorbell yazması yeterli

NVMe spesifikasyonu

NVMe spesifikasyonu NVM Express, Inc. tarafından yönetilir. 1.0'dan başlayan spec, günümüzde NVMe 2.x serisindedir. Gömülü sistemler için NVMe-oF (over Fabrics) ve NVMe-MI (Management Interface) ek değer sunar.

Bu bölümde

  • NVMe: PCIe üzerinde flash depolama; SATA'ya göre ~12× yüksek bant genişliği
  • 64K kuyruk × 65K derinlik: çok çekirdekli sistemlerde tam paralellik
  • 20 µs ortalama gecikme: SATA'nın 100 µs'inin beşte biri
  • NVMe 1.0 (2011) → NVMe 2.x (2022+): namespace yönetimi, ZNS, key-value eklendi

01 NVMe mimarisi — SQ/CQ ve doorbell

NVMe, host belleğinde oluşturulan Submission Queue (SQ) ve Completion Queue (CQ) çifti üzerine inşa edilir. SSD bu kuyruklara PCIe DMA ile doğrudan erişir.

Kuyruk çifti mimarisi

  Host Belleği (DMA-mapped):
  ┌────────────────────────────────────────────────────────────┐
  │  Admin SQ    [cmd0][cmd1][cmd2] ...          head=0, tail=3│
  │  Admin CQ    [cpl0][cpl1]       ...          head=2, tail=2│
  ├────────────────────────────────────────────────────────────┤
  │  I/O SQ[0]   [cmd0][cmd1] ...  (CPU 0 kullanır)            │
  │  I/O CQ[0]   [cpl0]       ...                              │
  ├────────────────────────────────────────────────────────────┤
  │  I/O SQ[1]   [cmd0] ...        (CPU 1 kullanır)            │
  │  I/O CQ[1]   []                                            │
  └────────────────────────────────────────────────────────────┘

  NVMe Kontrolcü (SSD tarafı):
  ┌────────────────────────────────────────────────────────────┐
  │  BAR0 Register Space:                                       │
  │  ├─ CAP  (Controller Capabilities)                         │
  │  ├─ CC   (Controller Configuration)                        │
  │  ├─ CSTS (Controller Status)                               │
  │  ├─ AQA  (Admin Queue Attributes)                          │
  │  ├─ ASQ  (Admin SQ Base Address)                           │
  │  ├─ ACQ  (Admin CQ Base Address)                           │
  │  └─ SQ0TDBL, CQ0HDBL, SQ1TDBL, ... (doorbell registers)  │
  └────────────────────────────────────────────────────────────┘

I/O akışı

  1. Host: NVMe komutu (64 bayt) → SQ[tail]'e yaz
  2. Host: SQ tail doorbell'a (tail+1) yaz → SSD'ye "yeni komut var" bildir
  3. SSD:  SQ'yu DMA ile oku (host belleğinden)
  4. SSD:  flash okuma/yazma işlemini yap
  5. SSD:  CQ[tail]'e completion entry yaz (16 bayt)
  6. SSD:  MSI-X interrupt gönder (veya polling modda değil)
  7. Host: CQ'yu kontrol et, CQ head doorbell'ı güncelle

Admin vs I/O kuyrukları

Kuyruk TürüAmaçSayıBoyut
Admin SQ/CQIdentify, Create/Delete Queue, Format, Firmware1 çift (zorunlu)Max 4096 entry
I/O SQ/CQRead, Write, Flush, Dataset ManagementMax 65535 çiftMax 65536 entry

Bu bölümde

  • SQ: host komut kuyruğu (DMA-mapped, 64 bayt/entry); doorbell yazması ile SSD'yi tetikle
  • CQ: SSD completion kuyruğu (16 bayt/entry); MSI-X interrupt veya polling ile tüket
  • Admin kuyruğu: controller yapılandırması; I/O kuyruğu: gerçek veri transferi
  • CPU başına kuyruk: kilit çekişmesi sıfır — NUMA-aware dağıtım mümkün

02 NVMe komut seti — Read, Write, Identify

NVMe komut seti, Admin ve I/O olmak üzere iki gruba ayrılır. Her komut 64 baytlık Submission Queue Entry (SQE) ile tanımlanır.

SQE yapısı

NVMe SQE yapısı (64 bayt)●
Offset  Boyut  Alan
0x00    1      Opcode (komut kodu)
0x01    1      FUSE flags | PSDT (Physical Region Page / SGL)
0x02    2      Command Identifier (CID) — CQE'de geri döner
0x04    4      Namespace ID (NSID)
0x08    8      Reserved / Metadata Pointer
0x10    16     Data Pointer (PRP1, PRP2 veya SGL)
0x28    4      CDW10 — komuta özel (örn. starting LBA low)
0x2C    4      CDW11 — komuta özel (örn. starting LBA high)
0x30    4      CDW12 — Number of Logical Blocks (NLB-1)
0x34    4      CDW13
0x38    4      CDW14
0x3C    4      CDW15

Temel I/O komutları

KomutOpcodeAçıklama
Read0x02LBA aralığını oku; PRP/SGL ile DMA hedefi
Write0x01LBA aralığına yaz; buffered veya FUA (Force Unit Access)
Flush0x00Yazma tamponunu kalıcı ortama boşalt (fsync benzeri)
Write Zeroes0x08LBA aralığını sıfırla (copy olmadan, controller desteklerse)

Admin komutları

PRP vs SGL

PRP (Physical Region Page): 4 KB hizalı sayfa listesi; basit, yaygın kullanım. SGL (Scatter-Gather List): NVMe 1.1'den itibaren; daha esnek, farklı boyutlu parçalar. Linux nvme sürücüsü her ikisini de destekler; SGL büyük transfer için daha verimli.

Bu bölümde

  • SQE 64 bayt: opcode + NSID + DMA pointer (PRP/SGL) + komut-özel CDW10-15
  • Read/Write: CDW10-11 başlangıç LBA; CDW12 blok sayısı (NLB-1)
  • Flush: yazma tamponunu fiziksel ortama yaz; FUA flag ile her Write'ı otomatik flush
  • Dataset Mgmt: TRIM/discard desteği; wear leveling ve SSD ömrü için kritik

04 Linux NVMe sürücüsü — nvme-cli ve namespace

Linux nvme sürücüsü, NVMe namespace'lerini block cihazı olarak sunar. nvme-cli ile controller ve namespace yönetimi yapılır.

bash — nvme-cli temel komutları●
# nvme-cli kurulumu
sudo apt-get install nvme-cli

# Cihaz listesi
sudo nvme list
sudo nvme list-subsys

# Namespace bilgisi
sudo nvme id-ns /dev/nvme0n1
# nsze: 976773168 (toplam LBA sayısı)
# ncap: 976773168 (kapasite)
# nuse: 976773168 (kullanılan)
# lbaf[0]: bs=512B

# 4K LBA formatına geç (performans için)
sudo nvme format /dev/nvme0n1 --lbaf=1
# Dikkat: mevcut veriyi siler!

# Yeni namespace oluştur (NVMe 1.2+)
sudo nvme create-ns /dev/nvme0 \
    --nsze=20971520 # 10 GB (4K blok ile) \
    --ncap=20971520 \
    --flbas=1       # 4K LBA format

# Namespace'i controller'a ata
sudo nvme attach-ns /dev/nvme0 --namespace-id=2 --controllers=0x1

# Reset (yeni namespace görünmesi için)
sudo nvme reset /dev/nvme0
# → /dev/nvme0n2 oluşur

# Hata log
sudo nvme error-log /dev/nvme0

# Sanitize (güvenli silme)
sudo nvme sanitize /dev/nvme0 --sanact=2  # Block Erase

  • Namespace çoğullama: tek fiziksel SSD'yi bağımsız namespace'lere böl
  • rotational=0: scheduler'a "disk değil" bildirir; none scheduler seç

05 NVMe performans tuning

NVMe'nin potansiyelini tam kullanmak için scheduler, queue derinliği, write cache ve power state ayarları kritiktir. Gömülü sistemlerde güç ve performans dengesi önemlidir.

bash — NVMe performans ayarları●
#!/bin/bash
# nvme_tune.sh — NVMe performans optimizasyonu

NVME_DEV="/dev/nvme0n1"
NVME_CTRL="/dev/nvme0"

echo "=== 1. I/O Scheduler: none (NVMe için ideal) ==="
echo "none" | sudo tee /sys/block/nvme0n1/queue/scheduler
cat /sys/block/nvme0n1/queue/scheduler
# [none] mq-deadline kyber bfq

echo "=== 2. Read-ahead: azalt (random I/O için) ==="
# Sequential: büyük readahead iyi; Random: küçük ya da 0
echo 0 | sudo tee /sys/block/nvme0n1/queue/read_ahead_kb
# Sequential workload için: echo 2048 ...

echo "=== 3. Write cache etkinleştir ==="
sudo nvme set-feature $NVME_CTRL \
    --feature-id=0x06 \    # Volatile Write Cache
    --value=1            # 1=enable
sudo nvme get-feature $NVME_CTRL \
    --feature-id=0x06 --human-readable

echo "=== 4. NVMe Power State ==="
# Power State 0: maksimum performans, yüksek güç
# Power State 4: minimum güç, düşük performans
sudo nvme set-feature $NVME_CTRL \
    --feature-id=0x02 \    # Power Management
    --value=0            # PS0: max performance

echo "=== 5. IRQ Affinity (CPU0'a sabit) ==="
# NVMe IRQ'larını belirli CPU'ya bağla
for irq in $(grep nvme /proc/interrupts | awk '{print $1}' | tr -d ':'); do
    echo 1 | sudo tee /proc/irq/$irq/smp_affinity_list
done

echo "=== 6. fio hız testi ==="
sudo fio \
    --name=randread \
    --filename=$NVME_DEV \
    --rw=randread \
    --bs=4k \
    --direct=1 \         # O_DIRECT: page cache bypass
    --numjobs=4 \        # paralel iş
    --iodepth=32 \
    --runtime=30 \
    --group_reporting

Power state ve güç tüketimi karşılaştırması

Power StatePerformansGüçGömülü Kullanım
PS0 (Active)Tam3-6WYoğun I/O dönemi
PS1%802-4WOrta yük
PS3 (Non-Op)Düşük~100 mWBekleme modu
PS4 (Non-Op)Çok düşük~5 mWEnerji tasarrufu kritiksek

Bu bölümde

  • Scheduler "none": NVMe kendi donanım kuyruğunu yönetir; kernel scheduling overhead'i sıfır
  • read_ahead_kb=0: rastgele erişimde gereksiz önyüklemeyi engeller
  • Write cache enable: güç kaybı riski varsa disable; performans için enable
  • Power state: gömülü bekleme modunda PS3/PS4 ile ~100 mW'a kadar düşür

06 io_uring + NVMe — sqpoll ve fixed buffers

io_uring, NVMe'nin düşük gecikme potansiyelini en iyi kullanan Linux I/O mekanizmasıdır. SQPOLL modu ile NVMe I/O için syscall overhead'i tamamen ortadan kalkar.

C — io_uring + NVMe SQPOLL modunda okuma●
#include <liburing.h>
#include <fcntl.h>
#include <time.h>

#define QUEUE_DEPTH  128
#define BLOCK_SIZE   4096
#define NUM_REQS     1000

static uint64_t now_ns(void)
{
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return ts.tv_sec * 1000000000ULL + ts.tv_nsec;
}

int nvme_io_uring_bench(const char *dev_path)
{
    struct io_uring ring;
    struct io_uring_params params;
    int ret, fd;
    char *buf;
    uint64_t t0, elapsed;
    long total_bytes = 0;

    /* 1. SQPOLL modu: kernel thread SQ'yu poll eder */
    memset(¶ms, 0, sizeof(params));
    params.flags = IORING_SETUP_SQPOLL;
    params.sq_thread_idle = 2000;  /* 2 sn boşta → uy */

    ret = io_uring_queue_init_params(QUEUE_DEPTH, &ring, ¶ms);
    if (ret < 0) {
        fprintf(stderr, "io_uring_queue_init: %s\n", strerror(-ret));
        /* SQPOLL başarısız → normal mod */
        ret = io_uring_queue_init(QUEUE_DEPTH, &ring, 0);
        if (ret < 0) return ret;
    }

    /* 2. O_DIRECT ile aç: page cache bypass → gerçek NVMe latency */
    fd = open(dev_path, O_RDONLY | O_DIRECT);
    if (fd < 0) { perror("open"); return -1; }

    /* 3. Registered file: fd'yi ring'e bir kez kaydet */
    int fds[1] = { fd };
    io_uring_register_files(&ring, fds, 1);

    /* 4. Fixed buffer: belleği kernel'a pin'le */
    ret = posix_memalign((void **)&buf, BLOCK_SIZE,
                          BLOCK_SIZE * QUEUE_DEPTH);
    if (ret) { perror("posix_memalign"); return -1; }

    struct iovec iov = { .iov_base = buf, .iov_len = BLOCK_SIZE * QUEUE_DEPTH };
    io_uring_register_buffers(&ring, &iov, 1);

    t0 = now_ns();

    /* 5. Batch submit */
    int submitted = 0, completed = 0;
    while (completed < NUM_REQS) {
        /* Mümkün olduğunca SQE doldur */
        while (submitted - completed < QUEUE_DEPTH &&
               submitted < NUM_REQS) {
            struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
            if (!sqe) break;

            int buf_idx = submitted % QUEUE_DEPTH;
            off_t offset = (off_t)(submitted % 1024) * BLOCK_SIZE;

            /* Fixed file (IOSQE_FIXED_FILE) + fixed buffer (prep_read_fixed) */
            io_uring_prep_read_fixed(sqe,
                0,                      /* registered file index */
                buf + buf_idx * BLOCK_SIZE,
                BLOCK_SIZE,
                offset,
                0);                     /* registered buffer index */
            sqe->flags |= IOSQE_FIXED_FILE;
            io_uring_sqe_set_data(sqe, (void *)(uintptr_t)submitted);
            submitted++;
        }

        /* SQPOLL modunda submit gerek yok; yoksa: */
        if (!(params.flags & IORING_SETUP_SQPOLL))
            io_uring_submit(&ring);

        /* CQE al */
        struct io_uring_cqe *cqe;
        if (io_uring_wait_cqe(&ring, &cqe) == 0) {
            if (cqe->res > 0) total_bytes += cqe->res;
            io_uring_cqe_seen(&ring, cqe);
            completed++;
        }
    }

    elapsed = now_ns() - t0;

    printf("Tamamlandı: %d okuma, toplam %ld bayt\n",
           NUM_REQS, total_bytes);
    printf("Süre: %.2f ms\n", elapsed / 1e6);
    printf("IOPS: %.0f\n", NUM_REQS / (elapsed / 1e9));
    printf("Throughput: %.1f MB/s\n",
           (double)total_bytes / (elapsed / 1e9) / (1024*1024));
    printf("Ortalama latency: %.1f µs\n",
           elapsed / 1e3 / NUM_REQS);

    io_uring_unregister_buffers(&ring);
    io_uring_unregister_files(&ring);
    io_uring_queue_exit(&ring);
    close(fd);
    free(buf);
    return 0;
}

int main(int argc, char **argv)
{
    const char *dev = argc > 1 ? argv[1] : "/dev/nvme0n1";
    printf("NVMe io_uring benchmark: %s\n", dev);
    return nvme_io_uring_bench(dev);
}

io_uring vs pread karşılaştırması (NVMe üzerinde)

YöntemOrtalama LatencyIOPS (4K, QD32)CPU kullanımı
pread (blocking)~80 µs~12KYüksek
libaio (AIO)~40 µs~50KOrta

Bu bölümde

  • SQPOLL: kernel polling thread ile submission syscall sıfır — en düşük latency
  • O_DIRECT: page cache atlayarak gerçek NVMe latency'yi ölç; sector-aligned buffer zorunlu

  • SQPOLL + fixed: pread'e göre 4-5× daha fazla IOPS; latency yarıya iner

bash — fio benchmark: CM4 NVMe gerçek performans●
# fio kurulumu
sudo apt-get install fio

# Sequential Read (büyük dosya okuma)
sudo fio \
    --name=seq_read \
    --filename=/dev/nvme0n1 \
    --rw=read \
    --bs=1M \
    --direct=1 \
    --numjobs=1 \
    --iodepth=8 \
    --runtime=30 \
    --group_reporting
# READ: bw=491MiB/s (515MB/s)  ← PCIe Gen2 x1 tavanına yakın

# Sequential Write
sudo fio \
    --name=seq_write \
    --filename=/dev/nvme0n1 \
    --rw=write \
    --bs=1M \
    --direct=1 \
    --numjobs=1 \
    --iodepth=8 \
    --runtime=30 \
    --group_reporting
# WRITE: bw=285MiB/s (299MB/s)

# Random 4K Read (IOPS testi)
sudo fio \
    --name=rand4k_read \
    --filename=/dev/nvme0n1 \
    --rw=randread \
    --bs=4k \
    --direct=1 \
    --numjobs=1 \
    --iodepth=32 \
    --runtime=30 \
    --group_reporting
# READ: IOPS=45.2k, BW=176MiB/s
# lat (usec): min=16, max=8940, avg=707.81
bash — wear leveling izleme ve TRIM●
# Wear leveling durumu
sudo nvme smart-log /dev/nvme0 | grep -E "percentage_used|data_units"
# percentage_used         : 3%
# data_units_read         : 12,345 [6,321 GB]
# data_units_written      : 5,678 [2,907 GB]
# W/A hesapla: written_to_flash / host_written
# → 1.0 idealdir; yüksek W/A wear leveling overhead'ini gösterir

# TRIM: artık kullanılmayan blokları bildir
sudo fstrim -v /mnt/nvme
# /mnt/nvme: 12.5 GiB (13421772800 bytes) trimmed

# Otomatik TRIM: fstab'da 'discard' mount seçeneği
# veya systemd-fstrim.timer (haftada bir çalışır)
sudo systemctl enable fstrim.timer
sudo systemctl start fstrim.timer

# Periyodik SMART izleme betiği
cat <<'EOF' | sudo tee /usr/local/bin/nvme_health_check.sh
#!/bin/bash
THRESHOLD=90  # % kullanım uyarı eşiği
USED=$(nvme smart-log /dev/nvme0 | \
       grep percentage_used | awk '{print $3}' | tr -d '%')
if [ "${USED:-0}" -ge "$THRESHOLD" ]; then
    logger -t nvme_health "UYARI: NVMe %$USED kullanım — SSD ömrü azalıyor!"
fi
EOF
sudo chmod +x /usr/local/bin/nvme_health_check.sh

# Her gün çalıştır
echo "0 6 * * * root /usr/local/bin/nvme_health_check.sh" | \
    sudo tee /etc/cron.d/nvme-health

  • SMART percentage_used: %90'ı geçince disk değişimi planla; izleme betiği kur
