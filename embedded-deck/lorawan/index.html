00 LoRa ve LoRaWAN Farkı

LoRa ve LoRaWAN sık sık birbirinin yerine kullanılır ancak farklı katmanları tanımlar. Bu farkı anlamak doğru sistem tasarımının temelidir.

Chirp Spread Spectrum — LoRa'nın sırrı

LoRa, her sembolü geniş bir frekans bandına süpüren (chirp) özel bir spread spectrum tekniği kullanır. Bu yaklaşım, Gaussian bandpass gürültüsünün altında bile sinyal decode edilebilmesini sağlar. İşte bu nedenle LoRa'nın link budget'ı 154–157 dB'e ulaşabilir — klasik FSK veya OOK modülasyonların çok ötesinde.

ÖNEMLİ

Semtech'in CSS modülasyon algoritması tescilli ve patentlidir. LoRa transceiver çiplerini üretmek için Semtech'ten lisans gerekmektedir. LoRaWAN protokolü ise LoRa Alliance tarafından açık standarttır — herkes uygulayabilir.

Bu bölümde

  • LoRa = Semtech'in tescilli fiziksel katman modülasyonu (CSS)
  • LoRaWAN = LoRa Alliance'ın açık MAC protokolü ve ağ mimarisi
  • CSS modülasyon: gürültü altında çalışma, 154+ dB link budget

01 LoRa RF Temelleri

LoRa'nın RF parametrelerini anlamak — Spreading Factor, Bandwidth, Coding Rate — menzil, hız ve güç tüketimi arasındaki dengeyi yönetmek için kritiktir.

Spreading Factor (SF)

SF, her sembolün kaç chip içerdiğini belirler: SF7 = 2^7 = 128 chip/sembol, SF12 = 2^12 = 4096 chip/sembol. Yüksek SF daha uzun menzil sağlar ama hava süresi (time-on-air) dramatik artar.

SFBit hızı (BW125)Menzil (tipik)SensitivityToA (10 byte)
SF75469 bps~2 km şehir içi-123 dBm~56 ms
SF83125 bps~4 km-126 dBm~103 ms
SF91758 bps~6 km-129 dBm~185 ms
SF10977 bps~9 km-132 dBm~370 ms
SF11537 bps~12 km-134.5 dBm~741 ms
SF12293 bps15+ km (açık alan)-137 dBm~1483 ms

Bandwidth (BW)

BW125125 kHz — en yaygın; iyi menzil/hız dengesi; EU868'de standart uplink
BW250250 kHz — %70 daha yüksek hız, ~3 dB daha az hassasiyet
BW500500 kHz — US915 downlink kanalı; en yüksek hız, en kısa menzil

Coding Rate (CR)

CR, hata düzeltme redundancy'sini kontrol eder: CR4/5 = %25 fazladan bit, CR4/8 = %100 fazladan bit. Gürültülü ortamda daha yüksek CR hataları tolere eder ancak hızı düşürür.

Link budget hesabı

bash — link budget●
# Örnek link budget hesabı — EU868, SF10, BW125
# TX gücü:    +14 dBm (EU868 maksimum)
# TX anten:   +2 dBi
# Kablo kaybı: -0.5 dB
# Yol kaybı:   değişken
# RX anten:   +2 dBi
# RX sensitivity: -132 dBm (SF10, BW125)
#
# Maksimum izin verilen yol kaybı (MAPL):
# MAPL = 14 + 2 - 0.5 + 2 - (-132) = 149.5 dB
#
# Okumaya göre menzil (Friis / Hata!) tahmin:
# Şehir içi (3.5 yol kayıp exp): ~5 km
# Açık alan (2.0 exp): ~20 km

python3 -c "
import math
mapl_db = 149.5
freq_hz = 868e6
c = 3e8
# Serbest uzay yol kaybı: FSPL = (4*pi*d*f/c)^2
# d = (c/(4*pi*f)) * 10^(MAPL/20)
d = (c/(4*math.pi*freq_hz)) * 10**(mapl_db/20)
print(f'Serbest uzay menzil: {d/1000:.1f} km')
"
# Serbest uzay menzil: 187.3 km (pratik: 5-20 km çevre koşullarına göre)

ADR — Adaptive Data Rate

LoRaWAN ağ sunucusu, her end device'ın sinyal kalitesine göre SF ve TX gücünü otomatik ayarlar. Ağ kapsama alanına yakın cihazlar SF7 kullanarak hızlı ve duty cycle verimli çalışır; uzak cihazlar SF12 ile bağlantıyı sürdürür.

Bu bölümde

  • SF7→SF12: her adım ~2x menzil, ~2x daha yavaş veri hızı, ~2x daha uzun hava süresi
  • BW125 standart; BW500 US915 downlink kanalı
  • ADR ile ağ sunucusu SF ve güç otomatik optimize eder — duty cycle verimliliği artar

02 LoRaWAN Ağ Mimarisi

LoRaWAN yıldız-yıldız topolojisi kullanır: end device'lar tüm görülebilen gateway'lere broadcast yapar; gateway'ler paketleri IP üzerinden ağ sunucusuna iletir.

Mimari bileşenleri

End Device (sensör, aktüatör)
    │  LoRa RF (tek yönlü veya çift yönlü)
    ▼
Gateway (Packet Forwarder)
    │  UDP/MQTT — Semtech Packet Forwarder protokolü
    ▼
Network Server (TTN / ChirpStack / Helium)
    │  Deduplicate, MAC işleme, ADR, join server
    ▼
Application Server
    │  MQTT, HTTP webhook, gRPC
    ▼
Uygulama (dashboard, veritabanı, alert)

Gateway — Packet Forwarder

Semtech UDP PFKlasik protokol — UDP ile ağ sunucusuna paket iletir; güvensiz (şifreleme yok)
Basic StationSemtech'in modern gateway yazılımı — TLS WebSocket, LNS protokolü; önerilir
ChirpStack Gateway OSTam hazır Linux imajı — Raspberry Pi + RAK2245/RAK2287 kartı için

Paket deduplication

Bir end device'ın gönderdiği RF paketi birden fazla gateway tarafından alınabilir. Ağ sunucusu aynı paketin kopyalarını frame counter ve MIC (Message Integrity Code) ile tespit eder ve sadece bir kez uygulama sunucusuna iletir. Bu mekanizma aynı zamanda gateway redundancy sağlar.

YASAL ZORUNLULUK

EU868 bölgesinde duty cycle kısıtlaması yasal zorunluluktur (ETSI EN 300 220). %1 duty cycle, her 100 saniyelik pencerede en fazla 1 saniyelik hava süresi kullanılabileceği anlamına gelir. Bu kısıtlamayı aşmak cihazı yasa dışı kılar ve lisans gerektirebilir.

Bu bölümde

  • Yıldız-yıldız topoloji: cihaz → gateway (RF) → ağ sunucusu (IP) → uygulama sunucusu
  • Paket deduplication: birden fazla gateway aynı paketi alırsa ağ sunucusu bir kez işler
  • EU868: %1 duty cycle yasal zorunluluk; US915: FHSS ile duty cycle yok

03 Device Sınıfları

LoRaWAN cihazları üç sınıfa ayrılır. Sınıf seçimi downlink alım yeteneğini ve güç tüketimini doğrudan belirler.

Class A — ALOHA + 2 RX penceresi

Class A en temel ve en düşük güç tüketen sınıftır. Cihaz sadece bir uplink gönderdikten sonra iki kısa downlink penceresi açar. Başka zamanlarda tamamen uyku halindedir.

Uplink TX
    │ ─── TX ───▶
    │
    ├── RX1: uplink sonrası 1 saniye (aynı kanal, düşük SF)
    │
    └── RX2: uplink sonrası 2 saniye (869.525 MHz, SF12 EU868)

Class B — Beacon senkronizasyonu

Ağ sunucusu beacon yayınlar (her 128 saniyede bir); cihaz beacon'a kilitlenerek önceden belirlenen ping slotlarında uyku pencerelerini açar. Downlink mesaj gateway'den belirli zaman diliminde gönderilebilir.

│← 128 saniye beacon periyodu →│
[Beacon] .... [Ping] . [Ping] . [Ping] .... [Beacon]
               ↑RX       ↑RX      ↑RX
               cihaz bu slotlarda uyanır

Class C — Sürekli RX

Class C cihazlar TX sırasında dışında sürekli alım penceresinde bekler. Gecikme minimumdur ancak güç tüketimi yüksektir — şebeke veya güneş enerjisiyle beslenen cihazlar için uygundur.

SınıfDownlink gecikmeGüç tüketimiTipik kullanım
Class AUplink sonrası (dakika–saatler)En düşük — pil ömrü yıllarSensör, sayaç, takip
Class B~1–128 saniye (ping slot)OrtaAktüatör, akıllı sayaç
Class CMilisaniyeler (neredeyse anlık)Yüksek — sürekli RXAçma/kapama, röle, kontrol

Bu bölümde

  • Class A: her cihaz desteklemeli — temel zorunluluk; pil için en verimli
  • Class B: beacon ile senkron ping slot — deterministik downlink penceresi
  • Class C: sürekli dinleme — anlık downlink, şebeke gücü gerektirir

04 Aktivasyon Yöntemleri — OTAA ve ABP

LoRaWAN'da cihazların ağa katılmasının iki yolu vardır: OTAA (dinamik, güvenli) ve ABP (statik, kolay). Üretim cihazları için OTAA zorunludur.

OTAA — Over-The-Air Activation

DevEUI64-bit cihaz kimliği — fabrikada atanır, benzersiz, IEEE EUI-64 formatı
AppEUI / JoinEUI64-bit uygulama/join sunucu kimliği
AppKey128-bit AES root anahtarı — cihazda güvenli depolanır, ağa iletilmez
DevAddrJoin sonrası atanan 32-bit ağ adresi — her oturumda değişebilir
NwkSKeyJoin sonrası türetilen ağ oturum anahtarı — MAC katmanı şifreleme
AppSKeyJoin sonrası türetilen uygulama oturum anahtarı — payload şifreleme
Cihaz                    Join Server / NS
   │                           │
   │── JoinRequest ──────────▶ │
   │   (DevEUI, AppEUI,        │
   │    AppNonce, DevNonce)     │
   │                           │ AppKey ile doğrula
   │                           │ NwkSKey, AppSKey türet
   │◀─ JoinAccept ─────────── │
   │   (AppNonce, NetID,        │
   │    DevAddr, DLSettings,    │
   │    RxDelay)                │
   │                           │
   Artık DevAddr, NwkSKey, AppSKey mevcut
   Normal uplink/downlink başlar

ABP — Activation By Personalization

ABP'de tüm oturum parametreleri (DevAddr, NwkSKey, AppSKey) fabrikada veya sunucuda manuel olarak girilir. JoinRequest gönderilmez; cihaz doğrudan iletişime başlar.

GÜVENLİK UYARISI

ABP'de oturum anahtarları değişmez — cihaz ele geçirilirse tüm trafik şifresi çözülebilir. Frame counter sıfırlanırsa (güç kesimi, yeniden başlatma) replay saldırısı mümkündür. Üretim cihazları için ABP kullanmayın; OTAA ile her JoinRequest yeni oturum anahtarı üretir.

Frame counter yönetimi

c — frame_counter.c●
/* Frame counter kalıcı depolamada saklanmalı */
#include <stdint.h>

static uint32_t uplink_fcnt = 0;

/* Güvenli frame counter artırma — EEPROM/Flash'a yaz */
uint32_t get_next_fcnt(void) {
    uplink_fcnt++;
    /* EEPROM'a yaz — güç kesintisine karşı */
    eeprom_write_dword(FCNT_ADDR, uplink_fcnt);
    return uplink_fcnt;
}

/* Yeniden başlatmada EEPROM'dan oku */
void init_fcnt(void) {
    uplink_fcnt = eeprom_read_dword(FCNT_ADDR);
    /* Güvenlik marjı: en son değerden biraz ilerle */
    uplink_fcnt += 100;
    eeprom_write_dword(FCNT_ADDR, uplink_fcnt);
}

Bu bölümde

  • OTAA: JoinRequest → JoinAccept → dinamik NwkSKey/AppSKey türetimi — üretim standardı
  • ABP: statik anahtarlar — sadece test ve geliştirme için; güvenli değil
  • Frame counter kalıcı depolamada saklanmalı — replay saldırısı önleme

Python pySX127x kütüphanesi

python — lora_sender.py●
#!/usr/bin/env python3
"""
pySX127x ile LoRa paket gönderici
pip install pySX127x RPi.GPIO spidev
"""
import time
from SX127x.LoRa import LoRa
from SX127x.board_config import BOARD
from SX127x.constants import (MODE, BW, CODING_RATE, SF_BW_TBL)

BOARD.setup()
BOARD.reset()

class LoRaSender(LoRa):
    def __init__(self, verbose=False):
        super(LoRaSender, self).__init__(verbose)
        self.set_mode(MODE.SLEEP)
        self.set_dio_mapping([1, 0, 0, 0, 0, 0])  # DIO0 = TxDone

    def on_tx_done(self):
        self.set_mode(MODE.STDBY)
        self.clear_irq_flags(TxDone=1)
        print("TX Done")

def configure_and_send(lora, payload: bytes):
    lora.set_pa_config(pa_select=1, max_power=21, output_power=14)
    lora.set_freq(868.1)          # 868.1 MHz — EU868 kanal 0
    lora.set_bw(BW.BW125)
    lora.set_coding_rate(CODING_RATE.CR4_5)
    lora.set_spreading_factor(9)  # SF9
    lora.set_rx_crc(True)
    lora.set_low_data_rate_optim(False)

    lora.write_payload(list(payload))
    lora.set_mode(MODE.TX)
    print(f"Sending {len(payload)} bytes: {payload.hex()}")

if __name__ == "__main__":
    lora = LoRaSender(verbose=False)
    while True:
        payload = b"\x01\x02\x18\x2c"  # device_id + temp(24.44C)
        configure_and_send(lora, payload)
        time.sleep(60)  # duty cycle: SF9@BW125 ~370ms/60s = 0.6% < 1%

Bu bölümde

  • SX1276 SPI bağlantısı: MOSI/MISO/SCK/CS + RESET + DIO0 interrupt
  • Device Tree overlay ile Linux kernel'de SPI cihazı tanımlanır
  • pySX127x ile Python'da hızlı prototip; duty cycle hesabına dikkat edilmeli

Payload decoder — JavaScript

javascript — TTN payload decoder●
// TTN Application Server → Payload Formatters → Uplink → Custom JavaScript
function decodeUplink(input) {
    var bytes = input.bytes;
    return {
        data: {
            device_id:   bytes[0],
            temperature: ((bytes[1] << 8) | bytes[2]) / 100.0,  // °C
            humidity:    bytes[3] / 2.0,                         // %RH
            battery_mv:  ((bytes[4] << 8) | bytes[5]),           // mV
            rssi:        input.rxMetadata[0].rssi,
            snr:         input.rxMetadata[0].snr,
        },
        warnings: [],
        errors: []
    };
}

08 Pratik: Uçtan Uca LoRaWAN Pipeline

Adım 1 — End device: Arduino + LMIC

cpp — arduino_sensor.ino●
#include <lmic.h>
#include <hal/hal.h>
#include <DHT.h>

#define DHT_PIN 2
DHT dht(DHT_PIN, DHT22);

/* OTAA kimlik bilgileri — ChirpStack'ten al */
static const u1_t PROGMEM APPEUI[8] = { 0x00,0x00,0x00,0x00,0x00,0x00,0x00,0x00 };
static const u1_t PROGMEM DEVEUI[8] = { 0x08,0x07,0x06,0x05,0x04,0x03,0x02,0x01 };
static const u1_t PROGMEM APPKEY[16]= { 0xAA,0xBB,0xCC,0xDD,0xEE,0xFF,0x00,0x11,
                                         0x22,0x33,0x44,0x55,0x66,0x77,0x88,0x99 };

void os_getArtEui(u1_t* buf) { memcpy_P(buf, APPEUI, 8); }
void os_getDevEui(u1_t* buf) { memcpy_P(buf, DEVEUI, 8); }
void os_getDevKey(u1_t* buf) { memcpy_P(buf, APPKEY, 16); }

static uint8_t payload[4];

void do_send(osjob_t* j) {
    float temp = dht.readTemperature();
    float hum  = dht.readHumidity();

    int16_t temp_i = (int16_t)(temp * 100);
    uint8_t hum_i  = (uint8_t)(hum * 2);

    payload[0] = (temp_i >> 8) & 0xFF;
    payload[1] = temp_i & 0xFF;
    payload[2] = hum_i;
    payload[3] = 0x01;  /* frame type */

    LMIC_setTxData2(1, payload, sizeof(payload), 0);
}

void onEvent(ev_t ev) {
    if (ev == EV_JOINED) Serial.println("JOINED");
    if (ev == EV_TXCOMPLETE) {
        os_setTimedCallback(&sendjob,
            os_getTime() + sec2osticks(60), do_send);
    }
}

void setup() {
    dht.begin();
    os_init();
    LMIC_reset();
    LMIC_setAdrMode(1);  /* ADR etkin */
    do_send(&sendjob);
}

void loop() { os_runloop_once(); }

[backend]
type = "semtech_udp"

[backend.semtech_udp]
udp_bind = "0.0.0.0:1700"

[integration.mqtt]
event_topic_template    = "eu868/gateway/{{ .GatewayID }}/event/{{ .EventType }}"
command_topic_template  = "eu868/gateway/{{ .GatewayID }}/command/#"
server = "tcp://localhost:1883"
EOF

sudo systemctl enable --now chirpstack-gateway-bridge
