00 Üretim OTA Gereksinimleri

Sahada çalışan gömülü Linux cihazlar için güncelleme sistemi, tüketici elektroniğinden çok farklı kısıtlar altında çalışır. Bir güncelleme hatası cihazı kalıcı olarak kullanılamaz hale getirebilir.

01 SWUpdate Mimarisi

SWUpdate, CPIO arşiv formatında bir güncelleme paketi alır ve sw-description dosyasındaki talimatlara göre farklı handler'lar aracılığıyla cihaza uygular.

SWUpdate çalışma akışı

Güncelleme paketi (.swu)
    │
    ├── sw-description        (libconfig formatında talimatlar)
    ├── rootfs.ext4.gz        (yeni root filesystem)
    ├── kernel.itb            (FIT image — kernel + DTB)
    ├── u-boot.bin            (bootloader güncellemesi)
    └── postinstall.lua       (kurulum sonrası script)
         │
         ▼
SWUpdate daemon (swupdate)
    │
    ├── sw-description parse
    ├── İmza doğrula (OpenSSL / GPG)
    ├── SHA-256 checksum doğrula
    ├── Handler seç (raw, ubivol, shellscript, lua...)
    ├── Pasif partition'a yaz
    ├── Bootloader ortam güncellemesi
    └── Reboot → yeni sistem açılır

02 sw-description Dosyası

sw-description, SWUpdate paketinin merkezi yapılandırma dosyasıdır. libconfig sözdizimini kullanır ve hangi dosyanın nereye, nasıl yazılacağını tanımlar.

Tam örnek sw-description

sw-description — A/B bölümlü sistem●
software = {
    version = "2.1.0";
    description = "Production firmware v2.1.0";

    /* Donanım revizyonu kısıtlaması */
    hardware-compatibility = ["1.0", "1.1", "2.0"];

    /* images bölümü: ne nereye yazılacak */
    images: (
        {
            filename = "rootfs-v2.1.ext4.gz";
            type     = "raw";
            device   = "/dev/mmcblk0p3";   /* slot B */
            sha256   = "a3f9b21c...";       /* SHA-256 checksum */
            compressed = "zlib";
            installed-directly = false;
        },
        {
            filename = "kernel-v5.15.itb";
            type     = "raw";
            device   = "/dev/mmcblk0p5";   /* kernel B */
            sha256   = "7e4f23ab...";
        }
    );

    /* scripts bölümü: kurulum adımları */
    scripts: (
        {
            filename = "postinstall.lua";
            type     = "lua";
            /* 'pre' veya 'post' (varsayılan: post) */
        },
        {
            filename = "set_bootenv.sh";
            type     = "shellscript";
        }
    );

    /* bootloader ortam değişkenleri */
    bootenv: (
        {
            name  = "bootpart";
            value = "1";         /* pasif partition index güncellendi */
        },
        {
            name  = "upgrade_available";
            value = "1";         /* U-Boot watchdog değişkeni */
        }
    );
};

SWUpdate paketi oluşturma

bash●
# Tüm dosyaları bir dizinde topla
mkdir -p swu_package/
cp sw-description     swu_package/
cp rootfs-v2.1.ext4.gz swu_package/
cp kernel-v5.15.itb   swu_package/
cp postinstall.lua    swu_package/
cp set_bootenv.sh     swu_package/

# sw-description CPIO arşivinin ilk dosyası olmalı!
cd swu_package/
echo "sw-description" | cpio -H newc -o > ../update.swu
echo "rootfs-v2.1.ext4.gz kernel-v5.15.itb postinstall.lua set_bootenv.sh" \
  | tr ' ' '\n' | cpio -H newc -o --append -F ../update.swu

# Otomatik CPIO script
cat <<'SCRIPT' > mkswu.sh
#!/bin/bash
FILES="sw-description sw-description.sig rootfs-v2.1.ext4.gz kernel-v5.15.itb postinstall.lua set_bootenv.sh"
echo $FILES | tr ' ' '\n' | cpio -H newc -o > update.swu
echo "SWU boyutu: $(du -sh update.swu | cut -f1)"
SCRIPT
chmod +x mkswu.sh

# İmzalama (OpenSSL RSA)
openssl dgst -sha256 -sign private.pem \
  -out sw-description.sig sw-description

03 SWUpdate Lua Handler

SWUpdate'in Lua entegrasyonu, güncelleme sürecine özel mantık eklemenizi sağlar: doğrulama kontrolleri, servis durdurma/başlatma, veri taşıma, rollback tetikleme.

Lua handler API

lua — postinstall.lua●
-- SWUpdate Lua handler API
-- swupdate.notify(level, msg) ile log yaz
-- swupdate.error(msg)   → güncellemeyi iptal et
-- swupdate.progress(percent, msg) → ilerleme bildir

local function check_disk_space()
    -- /dev/mmcblk0p3 için yeterli alan var mı?
    local pipe = io.popen("df -B1 /mnt/passive | tail -1 | awk '{print $4}'")
    local avail = tonumber(pipe:read("*a"))
    pipe:close()

    local required = 512 * 1024 * 1024  -- 512 MB
    if avail and avail < required then
        swupdate.error(string.format(
            "Yetersiz disk alanı: %d MB mevcut, %d MB gerekli",
            avail / (1024*1024), required / (1024*1024)))
        return false
    end
    return true
end

local function stop_services()
    swupdate.notify(swupdate.INFOINT, "Servisler durduruluyor...")
    os.execute("systemctl stop my-sensor-daemon.service")
    os.execute("systemctl stop my-mqtt-bridge.service")
end

local function start_services()
    swupdate.notify(swupdate.INFOINT, "Servisler başlatılıyor...")
    os.execute("systemctl start my-sensor-daemon.service")
    os.execute("systemctl start my-mqtt-bridge.service")
end

local function migrate_config()
    -- v1 → v2 konfigürasyon taşıma
    local old_cfg = "/mnt/data/sensor.conf"
    local new_cfg = "/mnt/data/sensor_v2.conf"

    if not os.rename(old_cfg, new_cfg) then
        swupdate.notify(swupdate.WARNINT, "Config migration atlandı")
    end
end

local function verify_new_rootfs()
    -- Yeni rootfs bütünlük kontrolü (dm-verity hash)
    local ret = os.execute(
        "veritysetup verify /dev/mmcblk0p3 /dev/mmcblk0p7 "
        .. "$(cat /etc/verity-root-hash)")
    if ret ~= 0 then
        swupdate.error("Rootfs verity doğrulama başarısız!")
    end
end

-- Ana giriş noktası
function post_install(img)
    swupdate.progress(0, "Post-install başladı")

    if not check_disk_space() then return false end

    swupdate.progress(25, "Disk alanı yeterli")
    migrate_config()
    swupdate.progress(50, "Config taşıması tamamlandı")
    verify_new_rootfs()
    swupdate.progress(75, "Rootfs doğrulandı")
    stop_services()

    -- Bootloader ortamını ayarla
    os.execute("fw_setenv upgrade_available 1")
    os.execute("fw_setenv bootcount 0")

    swupdate.progress(100, "Post-install tamamlandı")
    swupdate.notify(swupdate.INFOINT, "Güncelleme uygulandı, yeniden başlatılıyor...")
    return true
end

sw-description'da Lua çağrısı

sw-description (scripts bölümü)●
scripts: (
    {
        filename = "postinstall.lua";
        type     = "lua";
        function = "post_install";  /* Lua fonksiyon adı */
    }
);

Bu bölümde

  • Lua handler: swupdate.notify/error/progress API ile güncelleme akışı kontrolü
  • Disk alanı kontrolü, config migration, dm-verity doğrulama — tipik handler görevleri
  • function = "post_install" ile sw-description'dan Lua fonksiyonu çağrılır

04 hawkBit Entegrasyonu

hawkBit, Eclipse Foundation bünyesindeki açık kaynaklı IoT güncelleme yönetim sunucusudur. SWUpdate'in suricatta modülü ile hawkBit DMF/DDI API üzerinden iletişim kurar.

hawkBit mimarisi

hawkBit Server (Spring Boot — Docker)
    │
    ├── DDI API (Device Direct Integration) — cihazlar burayı polling yapar
    │   └── REST: /controller/v1/{tenantId}/controller/{deviceId}
    │
    ├── DMF API (Device Management Federation) — AMQP/RabbitMQ
    │
    └── Management API — Web UI, cihaz yönetimi, rollout oluşturma
         └── REST: /rest/v1/targets, /rest/v1/distributionsets

bash●
# hawkBit modunda SWUpdate daemon
swupdate -f /etc/swupdate/swupdate.cfg \
  -u "-t DEFAULT -u http://hawkbit.example.com:8080 -i device-001" \
  &

# Veya systemd servisi
cat <<'EOF' | sudo tee /etc/systemd/system/swupdate.service
[Unit]
Description=SWUpdate OTA daemon
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
ExecStart=/usr/bin/swupdate \
  -f /etc/swupdate/swupdate.cfg \
  -u "-t DEFAULT -u http://hawkbit.example.com:8080 -i device-001"
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
EOF
sudo systemctl enable --now swupdate
ROLLOUT YÖNETİMİ

hawkBit'te "Rollout" oluşturarak güncellemeyi kademeli uygulayabilirsiniz: cihaz filtresi belirleyin (ör. hw-revision=2.0 olanlar), başlangıç grubu %5 seçin, başarı oranı %95'in üzerindeyse otomatik bir sonraki gruba geç. Bu kademeli güncelleme, sorunlu bir sürümün tüm filoya yayılmasını önler.

Bu bölümde

  • hawkBit: DDI API polling, DMF AMQP push, Management UI — tam OTA yönetimi
  • suricatta modülü SWUpdate'in hawkBit istemcisi — yapılandırma dosyasından kontrol
  • Rollout yönetimi ile kademeli güncelleme — %5 → %20 → %100 aşamalı

05 RAUC Mimarisi

RAUC (Robust Auto-Update Controller), slot tabanlı bir güncelleme framework'üdür. Bundle dosyası, manifest ve imzalama ile temiz, bütünleşik bir yaklaşım sunar.

RAUC temel kavramlar

BundleSquashFS içinde manifest + görüntü dosyaları — imzalanmış, bütünleşik paket
SlotGüncellenebilir depolama birimi — bir partition, MTD volume veya dizin
Slot classAynı işlevi gören slot'ların grubu (rootfs, kernel, bootloader)
Bootloader adaptationRAUC'un U-Boot/GRUB/barebox'a boot slot bilgisi yazması
casyncContent-Addressable Chunked Archive — delta güncelleme için
dm-verityRAUC dm-verity hash tablosu ile rootfs bütünlüğünü garanti eder

RAUC sistem konfigürasyonu

ini — /etc/rauc/system.conf●
[system]
compatible=MyProduct-v2
bootloader=uboot

[keyring]
path=/etc/rauc/keyring.pem   ; Doğrulama sertifikası

[slot.rootfs.0]
device=/dev/mmcblk0p2
type=ext4
bootname=A
parent=

[slot.rootfs.1]
device=/dev/mmcblk0p3
type=ext4
bootname=B
parent=

[slot.kernel.0]
device=/dev/mmcblk0p5
type=raw
parent=rootfs.0

[slot.kernel.1]
device=/dev/mmcblk0p6
type=raw
parent=rootfs.1

[slot.bootloader.0]
device=/dev/mmcblk0p1
type=raw

RAUC komutları

bash●
# Güncelleme uygula
rauc install /path/to/update.raucb

# Slot durumlarını göster
rauc status
# === System Info ===
# Compatible: MyProduct-v2
# Booted from: rootfs.0 (A)
# === Slot States ===
# rootfs.0 (A) [booted]   version=2.0.1
# rootfs.1 (B) [inactive] version=2.1.0

# Bundle içeriğini incele
rauc info update.raucb

# Aktif slot'u işaretle (test başarılı)
rauc mark-good

# Rollback: aktif slot'u bad işaretle → reboot sonrası diğerine geç
rauc mark-bad booted

Bu bölümde

  • RAUC: slot tabanlı güncelleme — A/B rootfs, kernel, bootloader slot'ları
  • system.conf ile slot'lar tanımlanır; rauc install ile güncelleme uygulanır
  • rauc mark-good/bad ile bootloader rollback mekanizması tetiklenir

06 RAUC Bundle Oluşturma

RAUC bundle'ı SquashFS formatında imzalanmış bir dosyadır. manifest.raucm güncelleme içeriğini, imzalama ise güvenliği tanımlar.

manifest.raucm dosyası

ini — manifest.raucm●
[update]
compatible=MyProduct-v2
version=2.1.0
description=Production firmware v2.1.0
build=2026-04-12T10:00:00Z

[image.rootfs]
filename=rootfs.ext4.img
sha256=b3e8f1c2d4a59678901234567890abcdef123456789012345678901234567890
size=268435456

[image.kernel]
filename=kernel.itb
sha256=7f3a9b1e2c4d5678901234567890fedcba987654321098765432109876543210
size=8388608

[hooks]
filename=rauc_hooks.sh

Bundle imzalama — sertifika zinciri

bash — PKI kurulumu●
# CA sertifika oluşturma (bir kez)
openssl genrsa -out ca.key.pem 4096
openssl req -new -x509 -days 3650 -key ca.key.pem \
  -out ca.cert.pem \
  -subj "/CN=MyProduct CA/O=MyCompany/C=TR"

# Signing sertifikası (düzenli yenilenen)
openssl genrsa -out signing.key.pem 2048
openssl req -new -key signing.key.pem -out signing.csr \
  -subj "/CN=RAUC Signing/O=MyCompany/C=TR"
openssl x509 -req -days 365 -in signing.csr \
  -CA ca.cert.pem -CAkey ca.key.pem \
  -CAcreateserial -out signing.cert.pem

# Bundle oluştur ve imzala
mkdir -p bundle_dir/
cp rootfs.ext4.img bundle_dir/
cp kernel.itb      bundle_dir/
cp manifest.raucm  bundle_dir/
cp rauc_hooks.sh   bundle_dir/

rauc bundle \
  --cert=signing.cert.pem \
  --key=signing.key.pem \
  bundle_dir/ \
  update-v2.1.0.raucb

# Bundle boyutu ve bilgisi
ls -sh update-v2.1.0.raucb
rauc info --keyring=ca.cert.pem update-v2.1.0.raucb

casync ile delta bundle

bash — delta güncelleme●
# casync kurulumu
sudo apt install casync

# v2.0 → v2.1 delta oluştur
casync make \
  --store=v2.1.castr \
  rootfs-v2.1.ext4.img \
  rootfs-v2.1.caibx        # index dosyası

# manifest.raucm'da casync referansı
# [image.rootfs]
# filename=rootfs.caibx
# casync-store=http://update.example.com/store/v2.1.castr

# Delta güncelleme: sadece değişen chunk'lar indirilir
# Tipik rootfs güncellemesi: 256 MB yerine 20-40 MB (yaklaşık %85 tasarruf)

dm-verity ile rootfs bütünlük koruması

bash●
# rootfs için verity hash tablosu oluştur
veritysetup format rootfs.ext4.img rootfs-verity.img
# Root hash: a3f9b21c... (manifest'e ekle)

# manifest.raucm'da verity aktivasyonu
# [image.rootfs]
# verity-hash=a3f9b21c...

# RAUC kurulum sonrasında cihazda verity etkinleştirir
# /dev/mapper/rootfs-verified cihazından boot edilir
# Herhangi bir bit bozulması dm-verity tarafından tespit edilir

Bu bölümde

  • manifest.raucm: compatible, version, görüntü dosyaları ve SHA-256 checksum'ları
  • PKI: CA sertifika + imzalama sertifikası — bundle kriptografik imzalanır
  • casync delta: değişen chunk'lar — %70–85 bant genişliği tasarrufu

07 A/B Partition Stratejisi

A/B (veya double-copy) güncelleme stratejisi, çalışan sistemi (aktif slot) hiç dokunmadan pasif slota yeni sürümü yazar ve reboot sonrası atomik geçiş sağlar.

Partition düzeni

PartitionİçerikBoyut
/dev/mmcblk0p1U-Boot / bootloader16 MB
/dev/mmcblk0p2rootfs A (aktif)512 MB
/dev/mmcblk0p3rootfs B (pasif)512 MB
/dev/mmcblk0p4kernel A32 MB
/dev/mmcblk0p5kernel B32 MB
/dev/mmcblk0p6data (kalıcı)Kalan alan

U-Boot bootcount mekanizması

bash — U-Boot ortam●

U-Boot boot script — A/B seçimi

uboot — bootcmd (u-boot script)●
# /boot/boot.scr veya U-Boot ortam bootcmd
test -n "${BOOT_ORDER}" || setenv BOOT_ORDER "A B"
test -n "${BOOT_A_LEFT}" || setenv BOOT_A_LEFT 3
test -n "${BOOT_B_LEFT}" || setenv BOOT_B_LEFT 3

for BOOT_SLOT in "${BOOT_ORDER}"; do
    if test "${BOOT_SLOT}" = "A"; then
        if test ${BOOT_A_LEFT} -gt 0; then
            setexpr BOOT_A_LEFT ${BOOT_A_LEFT} - 1
            setenv LOADPART 4     # partition 4 = kernel A
            setenv ROOTPART 2     # partition 2 = rootfs A
            break
        fi
    elif test "${BOOT_SLOT}" = "B"; then
        if test ${BOOT_B_LEFT} -gt 0; then
            setexpr BOOT_B_LEFT ${BOOT_B_LEFT} - 1
            setenv LOADPART 5     # partition 5 = kernel B
            setenv ROOTPART 3     # partition 3 = rootfs B
            break
        fi
    fi
done

saveenv

load mmc 0:${LOADPART} ${kernel_addr_r} /boot/fitImage
bootm ${kernel_addr_r}#conf@${LOADPART}

RAUC ile otomatik rollback

bash — systemd service●
# Sistem açıldıktan sonra sağlık kontrolü ve mark-good
cat <<'EOF' | sudo tee /etc/systemd/system/rauc-mark-good.service
[Unit]
Description=RAUC mark slot good after successful boot
After=network-online.target myapp.service
Wants=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/bin/rauc mark-good
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl enable rauc-mark-good.service

# Uygulama sağlık testi başarısız olursa:
# rauc mark-bad booted → reboot → U-Boot önceki slot'a geçer

Bu bölümde

  • A/B partition: pasif slota yaz, reboot, başarılı ise mark-good, başarısız ise rollback
  • U-Boot BOOT_A_LEFT/BOOT_B_LEFT sayaçları bootcount mekanizmasını uygular
  • rauc-mark-good.service systemd servisi başarılı boot'ta otomatik onay verir

GitHub Actions CI/CD pipeline

yaml — .github/workflows/ota-release.yml●
name: OTA Release Pipeline
on:
  push:
    tags:
      - 'v*.*.*'

jobs:
  yocto-build:
    runs-on: self-hosted   # Yocto büyük; self-hosted runner gerektirir
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive

      - name: Setup Yocto
        run: |
          source poky/oe-init-build-env build
          bitbake myproduct-bundle

      - name: Collect artifacts
        run: |
          VERSION=$(echo "${GITHUB_REF#refs/tags/v}")
          cp build/tmp/deploy/images/*/myproduct-bundle-*.raucb \
             myproduct-v${VERSION}.raucb

      - name: Upload to hawkBit
        env:
          HAWKBIT_URL: ${{ secrets.HAWKBIT_URL }}
          HAWKBIT_TOKEN: ${{ secrets.HAWKBIT_TOKEN }}
          VERSION: ${{ github.ref_name }}
        run: |
          # Distribution set oluştur
          DS_ID=$(curl -s -X POST "${HAWKBIT_URL}/rest/v1/distributionsets" \
            -H "Authorization: GatewayToken ${HAWKBIT_TOKEN}" \
            -H "Content-Type: application/json" \
            -d "[{\"name\":\"MyProduct\",\"version\":\"${VERSION}\",
                  \"type\":\"app\"}]" | jq -r '.[0].id')

          # Bundle'ı yükle
          curl -X POST \
            "${HAWKBIT_URL}/rest/v1/softwaremodules/${DS_ID}/artifacts" \
            -H "Authorization: GatewayToken ${HAWKBIT_TOKEN}" \
            -F "file=@myproduct-${VERSION}.raucb"

          echo "hawkBit DS ID: ${DS_ID}"

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          files: myproduct-*.raucb
          generate_release_notes: true
SONUÇ

Tamamlanan pipeline: GitHub'a yeni tag push edildiğinde Yocto otomatik build başlar, RAUC bundle imzalanır, hawkBit sunucusuna yüklenir. hawkBit'te rollout oluşturularak %5 → %20 → %100 kademeli dağıtım yapılır. Cihazlar güncellemeyi alır, rauc-mark-good.service başarılı boot'u onaylar; başarısızlıkta U-Boot bootcount sayacı devreye girer ve önceki sürüme döner.

Bu bölümde

  • meta-rauc: inherit rauc-bundle ile Yocto'da RAUC bundle tarifi
  • GitHub Actions: tag push → Yocto build → bundle imzala → hawkBit yükle → release
  • Kademeli rollout hawkBit üzerinden; U-Boot bootcount ile otomatik rollback
