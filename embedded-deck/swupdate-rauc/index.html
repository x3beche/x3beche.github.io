00 Üretim OTA gereksinimleri

Bir cihaz sahaya çıktığında güncelleme sürecinde tek başarısızlık bile cihazı kalıcı olarak işlevsiz bırakabilir. Neden cp veya scp + reboot yetmez?

Klasik yöntemlerin sorunu

Geliştirme ortamında bir rootfs'i scp ile kopyalamak ve sistemi yeniden başlatmak kabul edilebilir görünür. Ancak üretimde bu yaklaşım şu senaryolarda felakete yol açar: güncelleme sırasında güç kesilirse dosya sistemi yarı yazılı kalır; yeni firmware hatalıysa geri dönüş mekanizması yoktur; güncelleme bütünlüğü doğrulanmadan uygulanırsa kötü amaçlı yazılım yüklenilebilir.

Basit cp yaklaşımı (SORUNLU):
  device ─── scp rootfs.tar.gz ──→ cihaz
               ↓
           tar -xzf /  (yarıda güç kesilirse → brick)
               ↓
           reboot      (hatalı firmware → kurtarılamaz)

Üretim OTA (DOĞRU):
  hawkBit / CDN
      │
      ▼
  swupdate / rauc
      │  ├─ imza doğrula   (X.509 / CMS)
      │  ├─ hash kontrol   (sha256 + verity)
      │  ├─ pasif slota yaz (A/B — aktif slot dokunulmaz)
      │  ├─ bootloader güncelle (U-Boot env)
      │  └─ watchdog / bootcount → otomatik rollback
      ▼
  yeni slot aktif — boot başarılı → commit
                  — boot başarısız → eski slot

Temel üretim gereksinimleri

Atomik güncelleme Ya tüm güncelleme uygulanır ya da hiçbiri — ara durum yoktur. A/B partition bu garantiyi sağlar; yazma sırasında aktif slot dokunulmadan kalır.
Otomatik rollback Boot sayacı (bootcount) belirli bir eşiği geçerse bootloader otomatik olarak önceki çalışan slota döner. Uygulama katmanının onayı olmadan yeni firmware kabul edilmez.
Kriptografik imzalama Güncelleme paketi X.509 sertifikasıyla imzalanır; cihaz üzerindeki kök CA ile doğrulanır. MITM ve sahte güncelleme saldırılarını engeller.
Güç kesintisi güvenliği Yazma işlemi pasif (B) slota yapılır. Güç kesilse bile aktif (A) slot sağlam kalır; cihaz bir sonraki boot'ta eski firmware ile çalışmaya devam eder.
Bütünlük doğrulama SHA-256 hash ve dm-verity ile her blok okumada rootfs bütünlüğü doğrulanır. Depolama katmanındaki sessiz bozulmalar (silent corruption) tespit edilir.
Delta güncelleme Tam image yerine yalnızca değişen bloklar aktarılır. RAUC casync veya SWUpdate rdiff handler ile bant genişliği %60–90 azalır; düşük bant genişlikli 4G/LTE bağlantılar için kritiktir.

NOT

Endüstriyel ve medikal cihazlarda OTA güncelleme süreci genellikle IEC 62443 veya FDA 21 CFR Part 11 gibi düzenleyici gereksinimler kapsamındadır. Kriptografik imzalama ve denetim kaydı (audit log) bu uyum açısından zorunlu bileşenlerdir; SWUpdate ve RAUC her ikisini de destekler.

Bu bölümde

  • Basit cp/scp güç kesintisinde brick riski taşır — A/B slot bu problemi yapısal olarak ortadan kaldırır
  • Üretim OTA: atomiklik + imza + rollback + bütünlük — dördü birlikte zorunlu
  • SWUpdate esnek handler sistemi, RAUC katı slot semantiği — farklı güçlü yanlarıyla birbirini tamamlar

01 SWUpdate mimarisi

SWUpdate, gömülü Linux için modüler bir güncelleme framework'üdür. Daemon modunda çalışır; handler sistemi sayesinde her kaynak tipi için özelleştirilebilir davranış sunar.

Kurulum ve daemon başlatma

bash — SWUpdate kurulum ve başlatma●
# Bağımlılıklar (Debian/Ubuntu)
sudo apt-get install -y \
  libconfig-dev libssl-dev libz-dev \
  libjson-c-dev libcurl4-openssl-dev \
  libmtd-dev libubi-dev lua5.3-dev

# Kaynak derleme
git clone https://github.com/sbabic/swupdate
cd swupdate
make defconfig
make menuconfig   # Bootloader support, Lua, hawkBit etkinleştir
make -j$(nproc)

# Daemon başlatma
swupdate \
  -f /etc/swupdate/swupdate.cfg \
  -e "stable,default" \
  -k /etc/swupdate/pubkey.pem \
  -l 3

# systemd entegrasyonu
systemctl enable --now swupdate.service

# Lokal .swu dosyasından güncelleme uygula (test)
swupdate -i firmware-2.4.1.swu \
         -k /etc/swupdate/pubkey.pem
NOT

SWUpdate güncelleme paketini doğrudan bir .swu dosyasından (CPIO arşivi) işler. Paket hem sw-description konfigürasyon dosyasını hem de güncellenecek image'ları içerir. -k ile belirtilen kamu anahtarı olmadan imza doğrulama açıksa hiçbir paket uygulanmaz; bu davranış kasıtlı olarak değiştirilemez.

02 sw-description dosyası

sw-description, SWUpdate'in her güncelleme paketi içindeki merkezi konfigürasyon dosyasıdır. libconfig sözdizimiyle yazılır; hangi image'ın nereye nasıl yazılacağını ve hangi bootloader değişkenlerinin güncelleneceğini tanımlar.

Tam sw-description örneği (A/B eMMC)

sw-description — libconfig syntax●
// sw-description — SWUpdate güncelleme manifestosu
software =
{
    version = "2.4.1";
    name    = "myproduct-firmware";

    // Donanım revizyonu kısıtlaması — uyumsuz HW reddedilir
    hardware-compatibility = ["1.0", "1.1", "2.0"];

    // "stable" grup, "default" seçim
    stable: {
        default: {

            images: (
                {
                    filename   = "rootfs-B.ext4";
                    type       = "raw";
                    device     = "/dev/mmcblk0p3";  // B slotu
                    sha256     = "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855";
                    compressed = "zstd";
                },
                {
                    filename   = "kernel-fitimage.itb";
                    type       = "raw";
                    device     = "/dev/mmcblk0p2";  // kernel partition
                    sha256     = "a4b2c8d1e5f378904b2c1d4e6f8a0b2c4d6e8f0a2b4c6d8e0f2a4b6c8d0e2f4";
                }
            );

            scripts: (
                {
                    filename = "pre-install.sh";
                    type     = "shellscript";
                    data     = "pre";   // image'lardan ÖNCE çalışır
                },
                {
                    filename = "post-install.sh";
                    type     = "shellscript";
                    data     = "post";  // tüm image'lardan SONRA
                }
            );

            bootenv: (
                {
                    name  = "rootfs_active";
                    value = "B";
                },
                {
                    name  = "bootcount";
                    value = "0";
                },
                {
                    name  = "upgrade_available";
                    value = "1";
                }
            );
        };
    };
};

sw-description anahtar alanları

version Güncelleme paketi sürümü — hawkBit bu değeri kullanarak mevcut cihaz sürümüyle karşılaştırır ve güncelleme gerekip gerekmediğine karar verir.
hardware-compatibility Cihaz üzerindeki HW revizyonu bu listede yoksa güncelleme reddedilir. Farklı PCB revizyonlarına yanlış firmware yüklenmesini önler; saha kazalarının temel önlemi.
images[].type raw, ubivol, rdiff, lua, shellscript — hangi handler'ın bu image için çağrılacağını belirler.
images[].sha256 Arşivden çıkarılan dosyanın SHA-256 hash'i. Eşleşmezse yazma işlemi hiç başlamaz; aktarım sırasındaki bozulmalar burada yakalanır.
images[].compressed zstd, gzip, zlib — SWUpdate yazarken on-the-fly açar. 512 MB rootfs, zstd ile 120–150 MB'a sıkışır; eMMC yazma süresi de kısalır.
bootenv[] U-Boot environment değişkenleri — güncelleme sonrası bootloader'ın yeni slotu seçip bootcount mekanizmasını devreye alması için zorunludur.

.swu paketi oluşturma ve imzalama

bash — .swu CPIO arşivi oluşturma●
# .swu dosyası CPIO arşividir; sw-description ilk sıraya gelmelidir
echo "sw-description"      | cpio -H newc -o >  firmware-2.4.1.swu
echo "sw-description.sig"  | cpio -H newc -o -A >> firmware-2.4.1.swu
echo "rootfs-B.ext4"       | cpio -H newc -o -A >> firmware-2.4.1.swu
echo "kernel-fitimage.itb" | cpio -H newc -o -A >> firmware-2.4.1.swu
echo "pre-install.sh"      | cpio -H newc -o -A >> firmware-2.4.1.swu
echo "post-install.sh"     | cpio -H newc -o -A >> firmware-2.4.1.swu

# İmzalama (CMS/PKCS#7 — DER formatı)
openssl cms -sign \
    -in      sw-description \
    -out     sw-description.sig \
    -signer  signing.crt \
    -inkey   signing.key \
    -outform DER \
    -nosmimecap -binary

# Doğrulama testi (cihaza göndermeden önce)
swupdate -k ca.pem -c -i firmware-2.4.1.swu
# "Valid" çıkması beklenir

03 SWUpdate Lua handler

Lua handler, standart handler'ların yetersiz kaldığı durumlarda özel doğrulama, cihaz hazırlama veya post-install mantığı eklemenizi sağlar. Tam Lua 5.3 ortamı gömülü olarak gelir.

Lua handler API

swupdate.notify(level, msg) Log mesajı gönderir. Seviyeler: swupdate.TRACE, swupdate.DEBUG, swupdate.INFO, swupdate.WARN, swupdate.ERROR.
swupdate.error(msg) Güncellemeyi hata mesajıyla iptal eder. Bu çağrıdan sonra SWUpdate tüm işlemi geri alır.
swupdate.progress(pct) 0–100 arası ilerleme yüzdesi raporlar. hawkBit arayüzünde canlı olarak görünür.
image.filename, image.version Handler'a iletilen image metadata'sına erişim. sw-description'daki tüm alanlar Lua tablosu olarak sunulur.

Tam Lua handler örneği — pre-install doğrulama

validate-hw.lua — pre-install doğrulama handler●
-- SWUpdate Lua handler: HW revizyon + disk alan kontrolü
local swupdate = require("swupdate")

function install_update(image)

    -- 1. Donanım revizyonunu cihaz ağacından oku
    local hw_rev_file = io.open(
        "/proc/device-tree/chosen/hw-revision", "r")
    if not hw_rev_file then
        swupdate.error("HW revision dosyasi okunamiyor")
        return 1
    end
    local hw_rev = hw_rev_file:read("*all"):gsub("%s+", "")
    hw_rev_file:close()

    swupdate.notify(swupdate.INFO,
        "Cihaz HW revision: " .. hw_rev)

    -- 2. Desteklenen revizyonları doğrula
    local supported = {
        ["1.0"] = true,
        ["1.1"] = true,
        ["2.0"] = true
    }
    if not supported[hw_rev] then
        swupdate.error("Desteklenmeyen HW revision: " .. hw_rev ..
            " — bu firmware bu donanim icin degil")
        return 1
    end

    -- 3. /data partition boş alan kontrolü (min 256 MB)
    local df = io.popen(
        "df -m /data | awk 'NR==2{print $4}'")
    local free_mb = tonumber(df:read("*all"))
    df:close()

    if free_mb == nil or free_mb < 256 then
        swupdate.error("Yetersiz disk alani: " ..
            tostring(free_mb) .. "MB (min 256MB gerekli)")
        return 1
    end

    swupdate.progress(50)
    swupdate.notify(swupdate.INFO,
        "On dogrulama basarili — kurulum devam ediyor")

    return 0  -- 0 = başarı; 1 = hata → güncelleme iptal
end

sw-description'da Lua handler referansı

    {

        type     = "lua";

DİKKAT

Lua handler'dan os.execute() veya io.popen() çağrıları yapıldığında çıktı otomatik olarak swupdate log'una yazılmaz. Kritik komutların çıkış kodunu her zaman kontrol edin ve hata durumunda swupdate.error() çağırın. Bu yapılmazsa komut başarısız olsa bile güncelleme başarılı sayılır — sessiz hata en tehlikeli senaryodur.

05 RAUC mimarisi

RAUC (Robust Auto-Update Controller), sıkı güvenlik ve slot semantiği üzerine tasarlanmış bir OTA framework'üdür. Bootloader entegrasyonu ve kriptografik imzalama birinci sınıf özelliklerdir.

RAUC temel kavramlar

Bundle (.raucb) Güncelleme paketi — SquashFS imajı içinde manifest.raucm + slot image'ları. X.509 ile imzalanır, verity formatında dm-verity hash ağacı içerir.
Slot Güncellenebilir sistem bileşeni (rootfs, kernel, bootloader, appfs). Her slot bir blok cihazı, MTD partisyonu veya UBIFS volume'üdür.
Slot Group Birlikte güncellenen slot seti — rootfs.0 + kernel.0 = A grubu; rootfs.1 + kernel.1 = B grubu. Grup üyeliği parent= direktifi ile tanımlanır.
Bootloader adaptation U-Boot, GRUB, Barebox ve EFI desteklenir. Aktif slot değiştirme ve bootcount yönetimi bootloader'a özgü API üzerinden yapılır; abstraction katmanı RAUC sağlar.
casync delta güncelleme Content-Addressable Sync — büyük image'ları sabit boyutlu chunk'lara böler, yalnızca değişen chunk'ları indirir. 500 MB rootfs için 30–80 MB indirme tipik.

compatible alanı bundle'daki manifest.raucm ile birebir eşleşmelidir. Eşleşmezse RAUC bundle'ı yanlış cihaza yüklendiğini tespit eder ve kurulumu reddeder. Bu kontrol sertifika doğrulamasından önce gerçekleşir ve ekstra bir güvenlik katmanı sağlar.

# Test ortamı için self-signed sertifika (üretimde HSM kullanın)
openssl req -x509 \
    -newkey ec:<(openssl ecparam -name prime256v1) \
    -keyout signing.key.pem \
    -out    signing.cert.pem \
    -days  3650 \
    -subj  "/CN=RAUC Signing/O=MyCompany" \
    -nodes

# Bundle oluştur ve imzala

    --cert    signing.cert.pem \
    --key     signing.key.pem \
    --keyring ca.cert.pem \
    bundle-staging/ \
    firmware-2.4.1.raucb

07 A/B partition yönetimi

A/B güncelleme stratejisinin kalbi, bootloader ile RAUC arasındaki bootcount mekanizmasıdır. Bu mekanizma hiç tamamlanmamış gibi davranan hatalı firmware'ı otomatik olarak yakalar ve önceki sürüme döner.

U-Boot bootcount mekanizması — adım adım

  rauc install tamamlandı
         │
         ▼
  U-Boot env yazıldı:
    rauc.slot         = "B"
    bootcount         = 0
    upgrade_available = 1
         │
         ▼  Power on
  U-Boot boot scripti:
    bootcount += 1        (şu an: 1)
    bootlimit  = 3
    if bootcount > bootlimit → rollback A slotuna
         │
         ▼  Linux başarıyla açıldı
  uygulama health check servisinden onay
         │
         ▼
  rauc mark-good current
    → bootcount = 0, upgrade_available = 0
    (artık rollback tetiklenemez)

  ─────────────────────────────────────────────
  SENARYO: Kernel panic / uygulama başlamıyor
  ─────────────────────────────────────────────
  Boot 1: bootcount = 1 → panic
  Boot 2: bootcount = 2 → panic
  Boot 3: bootcount = 3 → panic
  Boot 4: bootcount (4) > bootlimit (3) →
    U-Boot: rauc.slot = "A", bootcount = 0
    → önceki çalışan firmware ile boot

U-Boot bootcmd konfigürasyonu

u-boot-env — A/B bootcmd scripti●
# U-Boot ortam değişkenleri — /etc/u-boot-initial-env
bootlimit=3
bootdelay=2
upgrade_available=0
rauc_active_slot=a

# Slot başlatma argümanları
rauc_slot_a_args=setenv bootargs \
  root=/dev/mmcblk0p2 rw rootfstype=ext4 \
  rauc.slot=A quiet

rauc_slot_b_args=setenv bootargs \
  root=/dev/mmcblk0p4 rw rootfstype=ext4 \
  rauc.slot=B quiet

# Ana boot komutu
bootcmd=
  if test ${upgrade_available} = 1; then
    setexpr bootcount ${bootcount} + 1;
    saveenv;
    if test ${bootcount} > ${bootlimit}; then
      echo "RAUC: bootcount asimi rollback A";
      setenv rauc_active_slot a;
      setenv upgrade_available 0;
      setenv bootcount 0;
      saveenv;
    fi;
  fi;
  if test ${rauc_active_slot} = b; then
    run rauc_slot_b_args;
    load mmc 0:3 ${kernel_addr_r} fitimage.itb;
  else
    run rauc_slot_a_args;
    load mmc 0:1 ${kernel_addr_r} fitimage.itb;
  fi;
  bootm ${kernel_addr_r};

RAUC slot yönetim komutları

bash — RAUC slot API kullanımı●
# Tüm slot durumlarını göster
rauc status
# === System Info ===
# Compatible:   myproduct-v1
# Booted from:  rootfs.0 (A)
#
# === Slot States ===
# [rootfs.0] bootname:A  status:booted    version:2.3.0
# [rootfs.1] bootname:B  status:inactive  version:2.4.1

# Pasif slotu aktif işaretle (başarılı kurulum sonrası)
rauc mark-active other

# Mevcut boot'u başarılı onayla — bootcount sıfırlanır
rauc mark-good current

# Belirli slotu hatalı işaretle — sonraki boot'ta atlanır

# Manuel rollback — A slotuna dön

systemctl reboot

Sağlık kontrol servisi örneği

rauc-mark-good.service — systemd health check●
[Unit]
Description=RAUC Mark Good — firmware onay servisi
After=network-online.target myapp.service
Wants=network-online.target
Requires=myapp.service

[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/bin/rauc-health-check.sh

[Install]
WantedBy=multi-user.target
bash — /usr/bin/rauc-health-check.sh●
#!/bin/bash
# Uygulama ve kritik servisleri doğrula, sonra mark-good çağır

# Kritik servislerin çalıştığını kontrol et
for svc in myapp.service networking.service; do
    if ! systemctl is-active --quiet "$svc"; then
        logger -t rauc-health "FAIL: $svc çalışmıyor — rollback bekleniyor"
        exit 1   # mark-good çağrılmaz → bootcount artmaya devam eder
    fi
done

# Temel connectivity testi
if ! curl --silent --max-time 5 http://health.local/ping > /dev/null; then
    logger -t rauc-health "FAIL: network health check başarısız"
    exit 1
fi

# Her şey tamam — firmware'i onayla
rauc mark-good current
logger -t rauc-health "OK: firmware 2.4.1 başarıyla onaylandı"
DİKKAT

rauc mark-good komutunun çağrılması için sistemin gerçekten sağlıklı çalıştığını doğrulamanız gerekir. Sadece boot tamamlandı diye değil; kritik uygulama servisleri, ağ bağlantısı ve iş mantığının çalıştığı onaylandıktan sonra çağırın. Erken mark-good çağrısı rollback mekanizmasını devre dışı bırakır ve bozuk firmware ile cihazın sahada kalmasına yol açar.

meta-rauc katmanını kullanan, hawkBit'e bundle yükleyen ve GitLab CI'da otomatikleştirilen uçtan uca bir üretim pipeline'ı. Build'den cihaza dağıtıma kadar tüm adımlar.

GitLab CI pipeline

.gitlab-ci.yml — Yocto → RAUC bundle → hawkBit●
stages:
  - build
  - bundle
  - test
  - deploy

variables:
  YOCTO_MACHINE: myproduct
  BUNDLE_VERSION: "${CI_COMMIT_TAG:-${CI_COMMIT_SHORT_SHA}}"

## ─── Yocto build ───────────────────────────────────────────
yocto:build:
  stage: build
  tags: [yocto-runner]
  script:
    - source oe-init-build-env build
    - bitbake myproduct-image
  artifacts:
    paths:
      - build/tmp/deploy/images/${YOCTO_MACHINE}/
    expire_in: 7 days
  cache:
    key: "yocto-sstate-${CI_COMMIT_REF_SLUG}"
    paths: [build/sstate-cache/]

## ─── RAUC bundle imzalama ──────────────────────────────────
rauc:bundle:
  stage: bundle
  needs: [yocto:build]
  script:
    - # Manifest oluştur
    - |
      cat > manifest.raucm <<EOF
      [update]
      compatible=myproduct-v1
      version=${BUNDLE_VERSION}
      build=${CI_PIPELINE_ID}
      [bundle]
      format=verity
      [image.rootfs]
      filename=rootfs.ext4.zst
      EOF
    - cp build/tmp/deploy/images/${YOCTO_MACHINE}/rootfs.ext4.zst .
    - # Vault'tan sertifika al (HSM üzerinden imzalama)
    - vault read -field=certificate pki/issue/rauc-signer
        > /tmp/signing.cert.pem
    - # Bundle oluştur
    - rauc bundle
        --cert /tmp/signing.cert.pem
        --key  ${RAUC_SIGN_KEY}
        . firmware-${BUNDLE_VERSION}.raucb
    - rauc info --keyring ca.cert.pem firmware-${BUNDLE_VERSION}.raucb
  artifacts:
    paths: ["*.raucb"]

## ─── QEMU smoke test ───────────────────────────────────────
rauc:test:
  stage: test
  needs: [rauc:bundle]
  script:
    - # QEMU'da A/B boot + rauc install test
    - ./ci/qemu-rauc-test.sh firmware-${BUNDLE_VERSION}.raucb

## ─── hawkBit deploy (üretim — manuel onay) ─────────────────
hawkbit:upload:
  stage: deploy
  needs: [rauc:test]
  environment: production
  when: manual
  script:
    - # Distribution Set oluştur
    - |
      DS_ID=$(curl -sf -X POST \
        -H "Content-Type: application/json" \
        -u "admin:${HAWKBIT_PASSWORD}" \
        "${HAWKBIT_HOST}/rest/v1/distributionsets" \
        -d "{\"name\":\"firmware-${BUNDLE_VERSION}\",
             \"version\":\"${BUNDLE_VERSION}\",
             \"requiredMigrationStep\":false,
             \"type\":\"os\"}" | jq -r '.id')
      echo "Distribution Set ID: ${DS_ID}"
    - # Artifact yükle
    - |
      curl -sf -X POST \
        -H "Content-Type: application/octet-stream" \
        -u "admin:${HAWKBIT_PASSWORD}" \
        "${HAWKBIT_HOST}/rest/v1/softwaremodules/${DS_ID}/artifacts" \
        --data-binary @firmware-${BUNDLE_VERSION}.raucb
    - echo "hawkBit DS ${DS_ID} hazir — Rollout kampanyasi olusturabilirsiniz"

NOT

hawkBit rollout kampanyasında kademeli dağıtım için önce test grubu cihazlarını (örn. %5) hedefleyin. Başarı oranı eşiğini (%95 gibi) belirleyin; bu oran altında kalan otomatik duraklar. Saha verisi tatmin ediciyse geri kalan filoya %100 geçin. Bu yaklaşım bir güncelleme hatasının tüm filo yerine az sayıda cihazı etkilemesini sağlar.

Uçtan uca OTA pipeline özeti

  • meta-rauc + bblayers.conf — Yocto entegrasyonu tek layer ekleme ve inherit rauc-bundle ile tamamlanır
  • WIC kickstart A/B layout: rootfs-a (512M) + rootfs-b (512M) + userdata (2G)
  • GitLab CI: build → bundle (HSM imza) → QEMU test → hawkBit upload (manuel onay)
  • rauc mark-good: health check servisi onayından sonra — salt boot değil, uygulama sağlığı
  • bootcount > bootlimit → U-Boot otomatik rollback — brick'ten koruyan son savunma hattı
  • hawkBit kademeli rollout: %5 → %20 → %100 — hata erken tespit, filo koruması
