00 Device Tree neden gerekli

ARM'ın board-specific kod patlaması ve Linus'un "bu saçmalık bitiyor" kararıyla başlayan mimari dönüşüm.

2011 yılına kadar Linux kernel'ı ARM platformlarını arch/arm/mach-*/ dizinleri altında board bazında hardcoded C kodu ile destekliyordu. Her yeni board için ayrı bir board_init(), ayrı kaynak konfigürasyonları ve ayrı platform data yapıları yazılıyordu. Bu yaklaşım on yılda yüzlerce farklı board dosyasının kernel ağacına girmesiyle sürdürülemez hale geldi.

Kırılma noktası

TARİH

Mart 2011'de Linus Torvalds, ARM maintainer'larına yazdığı e-postada: "This whole ARM thing is a fucking pain in the ass" diyerek mevcut durumu eleştirdi. Aynı hafta içinde ARM platformları için Device Tree zorunlu hale getirildi. Bu tarihten itibaren yeni ARM board'ların mainline kernel'a kabul edilmesi için DTS dosyası şart koşuldu.

Çözüm: donanımı kernel'dan ayır

Device Tree fikri aslında yeni değildi — PowerPC ve SPARC mimarileri yıllardır Open Firmware (IEEE 1275) üzerinden benzer bir mekanizma kullanıyordu. ARM için de bu yaklaşım benimsendi: donanım tanımı kernel binary'sinden çıkarılır ve ayrı bir .dtb dosyasına taşınır. Aynı kernel binary'si farklı board'larda çalışabilir; yalnızca DTB değişir.

  .dts  Device Tree Source
   │    (metin, insan tarafından yazılır)
   │
   ▼   dtc (Device Tree Compiler)
  .dtb  Device Tree Blob
   │    (binary, kernel için)
   │
   ▼   U-Boot tarafından kernel'a aktarılır
  Kernel  of_* API ile DT parse eder
   │    Driver'lar compatible string'e göre probe olur
   ▼
  Çalışan sistem

01 DTS sözdizimi temelleri

Node yapısı, property türleri ve kök node ile temel bir DTS dosyasının anatomisi.

DTS (Device Tree Source) dosyası, ağaç (tree) yapısında düzenlenen node'lardan oluşur. Her node bir donanım bileşenini temsil eder ve içinde property'ler barındırır. Sözdizimi C-benzeri küme parantezleri kullanır.

Temel node yapısı

DTS syntax●
/* node-adı @ birim-adresi { property'ler; alt-node'lar; } */

node-name@unit-address {
    property-adi = "string değeri";
    sayisal-deger = <0x1000>;
    bool-property;              /* değersiz = true */
    byte-dizi = [DE AD BE EF];
};

Property türleri

TürSözdizimiÖrnek
String"değer"compatible = "ti,am335x-uart"
u32 (32-bit)<0x1234>clock-frequency = <48000000>
u64 (64-bit)/bits/ 64 <0x...>reg = /bits/ 64 <0x100000000>
Byte array[HH HH ...]local-mac-address = [00 11 22 33 44 55]
BooleanDeğersiz propertyinterrupt-controller;
Phandle&labelclocks = <&clk0>
String listesi"s1", "s2"clock-names = "core", "ahb"

Kök node ve address-cells

minimal.dts●
/dts-v1/;                    /* zorunlu başlık */

/ {                             /* kök node */
    #address-cells = <1>;       /* reg'in address kısmı 1x u32 */
    #size-cells = <1>;          /* reg'in size kısmı 1x u32 */

    model = "My Custom Board";
    compatible = "myvendor,myboard", "myvendor,mychip";

    /* UART node */
    uart0: serial@44E09000 {
        compatible = "ti,omap3-uart";
        reg = <0x44E09000 0x1000>;  /* base addr + size */
        status = "okay";
    };

    /* GPIO LED */
    leds {
        compatible = "gpio-leds";
        heartbeat {
            label = "board:green:heartbeat";
            gpios = <&gpio0 21 GPIO_ACTIVE_LOW>;
            linux,default-trigger = "heartbeat";
        };
    };
};

dtc ile derleme

bash●
# DTS → DTB
dtc -I dts -O dtb -o my-board.dtb my-board.dts

# DTB → DTS (geri dönüştür)
dtc -I dtb -O dts -o decoded.dts my-board.dtb

# Overlay için: -@ flag'i label'ları extern symbol olarak tutar
dtc -@ -I dts -O dtb -o my-overlay.dtbo my-overlay.dts

# Kernel'dan dtc'yi kullan
make ARCH=arm dtbs   # tüm board DTB'lerini derle
make ARCH=arm am335x-boneblack.dtb
NOT

#address-cells ve #size-cells değerleri alt node'ların reg property'sini nasıl yorumlayacağını belirler. Parent node'da #address-cells = <2> varsa alt node'un reg'i iki u32'lik adres + iki u32'lik boyut şeklinde okunur (64-bit adreslemede yaygın).

Bu bölümde

  • node-name@unit-address yapısı ve property türleri
  • #address-cells ve #size-cells'in reg yorumuna etkisi
  • /dts-v1/ başlığı ve kök node zorunluluğu
  • dtc -I dts -O dtb ile derleme; -@ ile overlay
  • make ARCH=arm dtbs ile kernel bazlı derleme

02 Temel property'ler ve binding'ler

compatible en kritik property'dir — driver eşleştirmesini ve donanım tanımı standartlarını belirler.

Linux kernel her bir DT node'unu compatible string'ini okuyarak hangi driver'ın probe edeceğine karar verir. Driver'ların beklediği property seti ise binding belgelerinde tanımlanır.

compatible property

DTS snippet●
/* Tekli compatible: tek eşleşme */
compatible = "arm,cortex-a9-pmu";

/* Çoklu: önce özgül, sonra genel — fallback için */
compatible = "ti,am335x-i2c", "ti,omap4-i2c";

/* Kural: "üretici,çip-revizyon" */
/* üretici: IANA vendor prefix veya kısa form */
compatible = "raspberrypi,rpi-backlight";
compatible = "st,stm32-i2c";
compatible = "nxp,pca9685-pwm";

status property

DTS snippet●
/* Periferik aktif: driver probe edilir */
status = "okay";

/* Periferik pasif: driver probe edilmez */
status = "disabled";

/* Reserved: başka yazılım kullanıyor (hypervisor vb.) */
status = "reserved";

/* Arızalı donanım */
status = "fail";

Standart property'ler

compatible"üretici,model" formatında string listesi. Driver eşleştirmenin temeli. Zorunlu sayılabilir.
regDonanımın bellek haritasındaki base address ve boyutu. Parent'ın #address-cells/#size-cells ile yorumlanır.
status"okay" | "disabled" | "reserved" | "fail". Varsayılan "okay", SoC DTSI'larında genellikle "disabled".
interruptsinterrupt-parent controller'a referansla interrupt numarası ve trigger tipi.
clocksclock provider phandle'ı ve clock ID'si. Sürücü bu clock'u enable/disable eder.
clock-namesclocks dizisindeki her clock'un sembolik adı. Driver, of_clk_get_by_name() ile isme göre alır.
pinctrl-0Varsayılan pin durumu için pinctrl konfigürasyon phandle'ı.
pinctrl-namesPinctrl state adları: "default", "sleep", "init" vb.

Binding belgeleri

bash●
# Binding'ler kernel kaynak kodunda
Documentation/devicetree/bindings/

# I2C binding'leri
Documentation/devicetree/bindings/i2c/

# SPI
Documentation/devicetree/bindings/spi/

# Belirli bir chip binding'i ara
find Documentation/devicetree/bindings/ -name "*tmp102*"
# → Documentation/devicetree/bindings/hwmon/ti,tmp102.yaml

# YAML binding doğrulama
make dt_binding_check DT_SCHEMA_FILES=bindings/hwmon/ti,tmp102.yaml
NOT

Modern kernel (v5.x+) binding'leri eski metin tabanlı .txt formatından YAML şemasına geçmektedir. make dtbs_check ile mevcut DTS dosyalarınızı YAML binding'lere karşı doğrulayabilirsiniz.

Bu bölümde

  • compatible string'i: "üretici,model" kuralı ve çoklu değer fallback
  • status: "okay" / "disabled" ve SoC DTSI'larında varsayılan disabled
  • reg, interrupts, clocks, pinctrl-0 standart property'leri
  • Documentation/devicetree/bindings/ ve YAML doğrulama

03 Phandle ve referanslar

Node'lar arası bağlantı için phandle mekanizması ve &label sözdizimi.

Device Tree'de birçok donanım bileşeni birbirine bağlıdır: bir sensor bir GPIO controller'a, bir periferik bir clock provider'a bağlanabilir. Bu bağlantılar phandle (pointer handle) mekanizması ile ifade edilir.

Label ve phandle

DTS snippet●
/* GPIO controller — label tanımı */
gpio0: gpio@44E07000 {
    compatible = "ti,omap4-gpio";
    reg = <0x44E07000 0x1000>;
    gpio-controller;
    #gpio-cells = <2>;           /* [gpio-num, flags] */
};

/* Clock provider */
dpll_core_ck: dpll_core_ck {
    compatible = "ti,am3-dpll-core-clock";
    #clock-cells = <0>;
};

/* LED: gpio0 phandle'ına referans */
leds {
    compatible = "gpio-leds";
    heartbeat {
        gpios = <&gpio0 21 GPIO_ACTIVE_LOW>;
        /* ^^^^^ label referansı: &gpio0 → phandle */
    };
};

Interrupt ve clock referansları

DTS snippet●
/* Interrupt controller tanımı */
intc: interrupt-controller@48200000 {
    compatible = "ti,am33xx-intc";
    interrupt-controller;
    #interrupt-cells = <1>;
    reg = <0x48200000 0x1000>;
};

/* UART: interrupt-parent ile intc'ye referans */
uart0: serial@44E09000 {
    compatible = "ti,omap3-uart";
    reg = <0x44E09000 0x1000>;
    interrupt-parent = <&intc>;
    interrupts = <72>;
    clocks = <&dpll_core_ck>;
    clock-names = "fck";
};

Güç regülatörü referansı

DTS snippet●
/* Regülatör tanımı */
reg_3v3: regulator-3v3 {
    compatible = "regulator-fixed";
    regulator-name = "3V3";
    regulator-min-microvolt = <3300000>;
    regulator-max-microvolt = <3300000>;
    regulator-always-on;
};

/* SD kart: besleme olarak reg_3v3'e referans */
mmc0: mmc@48060000 {
    compatible = "ti,omap4-hsmmc";
    reg = <0x48060000 0x1000>;
    vmmc-supply = <&reg_3v3>;
    bus-width = <4>;
    status = "okay";
};

DTSI ile ortak tanımlar

DTS snippet●
/* board.dts — SoC DTSI'yı dahil et */
/dts-v1/;
#include "am33xx.dtsi"      /* SoC ortak node'ları */
#include "am33xx-clocks.dtsi"

/ {
    model = "My AM335x Board";
    compatible = "myvendor,myboard", "ti,am33xx";
};

/* DTSI'da disabled olan uart0'ı aktif et */
&uart0 {
    pinctrl-names = "default";
    pinctrl-0 = <&uart0_pins>;
    status = "okay";
};

Bu bölümde

  • label: node_name ile phandle referansı oluşturma
  • GPIO referansı: &gpio0 + pin numarası + GPIO_ACTIVE_LOW
  • interrupt-parent ve clocks için phandle kullanımı
  • vmmc-supply gibi regülatör referansları
  • DTSI #include ile ortak SoC tanımları paylaşma

04 SoC ve board ayrımı (.dtsi)

Tek bir SoC'dan onlarca farklı board türetmek için DTSI katmanlı mimarisi.

Bir SoC ailesi (örneğin TI AM335x) onlarca farklı board'da kullanılır. Tüm bu board'ların SoC düzeyindeki tanımları ortaktır. DTSI (Device Tree Source Include) dosyaları bu ortak kısmı merkezi bir yerde tutar; her board DTS dosyası kendi özelliklerini ekler veya override eder.

Katmanlı yapı örneği: TI AM335x

  am33xx.dtsi
      │  SoC mimarisinin tüm periferalleri — hepsi disabled
      │  uart0...uart6, i2c0...i2c2, spi0...spi1, mmc0...mmc2 vb.
      │
      ├─ am335x-boneblack.dts
      │      am33xx.dtsi'yı include et
      │      Kullanılan peripheral'ları "okay" yap
      │      Board-specific: eMMC, LCD, Ethernet vb.
      │
      ├─ am335x-evm.dts
      │      Farklı bir board; aynı SoC DTSI
      │
      └─ am335x-boneblue.dts
             Robotik odaklı board; IMU, PWM eklendi

SoC DTSI yapısı (am33xx.dtsi özeti)

arch/arm/boot/dts/ti/omap/am33xx.dtsi (özet)●
/dts-v1/;

/ {
    #address-cells = <1>;
    #size-cells = <1>;

    cpus {
        cpu@0 { compatible = "arm,cortex-a8"; };
    };

    am33xx_pinmux: pinmux@44E10800 {
        compatible = "pinctrl-single";
        reg = <0x44E10800 0x238>;
        #pinctrl-cells = <1>;
    };

    uart0: serial@44E09000 {
        compatible = "ti,am3352-uart", "ti,omap3-uart";
        reg = <0x44E09000 0x2000>;
        status = "disabled";   /* board DTS aktif eder */
    };

    i2c0: i2c@44E0B000 {
        compatible = "ti,omap4-i2c";
        reg = <0x44E0B000 0x1000>;
        #address-cells = <1>;
        #size-cells = <0>;
        status = "disabled";
    };
};

Board DTS: BeagleBone Black (özet)

arch/arm/boot/dts/ti/omap/am335x-boneblack.dts (özet)●
/dts-v1/;
#include "am33xx.dtsi"
#include "am335x-bone-common.dtsi"

/ {
    model = "TI AM335x BeagleBone Black";
    compatible = "ti,am335x-bone-black", "ti,am335x-bone", "ti,am33xx";
};

/* DTSI'daki uart0'ı etkinleştir */
&uart0 {
    pinctrl-names = "default";
    pinctrl-0 = <&uart0_pins>;
    status = "okay";
};

/* eMMC: boneblack'e özgü */
&mmc2 {
    vmmc-supply = <&vmmcsd_fixed>;
    pinctrl-names = "default";
    pinctrl-0 = <&emmc_pins>;
    bus-width = <8>;
    status = "okay";
};

Raspberry Pi yapısı

bash●
# ARM64 DTS dizini
arch/arm64/boot/dts/broadcom/

# Katmanlı yapı:
bcm2711.dtsi            → SoC periferalleri (BCM2711 = RPi4 SoC)
bcm2711-rpi.dtsi        → Raspberry Pi ortak tanımlar
bcm2711-rpi-4-b.dts     → RPi 4 Model B spesifik

Bu bölümde

  • DTSI: SoC düzeyi ortak tanımlar, tüm peripheral'lar disabled
  • Board DTS: DTSI'yı include et, gerekli node'ları okay yap
  • am33xx.dtsi → am335x-boneblack.dts katmanlı örneği
  • Raspberry Pi: bcm2711.dtsi → bcm2711-rpi-4-b.dts
  • &node_label ile DTSI node'larını override etme

05 GPIO, I2C, SPI, UART tanımlama

Gömülü sistemlerin en sık kullandığı arayüzlerin DTS node tanımları ve binding'leri.

GPIO LED node

DTS snippet●
leds {
    compatible = "gpio-leds";

    /* Heartbeat LED — çalışma göstergesi */
    led-heartbeat {
        label = "board:green:heartbeat";
        gpios = <&gpio1 17 GPIO_ACTIVE_LOW>;
        linux,default-trigger = "heartbeat";
    };

    /* Kullanıcı LED */
    led-user {
        label = "board:yellow:user";
        gpios = <&gpio1 21 GPIO_ACTIVE_HIGH>;
        linux,default-trigger = "none";
        default-state = "off";
    };
};

/* GPIO buton */
gpio-keys {
    compatible = "gpio-keys";
    button-0 {
        label = "User Button";
        gpios = <&gpio0 27 GPIO_ACTIVE_LOW>;
        linux,code = <KEY_ENTER>;
        debounce-interval = <20>;
    };
};

I2C cihaz ekleme

DTS snippet●
/* I2C bus üzerindeki cihazlar bus node'u altına eklenir */
&i2c1 {
    pinctrl-names = "default";
    pinctrl-0 = <&i2c1_pins>;
    clock-frequency = <400000>;   /* 400 kHz Fast Mode */
    status = "okay";

    /* TI TMP102 sıcaklık sensörü — adres 0x48 */
    temp_sensor: sensor@48 {
        compatible = "ti,tmp102";
        reg = <0x48>;
        interrupt-parent = <&gpio0>;
        interrupts = <26 IRQ_TYPE_LEVEL_LOW>;
    };

    /* EEPROM — adres 0x50 */
    eeprom@50 {
        compatible = "atmel,24c32";
        reg = <0x50>;
        pagesize = <32>;
    };
};

SPI cihaz ekleme

DTS snippet●
&spi0 {
    pinctrl-names = "default";
    pinctrl-0 = <&spi0_pins>;
    status = "okay";

    /* ILI9341 SPI display — CS0 */
    display@0 {
        compatible = "ilitek,ili9341";
        reg = <0>;                        /* CS0 */
        spi-max-frequency = <32000000>;
        dc-gpios = <&gpio0 7 GPIO_ACTIVE_HIGH>;
        reset-gpios = <&gpio0 8 GPIO_ACTIVE_LOW>;
    };

    /* SPI NOR Flash — CS1 */
    flash@1 {
        compatible = "jedec,spi-nor";
        reg = <1>;
        spi-max-frequency = <50000000>;
        spi-tx-bus-width = <1>;
        spi-rx-bus-width = <4>;         /* QSPI okuma */
    };
};

UART tanımlama

DTS snippet●
&uart2 {
    pinctrl-names = "default";
    pinctrl-0 = <&uart2_pins>;
    status = "okay";
};

/* RS-485 ile UART */
&uart4 {
    pinctrl-names = "default";
    pinctrl-0 = <&uart4_pins>;
    linux,rs485-enabled-at-boot-time;
    rs485-rts-delay = <0 200>;    /* rts-before-send, rts-after-send (us) */
    rs485-rts-active-low;
    status = "okay";
};

Bu bölümde

  • gpio-leds binding: gpios property + GPIO_ACTIVE_LOW/HIGH
  • I2C: clock-frequency, cihaz reg (adres), interrupt bağlantısı
  • SPI: CS numarası reg ile, spi-max-frequency zorunlu
  • UART: pinctrl bağlantısı, RS-485 desteği
  • I2C/SPI cihazları bus node'unun alt node'ları olarak tanımlanır

06 Interrupt tanımlama

GIC, GPIO interrupt ve interrupts-extended ile arm SMP sistemlerinde interrupt yönlendirme.

Device Tree'de interrupt sistemi hiyerarşiktir: interrupt controller node'ları interrupt-controller property'si ile işaretlenir; cihazlar ise interrupt-parent veya interrupts-extended ile hangi controller'a bağlı olduklarını belirtir.

Interrupt controller tanımı

DTS snippet — GIC●
/* ARM GIC — Generic Interrupt Controller (ARM SMP standart) */
gic: interrupt-controller@a0021000 {
    compatible = "arm,cortex-a15-gic";
    interrupt-controller;
    #interrupt-cells = <3>;
    /* [type, num, flags]
       type: GIC_SPI (0) veya GIC_PPI (1)
       num: interrupt numarası
       flags: IRQ_TYPE_* */
    reg = <0xa0021000 0x1000>
          <0xa0022000 0x2000>;
};

/* Cihaz: GIC SPI interrupt */
uart0: serial@12000000 {
    compatible = "arm,pl011";
    reg = <0x12000000 0x1000>;
    interrupt-parent = <&gic>;
    interrupts = <GIC_SPI 96 IRQ_TYPE_LEVEL_HIGH>;
};

GPIO interrupt

DTS snippet — GPIO interrupt●
/* GPIO controller: hem GPIO hem interrupt controller */
gpio0: gpio@44E07000 {
    compatible = "ti,omap4-gpio";
    reg = <0x44E07000 0x1000>;
    interrupt-controller;
    #interrupt-cells = <2>;   /* [gpio-num, IRQ_TYPE_*] */
    gpio-controller;
    #gpio-cells = <2>;
};

/* Düşen kenar tetiklemeli GPIO interrupt */
accel@1D {
    compatible = "adi,adxl345";
    reg = <0x1D>;
    interrupts-extended = <&gpio0 14 IRQ_TYPE_EDGE_FALLING>;
};

/* Alternatif: interrupt-parent + interrupts ayrı */
sensor@48 {
    compatible = "ti,tmp102";
    reg = <0x48>;
    interrupt-parent = <&gpio0>;
    interrupts = <26 IRQ_TYPE_LEVEL_LOW>;
};

IRQ tip sabitleri

SabitDeğerAçıklama
IRQ_TYPE_NONE0Tip belirtilmemiş; driver kendi ayarlar
IRQ_TYPE_EDGE_RISING1Yükselen kenar tetiklemeli
IRQ_TYPE_EDGE_FALLING2Düşen kenar tetiklemeli
IRQ_TYPE_EDGE_BOTH3Her iki kenar da tetikler
IRQ_TYPE_LEVEL_HIGH4Yüksek seviye tetiklemeli
IRQ_TYPE_LEVEL_LOW8Düşük seviye tetiklemeli

interrupts-extended vs interrupt-parent

DTS snippet●
/* interrupt-parent: tüm interrupts aynı controller'a gider */
device@0 {
    interrupt-parent = <&gic>;
    interrupts = <GIC_SPI 10 IRQ_TYPE_LEVEL_HIGH>,
                 <GIC_SPI 11 IRQ_TYPE_LEVEL_HIGH>;
};

/* interrupts-extended: her interrupt farklı controller'a gidebilir */
device@0 {
    interrupts-extended = <&gic  GIC_SPI 10 IRQ_TYPE_LEVEL_HIGH>,
                           <&gpio0 14 IRQ_TYPE_EDGE_FALLING>;
};
NOT

interrupts-extended kullanılırsa interrupt-parent ve interrupts property'leri görmezden gelinir. İki farklı controller'a bağlanan cihazlar için tercih edilmeli.

Bu bölümde

  • interrupt-controller ve #interrupt-cells property'leri
  • GIC: 3-hücreli format — GIC_SPI/GIC_PPI, numara, IRQ_TYPE_*
  • GPIO interrupt: 2-hücreli format — pin numarası, IRQ_TYPE_*
  • interrupts-extended: farklı controller'lara birden fazla interrupt
  • IRQ_TYPE_EDGE_FALLING, IRQ_TYPE_LEVEL_HIGH vb. sabitler

07 Pinctrl ve pin multiplexing

SoC'larda çok amaçlı pinlerin hangi fonksiyona atanacağını DTS ile yönet.

Modern SoC'larda tek bir fiziksel pin birden fazla fonksiyon üstlenebilir: UART RX, GPIO girişi veya I2C SDA olarak kullanılabilir. Pinctrl subsystem hangi pinin hangi fonksiyona atandığını (multiplexing) ve elektriksel özelliklerini (pull-up, pull-down, drive strength) yönetir.

AM335x pinmux örneği

DTS snippet — AM335x●
&am33xx_pinmux {
    /* UART0 pinleri: console UART */
    uart0_pins: uart0-pins {
        pinctrl-single,pins = <
            /* offset          mode */
            AM33XX_PADCONF(AM335X_PIN_UART0_RXD,
                           PIN_INPUT_PULLUP, MUX_MODE0)
            AM33XX_PADCONF(AM335X_PIN_UART0_TXD,
                           PIN_OUTPUT_PULLDOWN, MUX_MODE0)
        >;
    };

    /* I2C0 pinleri */
    i2c0_pins: i2c0-pins {
        pinctrl-single,pins = <
            AM33XX_PADCONF(AM335X_PIN_I2C0_SDA,
                           PIN_INPUT_PULLUP, MUX_MODE0)
            AM33XX_PADCONF(AM335X_PIN_I2C0_SCL,
                           PIN_INPUT_PULLUP, MUX_MODE0)
        >;
    };

    /* LED GPIO: P8_13 → GPIO0_21 */
    user_leds_pins: user-leds-pins {
        pinctrl-single,pins = <
            AM33XX_PADCONF(AM335X_PIN_GPMC_A5,
                           PIN_OUTPUT, MUX_MODE7)  /* GPIO mode */
        >;
    };
};

BCM2835 (Raspberry Pi) pinctrl

DTS snippet — Raspberry Pi●
/* BCM2835 pinctrl: GPIO14/15 → UART0 alt function 4 */
uart0_gpio14: uart0_gpio14 {
    brcm,pins = <14 15>;
    brcm,function = <4>;           /* Alt0 = UART TX/RX */
};

uart0_gpio36: uart0_gpio36 {
    brcm,pins = <36 37 30 31>;
    brcm,function = <7>;           /* Alt3 = CTS/RTS dahil */
};

&uart0 {
    pinctrl-names = "default";
    pinctrl-0 = <&uart0_gpio14>;
    status = "okay";
};

Pinctrl state'leri

DTS snippet — default + sleep state●
/* SPI: normal çalışma ve düşük güç durumu */
spi0_pins_default: spi0-pins-default {
    pinctrl-single,pins = <
        AM33XX_PADCONF(AM335X_PIN_SPI0_SCLK,
                       PIN_INPUT_PULLUP, MUX_MODE0)
        AM33XX_PADCONF(AM335X_PIN_SPI0_D0,
                       PIN_INPUT_PULLUP, MUX_MODE0)
        AM33XX_PADCONF(AM335X_PIN_SPI0_D1,
                       PIN_OUTPUT_PULLDOWN, MUX_MODE0)
        AM33XX_PADCONF(AM335X_PIN_SPI0_CS0,
                       PIN_OUTPUT_PULLUP, MUX_MODE0)
    >;
};

spi0_pins_sleep: spi0-pins-sleep {
    pinctrl-single,pins = <
        AM33XX_PADCONF(AM335X_PIN_SPI0_SCLK,
                       PIN_INPUT_PULLDOWN, MUX_MODE7)  /* GPIO */
        AM33XX_PADCONF(AM335X_PIN_SPI0_D0,
                       PIN_INPUT_PULLDOWN, MUX_MODE7)
        AM33XX_PADCONF(AM335X_PIN_SPI0_D1,
                       PIN_INPUT_PULLDOWN, MUX_MODE7)
        AM33XX_PADCONF(AM335X_PIN_SPI0_CS0,
                       PIN_INPUT_PULLDOWN, MUX_MODE7)
    >;
};

&spi0 {
    pinctrl-names = "default", "sleep";
    pinctrl-0 = <&spi0_pins_default>;
    pinctrl-1 = <&spi0_pins_sleep>;
    status = "okay";
};
NOT

sleep state'e geçiş otomatiktir: runtime PM driver'ı suspend çağrıldığında kernel pinctrl subsystem'e haber verir ve pinler sleep konfigürasyonuna alınır. Bu, düşük güç modunda kaçak akımı ve paraziti azaltır.

Bu bölümde

  • Pinctrl subsystem: MUX_MODE ile fonksiyon seçimi, PIN_INPUT_PULLUP vb.
  • AM33XX_PADCONF makrosu ile AM335x pinmux tanımı
  • BCM2835: brcm,pins + brcm,function ile Raspberry Pi pinmux
  • pinctrl-names + pinctrl-0/1 ile default ve sleep state bağlantısı
  • Sleep state ile düşük güç modu pini yönetimi

08 Device Tree Overlay

Çalışan sisteme dinamik DT değişikliği — HAT eklentisi, opsiyonel donanım ve runtime yapılandırma.

Device Tree Overlay, ana DTB'yi yeniden derlemeye gerek kalmadan çalışan sisteme yeni node'lar eklemek veya varolan property'leri değiştirmek için kullanılır. Raspberry Pi HAT'lar, eklenti kartlar ve fabrika konfigürasyon farkları için idealdir.

Overlay sözdizimi

my-sensor-overlay.dts●
/dts-v1/;
/plugin/;                          /* overlay direktifi — zorunlu */

#include <dt-bindings/gpio/gpio.h>
#include <dt-bindings/interrupt-controller/irq.h>

/ {
    compatible = "brcm,bcm2711";    /* hedef SoC */

    fragment@0 {
        target = <&i2c1>;           /* ana DT'deki i2c1 node'unu hedefle */
        __overlay__ {
            status = "okay";
            clock-frequency = <100000>;

            bme280@76 {
                compatible = "bosch,bme280";
                reg = <0x76>;
            };
        };
    };

    fragment@1 {
        target-path = "/";              /* kök node'a ekle */
        __overlay__ {
            gpio-leds {
                compatible = "gpio-leds";
                hat-led {
                    label = "hat:green:status";
                    gpios = <&gpio 25 GPIO_ACTIVE_HIGH>;
                    linux,default-trigger = "none";
                };
            };
        };
    };
};

Overlay parametreleri

parametreli-overlay.dts●
/dts-v1/;
/plugin/;

/ {
    fragment@0 {
        target = <&i2c1>;
        __overlay__ {
            sensor@48 {
                compatible = "ti,tmp102";
                reg = <0x48>;
                status = "okay";
            };
        };
    };

    /* Parametre tanımları: dtoverlay=my-sensor,addr=0x4A */
    __overrides__ {
        addr = <&sensor_node>, "reg:0";
    };
};

Overlay derleme

bash●
# -@ flag'i: label'ları extern symbol olarak sakla (overlay için şart)
dtc -@ -I dts -O dtb \
    -o my-sensor-overlay.dtbo \
    my-sensor-overlay.dts

# Kernel'dan derleme (Makefile'a DTC_FLAGS += -@ ekle)
make ARCH=arm64 my-sensor-overlay.dtbo

Overlay yükleme yöntemleri

bash (target — Linux üzerinde)●
# configfs ile dinamik yükleme (kernel CONFIG_OF_OVERLAY=y gerektirir)
mkdir /sys/kernel/config/device-tree/overlays/my-sensor

# dtbo dosyasını configfs'e kopyala
cp my-sensor-overlay.dtbo \
   /sys/kernel/config/device-tree/overlays/my-sensor/dtbo

# Alternatif: dtbo içeriğini doğrudan yaz
cat my-sensor-overlay.dtbo \
    > /sys/kernel/config/device-tree/overlays/my-sensor/dtbo

# Kaldır
rmdir /sys/kernel/config/device-tree/overlays/my-sensor
/boot/config.txt (Raspberry Pi)●
# Boot sırasında overlay yükle
dtoverlay=my-sensor-overlay

# Parametreli
dtoverlay=my-sensor-overlay,addr=0x4A

# Birden fazla overlay
dtoverlay=i2c-sensor,addr=0x76
dtoverlay=spi-display
DİKKAT

Overlay yalnızca Device Tree'yi değiştirir — driver probe etmez. Overlay eklendikten sonra driver'ın yeni node'u algılaması için modprobe veya udev aracılığıyla yeniden probe tetiklenmesi gerekebilir. echo 1 > /sys/bus/i2c/devices/i2c-1/rescan gibi komutlar bazı durumlarda çalışır.

Bu bölümde

  • /plugin/ direktifi ile overlay DTS yapısı
  • fragment@N + target ile ana DT node'unu hedefleme
  • __overrides__ ile parametreli overlay
  • dtc -@ ile overlay derleme; .dtbo uzantısı
  • configfs ve Raspberry Pi config.txt ile yükleme yöntemleri

09 Debug ve doğrulama

Çalışan sistemde DT'yi oku, parse hatalarını incele ve sık yapılan hataları tespit et.

Device Tree sorunları genellikle iki kategoride ortaya çıkar: derleme aşamasında yakalanabilen sözdizimi hataları ve çalışma zamanında driver probe başarısızlığına yol açan binding hataları. Her iki kategori için farklı araçlar kullanılır.

Çalışan DT'yi okuma

bash (target)●
# /proc/device-tree: sysfs üzerinden DT binary gezintisi
ls /proc/device-tree/
ls /proc/device-tree/soc/
cat /proc/device-tree/model            # board model adı
xxd /proc/device-tree/soc/uart@44e09000/reg  # reg property ham değeri

# /sys/firmware/devicetree/base/ — aynı içerik, farklı yol
ls /sys/firmware/devicetree/base/

# Çalışan DT'yi okunabilir DTS'e dönüştür
dtc -I dtb -O dts /proc/device-tree 2>/dev/null | less

# fdtdump ile binary DTB'yi analiz et
fdtdump my-board.dtb | less

DTB dosyasını decode etme

bash●
# Binary DTB → okunabilir DTS
dtc -I dtb -O dts -o decoded.dts my-board.dtb

# Belirli bir node'u bul
dtc -I dtb -O dts my-board.dtb 2>/dev/null | grep -A 10 "serial@44e09000"

# fdtget: DTB'den tek property değeri oku
fdtget my-board.dtb /soc/serial@44e09000 status
# → okay

fdtget my-board.dtb /soc/serial@44e09000 compatible
# → ti,am3352-uart ti,omap3-uart

dmesg ile DT parse hataları

bash (target)●
# DT ile ilgili tüm kernel mesajları
dmesg | grep "OF: "

# Örnek çıktılar:
OF: fdt: Machine model: TI AM335x BeagleBone Black
OF: /ocp/i2c@4802a000: could not get clocks
OF: graph: no port node found in /ocp/lcdc@4830e000

# Probe başarısız driver'ları listele
dmesg | grep "probe"

# Compatible eşleşmeyen node'lar
dmesg | grep "Driver.*not found"

# Kernel açılışta DT binding doğrulama (5.x+)
dmesg | grep "dt-bindings"

Kernel driver'dan DT okuma (C API)

driver.c (kernel space)●
/* platform_driver probe fonksiyonu */
static int my_probe(struct platform_device *pdev)
{
    struct device_node *np = pdev->dev.of_node;
    u32 val;
    int ret;

    /* u32 property oku */
    ret = of_property_read_u32(np, "clock-frequency", &val);
    if (ret) {
        dev_err(&pdev->dev, "clock-frequency eksik\n");
        return ret;
    }

    /* string property oku */
    const char *str;
    of_property_read_string(np, "label", &str);

    /* boolean property var mı? */
    if (of_property_read_bool(np, "big-endian"))
        flags |= MY_FLAG_BIG_ENDIAN;

    /* GPIO al */
    int gpio = of_get_named_gpio(np, "reset-gpios", 0);

    return 0;
}

static const struct of_device_id my_of_table[] = {
    { .compatible = "myvendor,mydevice" },
    { }
};
MODULE_DEVICE_TABLE(of, my_of_table);

Sık yapılan hatalar

HataBelirtiÇözüm
Yanlış compatible Driver probe edilmez, dmesg'de eşleşme yok Binding belgesi ve kernel driver'daki of_device_id tablosunu karşılaştır
status = "disabled" unutulmuş DTSI'dan gelen disabled hali geçerli kalır Board DTS'de &node { status = "okay"; }; override ekle
Eksik #address-cells dtc uyarısı; reg property hatalı yorumlanır I2C/SPI bus node'larında #address-cells = <1>; #size-cells = <0>; ekle
Yanlış interrupt-parent Interrupt gelemez; driver init'te takılır /proc/interrupts ve dmesg ile aktif interrupt'ları doğrula
Adres çakışması Birden fazla node aynı reg aralığında dtc -I dtb -O dts ile decode edip reg değerlerini karşılaştır

Kernel build zamanı doğrulama

bash (host)●
# Tüm DTB'leri derleyip binding uyumunu kontrol et
make ARCH=arm dtbs_check

# Tek binding dosyası için
make ARCH=arm dt_binding_check \
  DT_SCHEMA_FILES=Documentation/devicetree/bindings/serial/snps-dw-apb-uart.yaml

# DTS dosyasını dtc ile sözdizimi kontrolü
dtc -I dts -O dtb \
    -W no-unit_address_vs_reg \
    my-board.dts -o /dev/null
NOT

Driver probe başarısız olduğunda /sys/bus/platform/drivers/<driver>/ dizini boş gelir. cat /sys/bus/platform/devices/<device>/driver sembolik linkinin nereye işaret ettiği kontrol edilmelidir. Hiçbir yere işaret etmiyorsa compatible eşleşmesi veya probe fonksiyonu başarısız demektir.

Bu bölümde

  • /proc/device-tree ve /sys/firmware/devicetree/base/ ile sysfs gezintisi
  • dtc -I dtb -O dts ile çalışan DT decode; fdtget ile tek property sorgusu
  • dmesg | grep "OF: " ile parse ve probe hatalarını yakalama
  • of_property_read_u32, of_get_named_gpio — kernel C API
  • make dtbs_check ile build-time binding doğrulaması
