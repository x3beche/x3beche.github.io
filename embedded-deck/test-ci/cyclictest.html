00 RT Latency Nedir?

Gerçek zamanlı sistemlerde en kritik metrik, bir olayın gerçekleşmesinden uygulamanın tepki vermesine kadar geçen maksimum süredir.

Genel amaçlı Linux çekirdeği (mainline), throughput (iş hacmi) optimizasyonu için tasarlanmıştır. Scheduler, uzun dönemde adil CPU dağıtımı yapmaya çalışır; bu süreçte anlık gecikmeler (latency) ikincil plana atılır. Sonuç: bir RT thread teorik olarak yüzlerce milisaniye bekleyebilir.

PREEMPT_RT yaması, çekirdeğin büyük bölümünü preemptible (kesilebilir) hale getirir. Kritik bölgeler spinlock yerine mutex kullanır, interrupt handler'lar threaded hale gelir. Hedef: worst-case latency'yi öngörülebilir ve düşük tutmak.

Temel kavramlar

Latency
Bir timer tetiklendiğinden, ilgili RT thread'in uyanıp çalışmaya başlamasına kadar geçen süre. Mikrosaniye (µs) cinsinden ölçülür.
Jitter
Ard arda gelen latency ölçümleri arasındaki değişkenlik. Düşük ortalama latency'ye rağmen yüksek jitter, RT sistemler için kabul edilemez.
Worst-case latency
Tüm test süresi boyunca gözlemlenen maksimum gecikme. RT sistemlerde bu değer hiçbir zaman belirli bir eşiği geçmemeli.
Latency bütçesi
Uygulama katmanının kabul edebileceği maksimum gecikme. Motor kontrolü: ~50µs, ses işleme: ~1ms, sanayi PLC: ~250µs.
WCET
Worst-Case Execution Time — bir görevin tamamlanması için gereken maksimum CPU süresi. Latency bütçesinin bir parçası.
NEDEN WORST-CASE?

Ortalama latency 10µs olsa bile, günde bir kez 500µs'lik bir spike tüm sistemi bozabilir. Güvenlik-kritik sistemlerde (tıbbi cihaz, araç kontrolü) worst-case latency yasal bir gereksinim olabilir. cyclictest, saatlerce çalışarak bu nadiren gerçekleşen spike'ları yakalar.

# 1 samples exceeding threshold: max = 18us

02 cyclictest Parametreleri

cyclictest, POSIX timer kullanarak RT thread'lerin wakeup gecikmesini ölçer. Doğru parametreler çok önemlidir.

Temel sözdizimi ve parametreler

-t N / --threads=N
Oluşturulacak thread sayısı. -t 4 = 4 thread, varsayılan 1. Her thread ayrı bir timer kullanır.
-p P / --priority=P
RT thread önceliği (1-99). -p 98 önerilir; 99 watchdog için saklanmalı.
-i I / --interval=I
Timer periyodu mikrosaniye cinsinden. -i 1000 = 1ms periyot. Daha kısa periyot → daha fazla ölçüm.
-d D / --distance=D
Multi-thread modda thread'ler arası başlangıç ofseti (µs). -d 0 tüm thread'leri aynı anda tetikler.
-h BUCKETS / --histogram
Histogram modu. -h 400 = 400µs'ye kadar bucket, sonrasını overflow sayar.
-l N / --loops=N
Toplam ölçüm sayısı. -l 1000000. -D DURATION ile süre sınırı da konabilir.
-D DURATION
Çalışma süresi: 30m, 2h, overnight formatında. Uzun testler için -l yerine tercih edilir.
--json
Makine tarafından okunabilir JSON çıktısı. CI entegrasyonunda parse için kullanılır.
-a CPU / --affinity
Thread'leri belirli CPU'lara bağla. -a 2,3 = thread'ler sadece CPU 2 ve 3'te çalışır.
--mlockall
Tüm belleği RAM'e kilitle. Page fault'ların latency spike'ına neden olmasını önler.

Tipik kullanım örnekleri

bash — cyclictest komutları●
# Hızlı smoke testi — 30 saniye, tek thread
cyclictest -t1 -p98 -i1000 -D 30s --mlockall

# Üretim testi — tüm CPU'lar, 2 saat, histogram çıktısı
cyclictest -t$(nproc) -p98 -i1000 -D 2h -h 400 --mlockall --json \
  > latency.json

# CPU 2-3'ü RT'ye ayır, diğer CPU'larda yük oluştur
cyclictest -t2 -p98 -i1000 -a 2,3 -D 30m -h 400 --mlockall

# Tipik çıktı:
# T: 0 (  1234) P:98 I:1000 C:1800000 Min:    3 Act:    5 Avg:    6 Max:   42
# T: 1 (  1235) P:98 I:1000 C:1800000 Min:    3 Act:    6 Avg:    6 Max:   51
# Sütunlar: Thread ID, PID, Öncelik, İnterval, Count, Min, Aktif, Ortalama, Maks
ROOT GEREKSİNİMİ

cyclictest'in RT öncelik ayarlaması için root veya CAP_SYS_NICE yetkisi gerekir. Embedded sistemlerde genellikle root olarak çalıştırılır. Container içinde çalıştırıyorsanız: --cap-add=SYS_NICE --cap-add=SYS_ADMIN

JSON çıktısını parse etme (CI için)

python3 — JSON sonuç parse●
import json, sys

with open('latency.json') as f:
    data = json.load(f)

MAX_ALLOWED_US = 50   # latency bütçesi
failed = False

for thread in data['thread'].values():
    max_lat = thread['max']
    avg_lat = thread['avg']
    print(f"Thread {thread['id']}: min={thread['min']}µs avg={avg_lat}µs max={max_lat}µs")
    if max_lat > MAX_ALLOWED_US:
        print(f"  HATA: max latency {max_lat}µs > bütçe {MAX_ALLOWED_US}µs", file=sys.stderr)
        failed = True

sys.exit(1 if failed else 0)

04 /proc/sys/kernel Tunables

Linux kernel parametreleri, RT performansını doğrudan etkileyen scheduler davranışını ayarlar.

Kritik RT tunables

bash — RT tuning parametreleri●
# ─── RT throttling — KAPATILMALI ────────────────────────────
# Varsayılan: RT thread'ler 1 saniyede en fazla 950ms çalışabilir
# Bu, RT thread'leri zorla kesilebilir kılar → spike oluşturur
echo -1 > /proc/sys/kernel/sched_rt_runtime_us   # limitsiz
echo 1000000 > /proc/sys/kernel/sched_rt_period_us

# ─── NUMA balancing — KAPATILMALI ───────────────────────────
# NUMA balancer bellek sayfalarını taşır → latency spike
echo 0 > /proc/sys/kernel/numa_balancing

# ─── Transparent HugePages — KAPATILMALI ────────────────────
# THP compaction background işlemi RT thread'leri engeller
echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag

# ─── Watchdog — KAPATILMALI (isteğe bağlı) ──────────────────
echo 0 > /proc/sys/kernel/watchdog
echo 0 > /proc/sys/kernel/nmi_watchdog

# ─── Sched features ─────────────────────────────────────────
echo NO_GENTLE_FAIR_SLEEPERS > /sys/kernel/debug/sched/features
echo NO_WAKEUP_PREEMPTION   > /sys/kernel/debug/sched/features

# ─── CPU frekansı — sabit tut ───────────────────────────────
for cpu in /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor; do
  echo performance > $cpu
done

Kalıcı ayarlar — sysctl.conf

/etc/sysctl.d/99-rt-tuning.conf●
# RT thread throttling kapat
kernel.sched_rt_runtime_us = -1
kernel.sched_rt_period_us  = 1000000

# NUMA dengeleme kapat
kernel.numa_balancing = 0

# vm.swappiness — RT sistemde swap istemiyoruz
vm.swappiness = 0

# Uygula: sysctl --system

Kernel komut satırı parametreleri

U-Boot / GRUB — kernel cmdline●
# /boot/cmdline.txt (Raspberry Pi) veya GRUB_CMDLINE_LINUX
console=ttyAMA0,115200 root=/dev/mmcblk0p2 rw
isolcpus=2,3          # CPU 2,3'ü scheduler'dan izole et
nohz_full=2,3         # tickless mode — timer interrupt azalt
rcu_nocbs=2,3         # RCU callback CPU 2,3'te çalışmasın
irqaffinity=0,1       # IRQ'lar CPU 0,1'de işlensin
nosoftlockup          # soft lockup detector kapat
skew_tick=1           # timer tick'leri dağıt
transparent_hugepage=never
UYARI: sched_rt_runtime_us = -1

RT throttling kapatılırsa, hatalı bir RT thread tüm sistemi dondurabilir. Bu ayar sadece test edilmiş, güvenilir RT uygulamalarınız varsa kullanılmalıdır. Geliştirme ortamında dikkatli olun — serial konsol üzerinden sistemi kurtaramayabilirsiniz.

05 IRQ Affinity ve CPU İzolasyonu

RT thread'lerin çalıştığı CPU'lara IRQ kesintilerinin yönlendirilmemesi, latency spike'larının en büyük kaynağını ortadan kaldırır.

IRQ listesi ve affinity görüntüleme

bash — IRQ affinity inceleme●
# Tüm IRQ'ları ve CPU affinity'sini göster
cat /proc/interrupts

# Belirli bir IRQ'nun affinity maskesi (hex bitmask)
cat /proc/irq/18/smp_affinity        # CPU bitmask
cat /proc/irq/18/smp_affinity_list   # daha okunaklı

# Örnek: 0x3 = binary 0011 = CPU 0 ve 1
# Tüm IRQ'ları CPU 0,1'e yönlendir
for irq in /proc/irq/*/smp_affinity; do
  echo 3 > $irq 2>/dev/null  # 0x3 = CPU 0,1
done

irqbalance durdurma

bash — irqbalance yönetimi●
# irqbalance manuel IRQ affinity ayarlarını bozar — durdur
systemctl stop irqbalance
systemctl disable irqbalance

# Alternatif: irqbalance'ı çalıştır ama RT CPU'lardan hariç tut
cat /etc/default/irqbalance
# IRQBALANCE_BANNED_CPUS="0c"  ← hex: CPU 2 ve CPU 3 bitmask

# IRQBALANCE_BANNED_CPUS hesaplama:
# CPU 0 = 0x01, CPU 1 = 0x02, CPU 2 = 0x04, CPU 3 = 0x08
# CPU 2 + CPU 3 = 0x04 + 0x08 = 0x0C

CPU izolasyonu — isolcpus

bash — isolcpus doğrulama●
# isolcpus aktif mi?
cat /sys/devices/system/cpu/isolated
# Çıktı: 2-3  (CPU 2 ve 3 izole)

# İzole CPU'da RT thread çalıştır
taskset -c 2 cyclictest -t1 -p98 -i1000 -D 30m -h 400

# numactl ile birlikte
numactl --cpunodebind=0 --membind=0 \
  taskset -c 2 cyclictest -t1 -p99 -i1000 -D 30m

# Kernel thread'lerini izole CPU'dan kaldır
tuna --cpus=2,3 --isolate
CPU TOPOLOJISI

Hyperthreading aktifse, bir fiziksel çekirdeğin iki sanal CPU'su (sibling) birbirini etkiler. RT thread'ler için tam izolasyon sağlamak adına sibling CPU'nun da izole edilmesi veya hyperthreading'in kapatılması gerekir: echo off > /sys/devices/system/cpu/smt/control

06 stress-ng ile Yük Altında Test

RT latency, sistem boştayken düşük olabilir ama gerçek üretim yükü altında dramatik biçimde artabilir. stress-ng, gerçekçi yük senaryoları oluşturur.

stress-ng kurulumu

bash — stress-ng kurulumu●
apt install -y stress-ng
# veya Yocto: IMAGE_INSTALL:append = " stress-ng"

# Temel stres test (tüm CPU'lar, 30 saniye)
stress-ng --cpu 4 --timeout 30s

Yük senaryoları

bash — stress-ng yük senaryoları●
# ─── CPU stres — FPU yoğun hesaplama ───────────────────────
stress-ng --cpu 2 --cpu-method matrixprod --timeout 1h

# ─── IO stres — disk I/O yoğun ─────────────────────────────
stress-ng --io 4 --timeout 1h

# ─── Bellek stres — cache thrashing ────────────────────────
stress-ng --vm 2 --vm-bytes 512M --vm-method rowhammer --timeout 1h

# ─── Matris hesaplama — SIMD / NEON stres ──────────────────
stress-ng --matrix 4 --matrix-size 256 --timeout 1h

# ─── Network stres — TCP/UDP yoğun ─────────────────────────
stress-ng --sock 4 --timeout 1h

# ─── Kombine senaryo — gerçekçi production yükü ────────────
stress-ng \
  --cpu 2           # non-RT CPU'larda CPU stres
  --io 2            # IO stres
  --vm 1 --vm-bytes 256M  # bellek stres
  --sock 2          # network stres
  --taskset 0,1     ← sadece non-RT CPU'larda çalış
  --timeout 2h &

# Arka planda stress-ng çalışırken cyclictest başlat
cyclictest -t2 -p98 -i1000 -a 2,3 -D 2h -h 400 --mlockall > latency-under-load.txt

hackbench ile scheduler stres

bash — hackbench kombinasyonu●
# hackbench = yoğun thread/process oluşturma ve haberleşme
# Scheduler'ı aşırı yüklemenin klasik yöntemi
hackbench -l 1000 -g 20 &
cyclictest -t4 -p98 -i1000 -D 30m -h 400 --mlockall

07 Pratik: Industrial Linux'ta Latency Bütçesi

Hedef: i.MX8MP tabanlı industrial Linux sistemde <50µs worst-case latency garantisi. Adım adım tuning süreci.

Başlangıç durumu ölçümü

bash — baseline ölçüm●
# Hiçbir tuning yapılmadan baseline ölç
cyclictest -t4 -p98 -i1000 -D 30m --mlockall > baseline.txt

# Baseline sonuç (kötü):
# T:0 Min:    4 Avg:   12 Max: 1832 ← 1.8ms! kabul edilemez
# T:1 Min:    5 Avg:   13 Max: 2145 ← 2.1ms!

Tuning adımları ve etkisi

bash — adım adım tuning●
# ── Adım 1: SMI kontrolü ────────────────────────────────────
hwlatdetect --duration=60
# SMIs: 0 → SMI sorun değil, devam et

# ── Adım 2: RT throttling kapat ─────────────────────────────
echo -1 > /proc/sys/kernel/sched_rt_runtime_us
cyclictest -t4 -p98 -i1000 -D 10m --mlockall
# Max: 892µs → hâlâ yüksek ama iyileşme var

# ── Adım 3: THP kapat ───────────────────────────────────────
echo never > /sys/kernel/mm/transparent_hugepage/enabled
cyclictest -t4 -p98 -i1000 -D 10m --mlockall
# Max: 234µs → iyileşme devam ediyor

# ── Adım 4: CPU isolation + IRQ affinity ────────────────────
# Kernel cmdline'a ekle ve yeniden boot: isolcpus=2,3 nohz_full=2,3
for irq in /proc/irq/*/smp_affinity; do
  echo 3 > $irq 2>/dev/null
done
systemctl stop irqbalance
cyclictest -t2 -p98 -i1000 -a 2,3 -D 10m --mlockall
# Max: 48µs → HEDEF TUTTURULDU! (<50µs)

# ── Adım 5: Yük altında doğrulama ───────────────────────────
stress-ng --cpu 2 --io 2 --vm 1 --vm-bytes 256M --taskset 0,1 --timeout 2h &
cyclictest -t2 -p98 -i1000 -a 2,3 -D 2h -h 400 --mlockall > final.txt

# Final sonuç (hedef karşılandı):
# T:0 Min:    4 Avg:    9 Max:   47 ← <50µs!
# T:1 Min:    4 Avg:    9 Max:   44 ← <50µs!

CI pipeline — otomatik latency testi

.gitlab-ci.yml — RT latency job●
rt-latency-test:
  stage: board-test
  script:
    # Tuning uygula
    - ssh root@board "echo -1 > /proc/sys/kernel/sched_rt_runtime_us"
    - ssh root@board "echo never > /sys/kernel/mm/transparent_hugepage/enabled"
    # cyclictest çalıştır (5 dakika, yük altında)
    - ssh root@board "stress-ng --cpu 2 --timeout 5m &"
    - ssh root@board "cyclictest -t2 -p98 -i1000 -a 2,3 -D 5m --json > /tmp/lat.json"
    - scp root@board:/tmp/lat.json ./latency-results.json
    # Latency bütçesi kontrolü (max 50µs)
    - python3 ci/check-latency.py --threshold 50 latency-results.json
  artifacts:
    paths:
      - latency-results.json
