00 LAVA nedir?

LAVA (Linaro Automated Validation Architecture), fiziksel gömülü kartların otomatik test edilmesi için geliştirilmiş açık kaynaklı bir board farm yönetim sistemidir.

Gömülü Linux geliştirmede en can sıkıcı sorunlardan biri, her commit sonrasında binlerce satır kernel veya userspace kodunu gerçek donanımda doğrulama gerekliliğidir. Simülatörler belirli hataları yakalayamaz; QEMU, gerçek SoC'nin donanım davranışını, DMA transfer gecikmelerini ya da güç yönetimi geçişlerini birebir yansıtamaz.

LAVA bu sorunu şöyle çözer: bir board farm içindeki onlarca fiziksel kartı merkezi bir sunucudan yönetir. Her karta power cycle, serial konsol erişimi, ağ boot ve görüntü yazma yeteneklerini sistematik bir pipeline aracılığıyla sunar. GitLab CI bir merge request açıldığında LAVA'ya iş (job) gönderir, LAVA uygun kartı seçip image'i deploy eder, boot eder, test suite'i çalıştırır ve sonuçları geri döner.

KİM KULLANIR?

Linaro'nun kendisi, ARM, Qualcomm, Texas Instruments ve pek çok Tier-1 Yocto projesi LAVA kullanır. Upstream Linux kernel CI pipeline'ı (kernelci.org) de LAVA üzerine inşa edilmiştir. Kendi board farm'ınızı kurmak için tek bir Raspberry Pi bile yeterlidir.

LAVA'nın sağladığı temel özellikler

Board yönetimi
Yüzlerce farklı kart tipini tek bir sistemde tanımla ve yönet. Her kart için güç kontrolü, serial konsol ve ağ bağlantısı ayarla.
Job kuyruğu
Birden fazla CI pipeline'ı aynı anda iş gönderebilir. LAVA uygun cihazı seçer ve sıraya koyar, öncelik yönetimi destekler.
Artifact deploy
Kernel, DTB, rootfs, bootloader gibi artifact'ları TFTP, NFS, SD kart yazma veya fastboot ile otomatik olarak dağıtır.
Test tanımları
YAML tabanlı test suite'ler yazar, lava-test-shell ile board üzerinde komut çalıştırır ve çıktıyı parse eder.
REST API
Her şey REST API üzerinden yönetilebilir. xmlrpc ve lavacli komut satırı aracı da mevcuttur.
Web arayüzü
Job durumu, log çıktısı, test sonuçları, cihaz sağlığı ve geçmiş run'lar için zengin bir web dashboard sunar.

LAVA dört temel bileşenden oluşur: Server, Dispatcher, Worker ve DUT.

LAVA Server

Django tabanlı web uygulamasıdır. PostgreSQL veritabanı kullanır. Job kabul eder, cihaz havuzunu yönetir, test sonuçlarını depolar. Genellikle Debian/Ubuntu üzerine kurulur; Docker image da mevcuttur.

Dispatcher

Cihaza yakın konumda çalışır. LAVA Server'dan iş alır ve ilgili DUT üzerinde uygular. TFTP sunucusu çalıştırabilir, serial konsol üzerinden iletişim kurar, güç kontrolü yapar. Her dispatcher birden fazla DUT'a hizmet verebilir.

Worker

Dispatcher'ın yürütme bağlamıdır. Her worker belirli bir dispatcher instance ile ilişkilidir. Birden fazla job'ı paralel çalıştırabilmek için bir dispatcher altında birden fazla worker tanımlanabilir.

DUT (Device Under Test)

Test edilen fiziksel karttır. LAVA şunlara ihtiyaç duyar:

  • PDU (Power Distribution Unit) veya smart plug — güç kontrolü için
  • Serial konsol — UART veya USB-serial adaptör
  • Ağ bağlantısı — TFTP/NFS boot için Ethernet
PDU GEREKSİNİMİ

LAVA'nın en kritik bileşeni PDU'dur. Her DUT'un programatik olarak güç açılıp kapatılabilmesi gerekir. Ucuz başlangıç için TP-Link Kasa serisi akıllı prizler veya APC AP7900 gibi rack PDU'lar kullanılabilir. Raspberry Pi için GPIO üzerinden kontrol de mümkündür.

02 Device Dictionary

Device dictionary, bir DUT'un LAVA'ya nasıl bağlandığını ve nasıl kontrol edileceğini tanımlayan YAML/Jinja2 şablonudur.

Her fiziksel kart için bir device dictionary oluşturulur. Bu dosya LAVA Server'a yüklenir ve ilgili cihaz seçildiğinde dispatcher tarafından kullanılır. Jinja2 tabanlı bir şablonlama sistemi sayesinde ortak ayarlar base template'lerde toplanır, her kart sadece farklı parametrelerini override eder.

Basit device dictionary örneği — Raspberry Pi 4

device-dictionary / rpi4-01.yaml●
# /etc/lava-server/dispatcher-config/devices/rpi4-01.yaml
{% extends 'rpi4b.jinja2' %}

# Power control — TP-Link Kasa akıllı priz
{% set power_on_command  = '/usr/local/bin/kasa-pdu on  192.168.1.50 3' %}
{% set power_off_command = '/usr/local/bin/kasa-pdu off 192.168.1.50 3' %}

# Serial konsol — USB-serial adaptör
{% set connection_command = 'telnet localhost 2001' %}
# minicom/picocom yerine ser2net kullanılır:
# ser2net port 2001 → /dev/ttyUSB0 115200

# TFTP — kernel ve DTB için
{% set tftp_mac_address = 'dc:a6:32:xx:xx:xx' %}
{% set tftp_server_ip   = '192.168.1.10' %}

# NFS rootfs
{% set nfs_server_ip = '192.168.1.10' %}
{% set nfs_rootfs    = '/srv/lava/nfs/rpi4-rootfs' %}

# Boot interrupt — U-Boot prompt bekle
{% set bootloader_prompt = 'U-Boot>' %}
{% set boot_interrupt_character = ' ' %}

Base template yapısı

Linaro, popüler kartlar için hazır base template'ler sağlar. Kendi SoC'niz için oluşturmanız gerekiyorsa şu yapıyı kullanabilirsiniz:

rpi4b.jinja2 — base template özeti●
{% extends 'base-uboot.jinja2' %}

{% block vland %}
interfaces:
  eth0:
    sysfs: "net/eth0"
    mac_addr: {{ tftp_mac_address }}
    tags:
      - 1G
    switch: "switchA"
    port: 3
{% endblock %}

{% block body %}
commands:
  connect: {{ connection_command }}
  hard_reset: {{ power_off_command }}; sleep 5; {{ power_on_command }}
  power_off: {{ power_off_command }}
  power_on:  {{ power_on_command }}
{% endblock %}

Önemli dictionary parametreleri

connection_command
Serial konsola bağlanma komutu. telnet localhost PORT veya ssh user@host screen /dev/ttyUSB0 formatında.
power_on/off_command
PDU komutları. Shell komutu olabilir: snmpset -v1 -c private ip .1.3.6.1.4.1... i 1
hard_reset_command
Güç döngüsü komutu. Off + sleep + On genellikle kapasitörler için 5-10 saniye bekleme içerir.
tftp_server_ip
TFTP sunucu adresi. Dispatcher ile aynı makinede çalışan tftpd-hpa kullanılabilir.
bootloader_prompt
U-Boot veya GRUB prompt stringi. LAVA bu stringi bekleyerek boot sürecini kontrol eder.
nfs_server_ip / nfs_rootfs
NFS export yolu. /etc/exports'ta *(rw,no_root_squash,async) ile export edilmiş olmalıdır.

03 Job Tanımı

LAVA job'ı, bir DUT üzerinde ne yapılacağını adım adım tanımlayan YAML dosyasıdır.

Bir LAVA job üç ana bölümden oluşur: metadata (job adı, cihaz tipi, timeout), actions (deploy + boot + test zinciri) ve notify (sonuç bildirimi). Actions bölümü sıralı olarak işlenir ve herhangi bir adım başarısız olursa job durdurulur.

job.yaml — tam örnek●
# LAVA Job Definition — Raspberry Pi 4, NFS boot + ptest
job_name: "rpi4-yocto-kirkstone-ptest"
device_type: rpi4b
priority: medium
visibility: public

timeouts:
  job:
    minutes: 45
  action:
    minutes: 10
  connection:
    seconds: 120

# Metadata — isteğe bağlı, arama/filtreleme için yararlı
metadata:
  image.name: "yocto-kirkstone-rpi4"
  kernel.version: "6.1.55"
  git.commit: "abc1234"
  build.url: "https://gitlab.example.com/..."

actions:

  # ─── DEPLOY ────────────────────────────────────────────
  - deploy:
      timeout:
        minutes: 8
      to: tftp
      kernel:
        url: http://artifacts.example.com/rpi4/Image
        type: image
      dtb:
        url: http://artifacts.example.com/rpi4/bcm2711-rpi-4-b.dtb
      nfsrootfs:
        url: http://artifacts.example.com/rpi4/core-image-full-cmdline.tar.gz
        compression: gz

  # ─── BOOT ──────────────────────────────────────────────
  - boot:
      timeout:
        minutes: 5
      method: u-boot
      commands: nfs
      auto_login:
        login_prompt: "login:"
        username: root
      prompts:
        - "root@rpi4:~#"

  # ─── TEST ──────────────────────────────────────────────
  - test:
      timeout:
        minutes: 30
      definitions:
        - repository: https://gitlab.example.com/lava-tests.git
          from: git
          path: ptest/ptest-runner.yaml
          name: ptest-runner
JOB GÖNDERME

Job'ı lavacli ile gönderin: lavacli --uri https://lava.example.com jobs submit job.yaml. Sonucu izlemek için: lavacli jobs logs JOB_ID. REST API ile de gönderilebilir: POST /api/v0.2/jobs/

04 Deploy Stratejileri

LAVA birçok farklı deploy yöntemini destekler. Hangi yöntemi seçeceğiniz, kartınızın boot yeteneğine bağlıdır.

TFTP + NFS (önerilen)

En esnek ve hızlı yöntemdir. Kernel ve DTB TFTP üzerinden, rootfs ise NFS üzerinden yüklenir. SD kart veya flash yazma gerekmez, bu da test döngüsünü dramatik biçimde hızlandırır.

deploy: tftp + nfsrootfs●
- deploy:
    to: tftp
    kernel:
      url: http://build-server/Image
      type: image
    dtb:
      url: http://build-server/soc.dtb
    modules:
      url: http://build-server/modules.tar.gz
      compression: gz
    nfsrootfs:
      url: http://build-server/rootfs.tar.gz
      compression: gz
      prefix: "./rootfs/"  # tar içinde prefix varsa

SD kart image yazma

SD kart writer (USB'ye bağlı) veya removable media ile çalışan kartlar için uygundur. Yavaş ama bazı kartlarda zorunludur.

deploy: removable media●
- deploy:
    to: removable-media
    image:
      url: http://build-server/sdcard.img.xz
      compression: xz
    device: /dev/sdb  # dispatcher üzerinde SD kart

Fastboot (Android/Qualcomm)

deploy: fastboot●
- deploy:
    to: fastboot
    images:
      boot:
        url: http://build-server/boot.img
      system:
        url: http://build-server/system.img
      vendor:
        url: http://build-server/vendor.img

05 Boot Stratejileri

Deploy sonrası LAVA, seçilen boot yöntemine göre kartı başlatır ve Linux prompt'unu bekler.

U-Boot (en yaygın)

LAVA, serial konsol üzerinden U-Boot'u keser (boot interrupt), ortam değişkenlerini ayarlar ve önceden tanımlanmış boot komutlarını çalıştırır. commands: nfs seçildiğinde base template'deki NFS boot komutu uygulanır.

boot: u-boot NFS●
- boot:
    method: u-boot
    commands: nfs
    type: image
    auto_login:
      login_prompt: "login:"
      username: root
    prompts:
      - 'root@[a-zA-Z0-9\-]+:~#'
    timeout:
      minutes: 5

GRUB

x86 ve bazı ARM kartlar için kullanılır. GRUB menüsü veya grub-editenv ile kernel parametreleri ayarlanır.

boot: grub●
- boot:
    method: grub
    commands:
      - insmod gzio
      - insmod part_gpt
      - linux /vmlinuz root=/dev/nfs nfsroot=192.168.1.10:/rootfs rw ip=dhcp
      - initrd /initramfs
      - boot
    prompts:
      - 'root@[a-zA-Z0-9\-]+:~#'

kexec

Hızlı kernel değiştirme için kullanılır. Mevcut çalışan sistemden yeni kernel'a geçer, PDU power cycle gerekmez. Test sürelerini dramatik biçimde kısaltır.

boot: kexec●
- boot:
    method: kexec
    kernel: /tmp/Image
    dtb:    /tmp/soc.dtb
    cmdline: "root=/dev/nfs nfsroot=192.168.1.10:/nfs/rootfs rw ip=dhcp"
    prompts:
      - 'root@[a-zA-Z0-9\-]+:~#'

06 Test Actions

LAVA'nın asıl gücü test action'larındadır. lava-test-shell ile board üzerinde komutlar çalıştırılır ve sonuçlar parse edilir.

lava-test-shell

Board üzerinde doğrudan shell komutları çalıştırmak için kullanılır. Sonuçlar lava-test-case ve lava-test-result komutlarıyla LAVA'ya raporlanır.

test: inline lava-test-shell●
- test:
    timeout:
      minutes: 15
    definitions:
      - repository:
          metadata:
            format: Lava-Test Test Definition 1.0
            name: basic-smoke
            description: "Temel sistem sağlık kontrolü"
          run:
            steps:
              - lava-test-case kernel-version --shell uname -r
              - lava-test-case uptime --shell uptime
              - lava-test-case eth0-up --shell ip link show eth0 | grep -q UP && echo pass || echo fail
              - lava-test-case memory-512mb --shell |
                  free -m | awk '/Mem/{if($2>512) print "pass"; else print "fail"}'
        from: inline
        name: smoke-test
        path: inline/smoke-test.yaml

Git repository'den test tanımı

Test tanımları ayrı bir git repo'da tutulur ve LAVA tarafından clone edilir. Bu yaklaşım test kodunu job tanımından bağımsız tutmanızı sağlar.

test/ptest-runner.yaml●
metadata:
  format: Lava-Test Test Definition 1.0
  name: ptest-runner
  description: "Yocto ptest suite çalıştır"
  maintainer: [test-team@example.com]
  os: [debian, oe]
  scope: [functional]

run:
  steps:
    - ptest-runner 2>&1 | tee /tmp/ptest.log
    - python3 /usr/lib/lava/lava_ptest_parser.py /tmp/ptest.log

parse:
  pattern: '^(?P<test_case_id>\S+)\s+(?P<result>PASS|FAIL)$'
SONUÇ PARSE

LAVA, test çıktısını parse.pattern regex'i ile tarar. test_case_id ve result named group'ları zorunludur. Result değerleri: pass, fail, skip, unknown.

07 GitLab CI Entegrasyonu

LAVA'yı GitLab CI pipeline'ına entegre etmek için lava-gitlab-connector veya doğrudan REST API kullanılır.

İki temel yaklaşım vardır: polling (CI job, LAVA job'ı tetikler ve sonucu bekler) ve callback (LAVA, tamamlanınca GitLab API'ya bildirim gönderir). Küçük farm'lar için polling daha basittir.

.gitlab-ci.yml — LAVA entegrasyonu

.gitlab-ci.yml●
stages:
  - build
  - lava-test

build-yocto:
  stage: build
  script:
    - bitbake core-image-full-cmdline
    - bitbake core-image-full-cmdline -c populate_sdk
  artifacts:
    paths:
      - build/tmp/deploy/images/rpi4/
    expire_in: 1 day

lava-board-test:
  stage: lava-test
  needs:
    - build-yocto
  image: python:3.11-slim
  variables:
    LAVA_SERVER: "https://lava.example.com"
    LAVA_TOKEN: "$LAVA_API_TOKEN"  # GitLab secret variable
    ARTIFACT_BASE: "https://artifacts.example.com/ci/${CI_PIPELINE_ID}"
  script:
    - pip install lavacli requests pyyaml
    # Artifact'ları artifact sunucusuna yükle
    - ./ci/upload-artifacts.sh build/tmp/deploy/images/rpi4/ "${ARTIFACT_BASE}"
    # Job YAML'ını CI değişkenleriyle oluştur
    - python3 ci/gen-lava-job.py --artifact-url "${ARTIFACT_BASE}" > /tmp/lava-job.yaml
    # LAVA'ya gönder ve bekle
    - python3 ci/lava-submit-wait.py --server "${LAVA_SERVER}" --token "${LAVA_TOKEN}" --job /tmp/lava-job.yaml
  artifacts:
    when: always
    reports:
      junit: lava-results.xml

lava-submit-wait.py — basit polling scripti

ci/lava-submit-wait.py●
import argparse, time, sys, requests, yaml

def main():
    ap = argparse.ArgumentParser()
    ap.add_argument('--server'); ap.add_argument('--token')
    ap.add_argument('--job')
    args = ap.parse_args()

    headers = {'Authorization': f'Token {args.token}'}
    base = f'{args.server}/api/v0.2'

    # Job'ı gönder
    with open(args.job) as f:
        job_def = f.read()
    r = requests.post(f'{base}/jobs/', json={'definition': job_def}, headers=headers)
    r.raise_for_status()
    job_id = r.json()['job_ids'][0']
    print(f"LAVA job submitted: {job_id}")

    # Tamamlanana kadar bekle (polling)
    while True:
        r = requests.get(f'{base}/jobs/{job_id}/', headers=headers)
        state = r.json()['state']
        health = r.json().get('health', 'Unknown')
        print(f"  state={state} health={health}")
        if state in ('Finished',):
            break
        time.sleep(30)

    if health != 'Complete':
        print(f"LAVA job failed: {health}")
        sys.exit(1)
    print("LAVA job passed!")

if __name__ == '__main__': main()

08 Pratik: Raspberry Pi 4 için LAVA

Sıfırdan bir LAVA ortamı kurarak Raspberry Pi 4 üzerinde ilk test job'ını çalıştırma adımları.

1. LAVA Server kurulumu (Debian 12)

bash — LAVA Server kurulumu●
# Linaro deposunu ekle
wget https://apt.lavasoftware.org/release/apt.lava.key
gpg --dearmor < apt.lava.key > /etc/apt/trusted.gpg.d/lava.gpg
echo "deb https://apt.lavasoftware.org/release bookworm main" \
  > /etc/apt/sources.list.d/lava.list

apt update
apt install -y lava

# Superuser oluştur
lava-server manage createsuperuser \
  --username admin --email admin@example.com

# PostgreSQL ve uWSGI otomatik başlatılır
# Web UI: http://localhost/ adresinden erişilebilir

2. Dispatcher ve ser2net kurulumu

bash — dispatcher kurulumu●
apt install -y lava-dispatcher ser2net tftpd-hpa nfs-kernel-server

# ser2net config — /dev/ttyUSB0'ı TCP 2001'e bağla
cat >> /etc/ser2net.conf << 'EOF'
2001:telnet:0:/dev/ttyUSB0:115200 8DATABITS NONE 1STOPBIT banner
EOF

# TFTP kök dizini
mkdir -p /srv/tftp
chown tftp:tftp /srv/tftp
echo 'TFTP_DIRECTORY="/srv/tftp"' >> /etc/default/tftpd-hpa

# NFS export
mkdir -p /srv/lava/nfs
echo "/srv/lava/nfs *(rw,no_root_squash,async,no_subtree_check)" >> /etc/exports
exportfs -ra

systemctl restart ser2net tftpd-hpa nfs-kernel-server

3. Raspberry Pi 4 U-Boot NFS boot ayarları

U-Boot — NFS boot environment●
# U-Boot prompt'ta (ilk kez elle girilir, sonra LAVA otomatik yapar):
U-Boot> setenv serverip 192.168.1.10
U-Boot> setenv nfsroot /srv/lava/nfs/rpi4-rootfs
U-Boot> setenv bootargs "root=/dev/nfs nfsroot=${serverip}:${nfsroot},v3 rw ip=dhcp console=ttyAMA0,115200"
U-Boot> setenv bootcmd "dhcp; tftp ${kernel_addr_r} Image; tftp ${fdt_addr_r} bcm2711-rpi-4-b.dtb; booti ${kernel_addr_r} - ${fdt_addr_r}"
U-Boot> saveenv

4. Device dictionary yükle ve ilk job gönder

bash — device dictionary yükle ve job çalıştır●
# lavacli konfigürasyonu
lavacli identities add --uri https://lava.example.com \
  --username admin --token ADMIN_TOKEN default

# Cihaz tipi ekle (web UI veya CLI)
lava-server manage add-device-type rpi4b

# Cihaz ekle
lava-server manage add-device --device-type rpi4b \
  --worker dispatcher-01 rpi4-01

# Device dictionary'yi yükle
lavacli devices dict set rpi4-01 < rpi4-01.yaml

# İlk test job'ını gönder
lavacli jobs submit job.yaml
# Çıktı: Submitted as job: 1

# Logları canlı izle
lavacli jobs logs 1
SORUN GİDERME

Serial konsol çalışmıyorsa: minicom -D /dev/ttyUSB0 -b 115200 ile doğrula. TFTP boot olmuyorsa: tcpdump -i eth0 port 69 ile TFTP isteklerini izle. Dispatcher log'ları için: journalctl -u lava-dispatcher -f
