00 OTA neden kritik

Sahadaki on binlerce cihazı güvenle güncellemek için atomik, imzalı ve rollback destekli bir OTA altyapısı zorunludur — başarısız güncelleme iş durması veya saha ziyareti demektir.

Ölçek problemi

Bir araç üreticisi ya da endüstriyel IoT sağlayıcısı olarak 10.000 cihazı düşünün. Her biri farklı bir lokasyonda, kimi fabrika içinde kimi uzak bir tarlada. Manuel güncelleme — USB stick ile — pratik değildir: hem maliyetli hem de yavaş. Otomasyon olmadan güvenlik yamalarını zamanında uygulamak mümkün olmaz.

Brick olmuş cihaz riski

Yarıda kesen bir güncelleme — güç kesilmesi, ağ kopması, disk dolması — cihazı boot edemez hale getirebilir. Atomic update ilkesi şudur: güncelleme ya tamamen uygulanır ya da hiç. Atomik olmayan bir OTA sistemi production'da kabul edilemez.

Güvenlik yamaları

CVE açıklandığında sahadaki cihazlara 24–48 saat içinde yama ulaştırabilmek, modern bir OTA altyapısının olmazsa olmaz özelliğidir. Özellikle ağa bağlı gömülü cihazlarda kernel, OpenSSL veya uygulama katmanındaki açıklar ciddi risk oluşturur.

Bu bölümde

  • 10.000 cihaz → manuel güncelleme imkânsız — OTA zorunluluk
  • Atomic update: ya tam uygula ya hiç — brick riski ortadan kalkar
  • SWUpdate, Mender, RAUC, OSTree, balenaCloud karşılaştırması
  • Seçim kriterleri: lisans, ekosistem, server, delta, özelleştirme

01 SWUpdate temelleri

SWUpdate, Stefano Babic tarafından geliştirilen LGPL lisanslı, Yocto ve Buildroot ile tam uyumlu bir embedded Linux güncelleme framework'üdür.

SWUpdate nedir

SWUpdate tek bir çalıştırılabilir — swupdate — içinde birçok güncelleme senaryosunu destekler: yerel dosyadan güncelleme, web arayüzü, suricatta ile uzak server bağlantısı. Güncelleme paketi .swu uzantılı bir CPIO archive'dir; içinde sw-description tarif dosyası ve binary içerikler bulunur.

.swu dosyası (CPIO archive)
├── sw-description          ← güncelleme tarifi (libconfig syntax)
├── sw-description.sig      ← RSA/ECDSA imzası (opsiyonel)
├── rootfs.squashfs         ← yeni sistem image'ı
├── kernel.img              ← yeni kernel (opsiyonel)
└── postupdate.sh           ← güncelleme sonrası script (opsiyonel)

Temel kullanım

shell●
# Yerel .swu dosyasından güncelleme
swupdate -i myupdate.swu

# Verbose log ile
swupdate -i myupdate.swu -v

# Web arayüzü başlat (port 8080)
swupdate -w "-document_root /usr/share/swupdate/www"

# Sadece doğrulama — uygulamadan önce test
swupdate -i myupdate.swu --dry-run

# Yüklü versiyonu göster
swupdate --version

Yocto meta-swupdate layer

bblayers.conf●
BBLAYERS += " \
  /path/to/meta-swupdate \
  /path/to/meta-swupdate-boards \
"

# local.conf'a ekle
IMAGE_INSTALL:append = " swupdate swupdate-www"

# swupdate.cfg örneği (cihaza gömülü)
SWUPDATE_CONFIGS = "swupdate.cfg"

.swu paketi oluşturma

shell — .swu oluştur●
# Dosyaları CPIO ile paketle
# sw-description her zaman ilk olmalı
echo sw-description sw-description.sig rootfs.squashfs postupdate.sh | \
  cpio -ovL -H crc > myupdate.swu

# İçeriği kontrol et
cpio -itv < myupdate.swu
NOT

SWUpdate CPIO formatında sw-description dosyasının archive içinde ilk gelmesi zorunludur. Aksi takdirde swupdate paketi ayrıştıramaz ve güncelleme başlamadan hata verir.

Bu bölümde

  • SWUpdate: LGPL, CPIO tabanlı .swu paketi, Yocto/Buildroot uyumlu
  • sw-description: libconfig syntax güncelleme tarifi
  • swupdate -i, -v, -w, --dry-run komutları
  • .swu oluşturma: cpio ile — sw-description ilk olmalı

02 sw-description ve handler'lar

sw-description, SWUpdate'in neyi nereye nasıl yazacağını tanımlayan libconfig formatındaki güncelleme tarif dosyasıdır.

sw-description anatomisi

sw-description●
software =
{
    version = "1.2.0";
    description = "Firmware update v1.2.0";

    /* Donanım uyumluluk listesi */
    hardware-compatibility = ["1.0", "1.1"];

    images:
    (
        {
            filename  = "rootfs.squashfs";
            type      = "raw";
            device    = "/dev/mmcblk0p3";    /* standby partition */
            sha256    = "a1b2c3d4...";      /* bütünlük doğrulama */
            installed-directly = true;    /* RAM'e almadan yaz */
        }
    );

    scripts:
    (
        {
            filename  = "postupdate.sh";
            type      = "shellscript";
        }
    );

    bootenv:
    (
        {
            name  = "upgrade_available";
            value = "1";
        },
        {
            name  = "bootcount";
            value = "0";
        }
    );
}

Handler türleri

rawDoğrudan block device'a yazar — disk partition güncelleme için en yaygın
rawfileBir partition içindeki dosyayı günceller — tek dosya güncellemesi
shellscriptPre veya post install script çalıştırır — environment değişkenleri geçilir
ubivolUBI volume'a yazar — NAND flash sistemlerde
rdifflibrsync binary diff uygular — delta güncelleme
ssblSecond stage bootloader günceller — dikkatli kullanılmalı

shellscript handler örneği

postupdate.sh●
#!/bin/sh
# SWUpdate bu script'i INSTALL sonrası çalıştırır
# Ortam değişkenleri: $SWUPDATE_IMAGE_NAME, $SWUPDATE_HANDLER_ARGS

set -e

# Yeni partition'ı aktif et
CURRENT_PART=$(fw_printenv active_part | cut -d= -f2)

if [ "$CURRENT_PART" = "a" ]; then
    NEW_PART="b"
else
    NEW_PART="a"
fi

fw_setenv candidate_part "$NEW_PART"
fw_setenv upgrade_available 1
fw_setenv bootcount 0

echo "Güncelleme tamamlandı. Yeni partition: $NEW_PART. Reboot gerekli."
exit 0

ubivol handler — NAND flash

sw-description — NAND●
images:
(
    {
        filename  = "rootfs.ubifs";
        type      = "ubivol";
        volume    = "rootfs_b";    /* UBI volume adı */
        mtdnum    = "3";            /* /dev/mtd3 */
    }
);

version ve hardware-compatibility

SWUpdate, sw-description içindeki hardware-compatibility listesini cihaz donanım revizyonu ile karşılaştırır. Eşleşme yoksa güncelleme reddedilir. Bu mekanizma, yanlış firmware'ın yanlış donanıma yüklenmesini önler:

/etc/hwrevision (cihazda)●
myboard 1.1

Bu bölümde

  • sw-description: version, hardware-compatibility, images, scripts, bootenv bölümleri
  • raw, rawfile, shellscript, ubivol, rdiff handler türleri
  • shellscript handler: post-install script ile partition geçişi
  • hardware-compatibility: yanlış donanıma yüklemeyi engeller

03 A/B partition ile SWUpdate

SWUpdate'in A/B partition desteği, aktif olmayan partition'a yazarak ve bootloader env değişkenleriyle geçiş yaparak atomik güncelleme sağlar.

Partition layout

PartitionEtiketBoyutAçıklama
/dev/mmcblk0p1boot64 MBU-Boot env + kernel
/dev/mmcblk0p2rootfs_a256 MBSistem A (aktif)
/dev/mmcblk0p3rootfs_b256 MBSistem B (standby)
/dev/mmcblk0p4dataKalanKalıcı veri

Aktif partition tespiti

shell●
# Hangi partition aktif?
fw_printenv active_part
# active_part=a

# Standby partition'ı hesapla
ACTIVE=$(fw_printenv active_part | cut -d= -f2)
if [ "$ACTIVE" = "a" ]; then STANDBY_DEV=/dev/mmcblk0p3
else STANDBY_DEV=/dev/mmcblk0p2; fi
echo "Standby: $STANDBY_DEV"

sw-description — A/B aware

sw-description●
software =
{
    version = "2.0.0";
    hardware-compatibility = ["2.0"];

    /* stable: A'dan B'ye güncelleme */
    stable:
    {
        images:
        (
            {
                filename = "rootfs.squashfs";
                type     = "raw";
                device   = "/dev/mmcblk0p3";  /* B partition */
                sha256   = "deadbeef...";
            }
        );
        bootenv:
        (
            { name = "upgrade_available"; value = "1"; },
            { name = "bootcount";         value = "0"; },
            { name = "candidate_part";    value = "b"; }
        );
    };
}

Güncelleme akışı

1. swupdate -i firmware-2.0.swu
2. sw-description parse edilir — version/hw-compat kontrol
3. rootfs.squashfs → /dev/mmcblk0p3 (B partition) yazılır
4. SHA-256 doğrulanır
5. postupdate.sh çalışır: fw_setenv upgrade_available 1
6. swupdate başarı döner
7. Uygulama reboot verir
8. U-Boot: upgrade_available=1 → B'den boot dener
9. Linux başarıyla başlar → başarı onayı verilir
10. fw_setenv upgrade_available 0; fw_setenv active_part b

Başarı onayı — user space'den

shell — commit script●
# Uygulama sağlıklı başladıktan sonra çalıştır
# (örneğin systemd oneshot service ile)

fw_setenv upgrade_available 0
fw_setenv bootcount 0
CURRENT=$(fw_printenv candidate_part | cut -d= -f2)
fw_setenv active_part "$CURRENT"

logger "OTA commit: active_part=$CURRENT"
DİKKAT

Başarı onayı verilmezse (upgrade_available sıfırlanmazsa) U-Boot bootcount sayacı artmaya devam eder. bootlimit'e ulaşıldığında otomatik rollback gerçekleşir — güncellemeniz geri alınır. Başarı onayını uygulama sağlık kontrolünden sonra verin.

Bu bölümde

  • A/B layout: rootfs_a + rootfs_b + data partition
  • fw_printenv / fw_setenv ile aktif partition yönetimi
  • sw-description stable bölümü: standby partition'a yaz
  • Başarı onayı: upgrade_available=0 — rollback önlenir

04 Mender temelleri

Mender, daemon tabanlı bir OTA client'ı, Yocto entegrasyonu ve hazır sunucu altyapısıyla production'a hızla geçilmesini sağlayan açık kaynaklı bir güncelleme platformudur.

Mender mimarisi

Mender Server (Hosted veya self-hosted)
    │   HTTPS polling (varsayılan: 30 dakika)
    ▼
mender client (daemon, cihazda çalışır)
    │   artifact indir → doğrula → yükle
    ▼
A/B rootfs partition (ext4 veya squashfs)
    │   commit / rollback
    ▼
U-Boot env (bootcount, mender_boot_part)

Mender artifact oluşturma

shell●
# Rootfs image'dan Mender artifact oluştur
mender-artifact write rootfs-image \
  -t raspberrypi4      \  # hedef device type
  -n "release-1.0"      \  # artifact adı (unique olmalı)
  -f rootfs.ext4       \  # kaynak image
  -o update-1.0.mender    # çıktı dosyası

# Artifact içeriğini incele
mender-artifact read update-1.0.mender

Mender client komutları

shell — hedef cihaz●
# Yerel artifact yükle
mender install update-1.0.mender

# Yükleme başarılıysa commit
mender commit

# Rollback (henüz commit yapılmamışsa)
mender rollback

# Mevcut artifact adını göster
mender show-artifact

# Daemon'u yeniden başlat (server'a tekrar bağlan)
systemctl restart mender-updated

Yocto meta-mender entegrasyonu

local.conf●
INHERIT += "mender-full"

MENDER_DEVICE_TYPE     = "raspberrypi4"
MENDER_ARTIFACT_NAME   = "release-1.0"
MENDER_SERVER_URL      = "https://hosted.mender.io"
MENDER_TENANT_TOKEN    = "your-tenant-token"

# Partition boyutları (MB)
MENDER_STORAGE_TOTAL_SIZE_MB  = "7800"
MENDER_DATA_PART_SIZE_MB      = "1024"
MENDER_BOOT_PART_SIZE_MB      = "40"

Hosted Mender vs self-hosted

SeçenekAvantajDezavantaj
Hosted Mender (ücretsiz tier)10 cihaza kadar ücretsiz, hızlı başlangıç, bakım yokVeri dışarıda, cihaz limiti
Self-hosted (open source)Tam kontrol, sınırsız cihaz, offlineDocker stack yönetimi, güncelleme sorumluluğu

Bu bölümde

  • Mender: daemon tabanlı client, HTTPS polling, A/B partition
  • mender-artifact write ile .mender artifact oluşturma
  • mender install / commit / rollback / show-artifact
  • meta-mender Yocto layer ile tam entegrasyon

05 Delta güncelleme

Delta güncelleme, yalnızca değişen blokları göndererek bant genişliği tüketimini ve güncelleme süresini dramatik şekilde azaltır.

Full vs delta karşılaştırması

YöntemBoyut (256 MB rootfs, küçük güncelleme)Bant Genişliği
Full image~80 MB (squashfs ile)Her güncelleme aynı
rdiff (binary delta)~2–8 MB (değişikliklere göre)%95 tasarruf tipik
casync (content hash)Yalnızca değişen chunk'larBüyük image'larda çok etkili
mender binary-delta~1–5 MB (Mender delta artifact)xdelta3 tabanlı

SWUpdate rdiff handler

shell — delta oluşturma●
# librsync rdiff aracı ile binary diff oluştur
# Önce signature dosyası oluştur (referans image'dan)
rdiff signature rootfs-v1.0.squashfs rootfs-v1.0.sig

# Yeni image ile delta hesapla
rdiff delta rootfs-v1.0.sig rootfs-v1.1.squashfs rootfs.rdiff

# Doğrulama: delta'yı uygula ve karşılaştır
rdiff patch rootfs-v1.0.squashfs rootfs.rdiff rootfs-v1.1-patched.squashfs
sha256sum rootfs-v1.1.squashfs rootfs-v1.1-patched.squashfs

sw-description — rdiff handler

sw-description●
images:
(
    {
        filename  = "rootfs.rdiff";
        type      = "rdiff_image";
        device    = "/dev/mmcblk0p3";
        properties: {
            rdiffbase = ["/dev/mmcblk0p2"];  /* kaynak (aktif partition) */
        };
    }
);

Mender binary-delta artifact

shell●
# mender-binary-delta modülü ile delta artifact
mender-artifact write module-image \
  --type binary-delta \
  -t raspberrypi4 \
  -n "release-1.1-delta" \
  -f rootfs-v1.0-to-v1.1.delta \
  -o update-1.1-delta.mender

# Delta artifact'ı hedef cihaza yükle (aynı komut)
mender install update-1.1-delta.mender

casync ile büyük image'lar

shell — casync●
# casync: içerik bazlı chunk'lar ile delta
# Önce v1.0 store oluştur
casync make rootfs-v1.0.castr rootfs-v1.0.squashfs

# v1.1 için sadece değişen chunk'ları gönder
casync make rootfs-v1.1.castr rootfs-v1.1.squashfs

# Hedef: casync extract ile uygula
casync extract rootfs-v1.1.castr --seed=rootfs-v1.0.squashfs rootfs-new.squashfs
NOT

rdiff, referans ve hedef image arasındaki blok benzerliğine dayanır. Büyük binary değişiklikler (yeni kernel, yeniden derlenen uygulama) delta boyutunu önemli ölçüde artırabilir. Gerçek kazanım, image büyüklüğüne ve değişiklik oranına göre değişir — production'da ölçüm yapın.

Bu bölümde

  • Delta güncelleme: yalnızca değişen bloklar — %90+ bant genişliği tasarrufu
  • rdiff (librsync): SWUpdate rdiff_image handler
  • Mender binary-delta artifact: xdelta3 tabanlı
  • casync: içerik adreslenebilir chunk'lar — büyük image'lar için

06 İmzalama ve güvenlik

İmzalanmamış güncelleme kabul eden bir OTA sistemi, saldırganın istediği kodu cihaza yüklemesine kapı açar — imzalama zorunludur.

SWUpdate RSA imzalama

shell — anahtar üretimi ve imzalama●
# RSA-4096 anahtar çifti üret
openssl genrsa -out private.pem 4096
openssl rsa -in private.pem -pubout -out public.pem

# sw-description dosyasını imzala
openssl dgst -sha256 -sign private.pem \
  -out sw-description.sig sw-description

# .swu paketini imzalı olarak oluştur
echo sw-description sw-description.sig rootfs.squashfs | \
  cpio -ovL -H crc > myupdate.swu

SWUpdate imza doğrulama

/etc/swupdate.cfg (cihazda)●
globals:
{
    signingkey   = "/etc/swupdate/public.pem";
    loglevel     = "3";
    no-downgrade = true;   /* eski versiyona geçişi engelle */
}
shell — imzalı güncelleme●
# swupdate imzayı otomatik doğrular (cfg'de key tanımlıysa)
swupdate -i myupdate.swu -v

ECDSA imzalama (önerilen — RSA'dan küçük)

shell — ECDSA P-256●
openssl ecparam -name prime256v1 -genkey -noout -out ec-private.pem
openssl ec -in ec-private.pem -pubout -out ec-public.pem

openssl dgst -sha256 -sign ec-private.pem \
  -out sw-description.sig sw-description

Mender artifact imzalama

shell●
# Artifact oluştururken imzala
mender-artifact write rootfs-image \
  -t raspberrypi4 \
  -n "release-2.0" \
  -f rootfs.ext4 \
  -k private.pem \        /* imzalama anahtarı */
  -o update-2.0.mender

# Cihazda public key yapılandırması
# /etc/mender/mender.conf içinde:
# "ArtifactVerifyKey": "/etc/mender/public.pem"

Anti-rollback: versiyon kontrolü

sw-description — no-downgrade●
software =
{
    version = "2.0.0";
    /* SWUpdate cihaz versiyonu ile karşılaştırır */
    /* no-downgrade: true → 1.x.x → 2.0.0 OK; 2.0.0 → 1.x.x REJECT */
};
DİKKAT

Public key'i cihaz image'ına gömün — asla runtime'da indirmeyin. Private key'i CI/CD sisteminizde güvenli şekilde saklayın (örneğin HashiCorp Vault veya GitHub Secrets). Private key sızdığında tüm sahadaki cihazlar tehlikeye girer.

Bu bölümde

  • SWUpdate: openssl ile RSA/ECDSA sw-description imzalama
  • swupdate.cfg: signingkey ile otomatik doğrulama
  • Mender: -k flag ile artifact imzalama, /etc/mender/public.pem
  • Anti-rollback: no-downgrade ile eski versiyona geçiş engeli

07 HawkBit server entegrasyonu

Eclipse HawkBit, cihaz gruplarını, deployment campaign'lerini ve güncelleme geçmişini yöneten açık kaynaklı bir OTA management server'ıdır — SWUpdate suricatta modülü ile entegre çalışır.

HawkBit mimarisi

HawkBit Server (Docker Compose)
├── hawkbit-update-server   ← REST API + UI
├── rabbitmq                ← async messaging
└── mysql / mariadb         ← artifact metadata

        ↕ HTTPS (polling)

SWUpdate (cihazda, suricatta modu)
└── swupdate -f swupdate.cfg -u "http://hawkbit:8080;tenant=DEFAULT;id=dev1"

HawkBit Docker ile başlatma

docker-compose.yml●
version: "3"
services:
  hawkbit:
    image: hawkbit/hawkbit-update-server:latest
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: "jdbc:mysql://mysql:3306/hawkbit"
      SPRING_DATASOURCE_USERNAME: hawkbit
      SPRING_DATASOURCE_PASSWORD: hawkbit
      SPRING_RABBITMQ_HOST: rabbitmq
    depends_on:
      - mysql
      - rabbitmq

  mysql:
    image: mysql:8
    environment:
      MYSQL_DATABASE: hawkbit
      MYSQL_USER: hawkbit
      MYSQL_PASSWORD: hawkbit
      MYSQL_ROOT_PASSWORD: root

  rabbitmq:
    image: rabbitmq:3-management
shell●
docker compose up -d
# UI: http://localhost:8080 (admin/admin)

swupdate.cfg — suricatta konfigürasyonu

/etc/swupdate.cfg●
suricatta:
{
    tenant   = "DEFAULT";
    id       = "device-001";
    url      = "http://192.168.1.10:8080";
    polldelay = 20;   /* saniye — sunucu ne sıklıkla sorulsun */
    retry     = 3;
    retrywait = 10;
}

globals:
{
    signingkey = "/etc/swupdate/public.pem";
    loglevel   = "3";
}

Suricatta modunu başlatma

shell — cihazda●
# suricatta modunda çalıştır
swupdate -f /etc/swupdate.cfg \
  -u "--tenant DEFAULT --id device-001 --url http://hawkbit:8080"

# systemd service olarak
# /etc/systemd/system/swupdate.service
# ExecStart=/usr/bin/swupdate -f /etc/swupdate.cfg -u "..."

HawkBit'te campaign oluşturma

HawkBit UI üzerinden: Upload sekmesinde .swu artifact yükleyin → Distributions'da versiyon tanımlayın → Targets'ta cihaz grupları oluşturun → Deployment'ta rollout başlatın. Cihazlar bir sonraki polling döngüsünde güncellemeyi otomatik olarak alır.

NOT

HawkBit'in confirmation-flow özelliği ile cihaz güncellemeyi onaylamadan uygulamaz — kullanıcı arayüzü olan sistemlerde operatör onayı zorunlu kılınabilir. Bu özelliği hawkbit.server.ui.flow.confirmation ile etkinleştirin.

08 Pratik: güncelleme pipeline

Production OTA pipeline'ı CI/CD build, artifact imzalama, QEMU testi, canary rollout ve izleme adımlarından oluşur.

CI/CD entegrasyonu

.gitlab-ci.yml (örnek)●
stages:
  - build
  - package
  - sign
  - test
  - release

build-rootfs:
  stage: build
  script:
    - bitbake core-image-minimal
    - cp tmp/deploy/images/raspberrypi4/core-image-minimal-raspberrypi4.squashfs artifacts/

package-swu:
  stage: package
  script:
    - envsubst < sw-description.tmpl > sw-description
    - echo sw-description rootfs.squashfs postupdate.sh |
        cpio -ovL -H crc > update-${CI_PIPELINE_IID}.swu

sign-swu:
  stage: sign
  script:
    - openssl dgst -sha256 -sign ${SIGNING_KEY_PATH} \
        -out sw-description.sig sw-description
    - # Yeniden paketle (imzayla birlikte)
    - echo sw-description sw-description.sig rootfs.squashfs |
        cpio -ovL -H crc > update-${CI_PIPELINE_IID}-signed.swu

QEMU'da güncelleme testi

shell — QEMU test●
# Sanal disk oluştur (A/B layout)
qemu-img create -f qcow2 test-disk.qcow2 2G

# QEMU'da boot et
qemu-system-aarch64 \
  -M virt -cpu cortex-a53 -m 512M \
  -kernel Image \
  -drive file=test-disk.qcow2,format=qcow2 \
  -netdev user,id=net0,hostfwd=tcp::2222-:22 \
  -device virtio-net-device,netdev=net0

# SSH ile bağlan ve güncellemeyi test et
ssh -p 2222 root@localhost "swupdate -i /tmp/update.swu -v"
